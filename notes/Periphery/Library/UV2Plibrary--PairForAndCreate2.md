# Uniswap V2 Deep Dive - Part A
> [!NOTICE]
>
> This dissection follows the original Uniswap V2 source code and explanations as they are widely referenced and discussed throughout the ecosystem.
>
> My implementation is written using Solidity `^0.8.20` (and newer compiler versions), which introduces stricter type-conversion rules than the Solidity version used by Uniswap V2.
>
> As a result, some code snippets in these notes may appear slightly different in my implementation.
>
> Example:
>
> Original Uniswap V2:
>
> ```solidity
> pair = address(uint(keccak256(...)))
> ```
>
> Modern Solidity:
>
> ```solidity
> pair = address(uint160(uint256(keccak256(...))))
> ```
>
> The underlying logic, behavior, and address derivation process remain exactly the same.
>
> The newer syntax simply makes the conversion from:
>
> ```text
> bytes32
>   ↓
> uint256
>   ↓
> uint160
>   ↓
> address
> ```
>
> explicit for compiler safety.
>
> Throughout this dissection, the focus is on understanding the protocol logic rather than compiler-version-specific syntax differences.

# Section 6 - Salt

This caused MANY questions.

---

# Question

Why not:

```text
salt = "elephant"?
```
# Important Clarification

## Can Salt Actually Be "elephant"?

Yes.

CREATE2 itself does not care what the salt means.

It only requires:

```solidity
bytes32 salt
```

So something like:

```solidity
bytes32 salt = keccak256("elephant");
```

is completely valid.

CREATE2 would happily deploy using that salt.

---

The reason Uniswap does NOT use:

```text
"elephant"
```

is not because CREATE2 forbids it.

It is because Uniswap needs:

```text
One Unique Salt
Per Unique Pair
```

For example:

```text
WETH-USDC
WETH-DAI
WBTC-WETH
LINK-USDC
```

must all generate different salts.

Using:

```text
"elephant"
```

for every pair would break that requirement.

Therefore Uniswap derives the salt from:

```solidity
keccak256(
    abi.encodePacked(
        token0,
        token1
    )
)
```

which guarantees that different token pairs produce different salts.

---

### Mental Model

CREATE2 Requirement:

```text
Any bytes32 Salt
```

Uniswap Requirement:

```text
Unique Pair-Specific Salt
```

These are two different requirements.


---

# Reality

Salt must uniquely identify pair.

---

If:

```text
WETH-USDC
```

and

```text
WETH-DAI
```

used:

```text
elephant
```

both would collide.

---

# Therefore

Uniswap creates salt from:

```solidity
keccak256(
    abi.encodePacked(
        token0,
        token1
    )
)
```

---

Q: Difference between Creation Code and Runtime Code?

Answer:

```text
Creation code executes during deployment and
returns runtime code. Runtime code is stored
on-chain and executes after deployment.
```

Answer:


Answer:

```text

They affect storage initialization,
not the runtime code itself.
```

Answer:

```text
Because init code is what Ethereum receives
and executes during deployment.
```

# STORAGE AND RUNTIME EXTRA KNOWLOEDGE(IMPORTANT OUTSIDE OF THIS)
# Runtime Code, Storage, and How They Connect

---

## Q: Is runtime code what we actually interact with when we call a contract after deployment?

### Short Answer

YES. Runtime bytecode is the actual contract logic stored on-chain. Every function call executes this runtime code.

---

### Detailed Explanation

When you call a function on a deployed contract — like `swap()`, `transfer()`, `mint()` — the EVM looks up the contract address, loads the runtime bytecode stored there, and executes the matching function.

The runtime bytecode contains all the function logic: every `if` statement, every loop, every calculation. It's what makes the contract do what it does.

The deployment transaction sends the creation bytecode. The constructor runs once and disappears. What remains at the contract address forever is the runtime bytecode. That's what you interact with every time you call the contract.

---

### What's at the Contract Address

The contract address stores three things:

- Runtime Bytecode: the function logic (swap, mint, burn, getReserves, etc.)
- Storage: state variables, balances, mappings
- ETH Balance: any ETH sent to the contract

The runtime bytecode is the code. Storage is the data. They are separate but both tied to the same address.

---

### 🧒 Child Analogy

A house at 123 Main Street. The house itself (walls, rooms, layout) is the runtime bytecode. It never changes after construction. The furniture inside (beds, tables, chairs) is the storage. It changes when you buy new things or rearrange. The money in the safe is the ETH balance.

When someone visits 123 Main Street, they interact with the house (runtime code). They sit on the furniture (read storage). They might move things around (write storage). Everything is at the same address.

---

---

## Q: Is storage info also inside runtime bytecode?

### Short Answer

NO. Storage is completely separate from runtime bytecode. They are different things stored in different places.

---

### Detailed Explanation

Runtime bytecode is CODE. It contains instructions and logic. Storage is DATA. It contains values like token balances, reserves, and mappings.

They live in different areas of the EVM. The code section is immutable. Once deployed, the runtime bytecode never changes. The storage section is mutable. It changes with every swap, every mint, every burn, every transfer.

The runtime code contains instructions to READ from storage (SLOAD opcode) and WRITE to storage (SSTORE opcode). But the storage values themselves are not inside the code. The code says "go look at storage slot 0 and tell me what's there." Storage slot 0 holds the value. The code just knows which slot to check.

---

### Visual: Two Separate Areas

The EVM code store contains runtime bytecode for each address. Each address points to its own code. This never changes after deployment.

The EVM state trie contains storage for each address and slot combination. The key is the address plus the slot number. The value is whatever is stored there. This changes constantly.

For example, at address 0xB4e1..., the code section has all the function logic. The storage section at slot 0 holds the token0 address. At slot 1 holds the token1 address. At slot 2 holds the packed reserves and timestamp. The code knows which slot to read, but the VALUES are in storage, not in the code.

---

### 🧒 Child Analogy

A cookbook and a fridge. The cookbook is the runtime bytecode. It contains recipes (instructions). "To make cookies: mix 2 cups flour, 1 cup sugar, 3 eggs." The cookbook itself never changes.

The fridge is the storage. It contains ingredients (data). Flour: 2 cups. Sugar: 1 cup. Eggs: 3. The fridge changes every time you cook.

The cookbook tells you HOW to use ingredients. It says "go look in the fridge for flour." But the cookbook doesn't contain the flour itself. The flour is in the fridge. They are separate things.

---

---

## Q: Then how is storage tied to the contract? I'm guessing by its address?

### Short Answer

YES. Storage is tied to the contract by its ADDRESS. The address is the link that connects code, storage, and ETH balance together.

---

### Detailed Explanation

Everything related to a contract lives at its address. The EVM uses the contract address as the key to find the runtime bytecode. It uses the same address (plus the storage slot number) as the key to find storage values. It uses the same address to check the ETH balance.

When you call a function on a contract, the EVM looks up that address. It finds the runtime bytecode there. As the code executes, whenever it reads storage (SLOAD), the EVM looks up storage at THAT SAME ADDRESS for the given slot. Whenever it writes storage (SSTORE), it writes to THAT SAME ADDRESS.

The address is the common identifier. Everything is namespaced under it. One address. Three things: code, storage, balance. All tied together by that single 20-byte address.

---

### Visual: The Address as a Folder

```text
Address: 0xB4e1...

├── code (runtime bytecode — immutable)
├── storage (all state variables — mutable)
└── balance (ETH held by contract)
```

One address. Three things tied to it.
---

---

### How the EVM Finds Everything

When someone calls 0xB4e1... and invokes the swap function, the EVM first looks up the address to find the code. It loads the runtime bytecode and starts executing. When the code does SLOAD to read reserves, the EVM looks up storage at address 0xB4e1... plus the slot number where reserves are stored. When the code does SSTORE to update reserves, it writes to storage at that same address and slot. When the code checks the contract's ETH balance, the EVM checks the balance at address 0xB4e1...

Everything flows through the address.

---

### 🧒 Child Analogy

A house at 123 Main Street. The address "123 Main Street" ties everything together. The house structure (code) is at that address. The furniture inside (storage) is at that address. The money in the safe (balance) is at that address.

If someone asks "where is the living room?" you say "123 Main Street." If they ask "where is the couch?" same address. If they ask "where is the safe?" same address.

The address is the link. Everything at 123 Main Street belongs to that address. The furniture isn't inside the walls. The money isn't part of the floor. They are separate things. But they all live at the same address. The address connects them all.

---

---

## Q: Where is the constructor code gone after deployment?

### Short Answer

The constructor code goes NOWHERE. It's not stored. It's not hidden. It's not recoverable. The EVM executes it once and throws it away forever.

---

### Detailed Explanation

During deployment, the EVM loads the full creation bytecode. This includes the constructor code at the beginning. The EVM executes everything. The constructor runs, sets initial state variables, validates inputs. At the end, the EVM hits a RETURN opcode. This RETURN says "here is the runtime code. Store this part. Discard everything else."

The EVM copies only the runtime portion to the contract's on-chain code storage. Everything else — the constructor code, the metadata — is discarded from memory. The deployment transaction completes. The memory is wiped. The discarded code is gone.

The constructor is not stored on-chain. Not in the transaction receipt. Not in event logs (unless the constructor emitted events). Not recoverable from the runtime bytecode. Not saved anywhere by the EVM.

You can see the RESULT of the constructor. The storage values it set are there. The token0 address. The token1 address. The factory address. These values remain in storage. But the constructor CODE that set those values is gone.

---

### 🧒 Child Analogy

Building a house. The construction crew arrives with tools and materials. They build the house. They set up everything inside. When they finish, they pack up their tools and leave.

## Q: So we cannot find a trace of the constructor in runtime after deployment?

### Short Answer

CORRECT. You CANNOT find the constructor code in the runtime bytecode. It's permanently gone.

---

### Detailed Explanation

The EVM splits the creation bytecode during deployment. The constructor executes once. The EVM returns only the runtime code to be stored on-chain. The constructor never makes it to the contract's on-chain bytecode.

What you CAN see on-chain is the runtime bytecode (all the functions) and the storage values (the state the constructor set). You can see that token0 equals some address. You can see that token1 equals some address. These are the results of the constructor's work. But you cannot see the constructor code that did that work.

It's like seeing a painted wall and knowing it was painted. You can see the color. You can see the finish. But you cannot see the painter. The painter is long gone. Only the work remains.

---

### 🧒 Child Analogy

A balloon pump and a balloon. The pump inflates the balloon. The balloon stays inflated. The pump goes back in the drawer. If you look at the inflated balloon, you cannot find the pump inside it. The balloon doesn't contain the pump. The pump did its job and left. The balloon is the result.

## Q: So creation bytecode has constructor info like what args it takes. During deployment, I put in the values like msg.sender = something. But creation code only has the args structure, not the values. Where does the value part get compiled? How does it reach the constructor?

### Short Answer

Constructor arguments are NOT compiled into the bytecode. They are APPENDED at deployment time. The bytecode contains the logic to READ them. The actual values are attached right before sending the deployment transaction.

---

### Detailed Explanation

The creation bytecode contains the constructor LOGIC. It has instructions that say "read the next 32 bytes from calldata and store as token0. Read the next 32 bytes from calldata and store as token1." But the actual addresses for token0 and token1 are not in the bytecode. They are blank slots waiting to be filled.

When you deploy, your deployment tool takes the creation bytecode. It takes your constructor arguments. It ABI-encodes those arguments. It concatenates them to the end of the bytecode. The full transaction data becomes: creation bytecode plus ABI-encoded constructor arguments.

The EVM executes this full data. The constructor code at the start reads from the end. It knows "the arguments start after my code ends." It decodes them and assigns them to the state variables.

---

### The Transaction Data Layout

The deployment transaction data field contains two parts glued together. First is the creation bytecode. This has all the code: constructor logic plus runtime code plus metadata. Second is the ABI-encoded constructor arguments. If the constructor takes an address and a uint256, this part is 64 bytes: 32 bytes for the padded address and 32 bytes for the padded uint256.

The constructor code accesses these trailing bytes via the CODECOPY opcode or by reading calldata at specific offsets. It knows the arguments are right after the bytecode.

---

### 🧒 Child Analogy

Think of a form with blank fields. The creation bytecode is the form itself. It says:

"Store ____ as token0. Store ____ as token1."

The blanks are empty. The form doesn't have the answers. When you deploy, you fill in the blanks. You write "USDC address" in the first blank and "WETH address" in the second. You attach the filled answers to the form and submit the whole package.

The EVM reads the form (bytecode). It finds the attached answers (constructor args). It fills in the blanks. Then it throws away the form and the answers, keeping only the final result (runtime code and storage values).

## Q: Do we send the ABI during deployment?

### Short Answer

NO. The ABI is NEVER sent to the blockchain. Only the creation bytecode (plus constructor arguments) is sent.

---

### Detailed Explanation

The deployment transaction contains only the creation bytecode in its data field. The ABI stays off-chain. It's a JSON file in your project folder. It describes the contract's interface: what functions exist, what parameters they take, what they return.

The ABI is for frontends, wallets, and other contracts to know how to encode function calls. It's the instruction manual. The blockchain doesn't need it. The blockchain only needs the bytecode.

Etherscan shows the ABI because someone uploaded it when they verified the contract. Or Etherscan might reconstruct it by analyzing the bytecode for function selectors. But the ABI was never part of the deployment transaction.

---

### 🧒 Child Analogy

When you buy a microwave, you get the microwave itself and an instruction manual. To install the microwave in your kitchen, you only need the microwave. You don't nail the instruction manual to the wall.

The microwave is the creation bytecode. The instruction manual is the ABI. One goes to the kitchen (blockchain). The other stays in your drawer (off-chain).

## One-Liner Summary

Runtime bytecode is what lives on-chain and executes when you call a contract. Storage is separate data tied to the same address. The constructor runs once and disappears forever. Constructor arguments are appended at deployment, not compiled in. The ABI is never sent on-chain. Everything is tied together by the contract address.
