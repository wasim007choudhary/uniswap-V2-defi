# Part 2 — Understanding `SwapParams`

Before reading any function, we decided to first understand the data that drives the entire arbitrage.

The first thing we see inside the contract is:

```solidity
struct SwapParams {
    // Router to execute first swap - tokenIn for tokenOut
    address router0;

    // Router to execute second swap - tokenOut for tokenIn
    address router1;

    // Token in of first swap
    address tokenIn;

    // Token out of first swap
    address tokenOut;

    // Amount in for the first swap
    uint256 amountIn;

    // Revert the arbitrage if profit is less than this minimum
    uint256 minProfit;
}
```

Instead of immediately reading each variable, we first wanted to understand **why this struct even exists.**

---

# ❓ Question

Why use a struct?

Couldn't we simply write:

```solidity
function swap(
    address router0,
    address router1,
    address tokenIn,
    address tokenOut,
    uint256 amountIn,
    uint256 minProfit
)
```

instead?

---

## 💡 Realization

Yes.

The contract would work perfectly.

Nothing would break.

So Solidity is **not forcing us** to create a struct.

This was simply a design decision by the author.

---

## Why is the struct better?

Instead of carrying six different variables everywhere,

we carry one object.

Without the struct we'd constantly pass:

```text
router0

router1

tokenIn

tokenOut

amountIn

minProfit
```

With the struct we simply pass:

```text
SwapParams
```

It groups everything required to execute **one arbitrage trade**.

Think of it as an instruction sheet.

---

# ❓ Question

Who fills this struct?

Initially we wondered:

Does the contract create it?

Does `_swap()` fill it?

---

## 💡 Realization

No.

The contract never fills it.

The contract only **receives** it.

Suppose Bob finds an arbitrage opportunity.

Bob decides:

```text
Buy from Uniswap.

Sell on SushiSwap.

Start with DAI.

Buy WETH.

Trade 100 DAI.

Require at least 5 DAI profit.
```

Bob packages all of those instructions into one object.

```text
SwapParams

router0

↓

Uniswap Router

router1

↓

SushiSwap Router

tokenIn

↓

DAI

tokenOut

↓

WETH

amountIn

↓

100 DAI

minProfit

↓

5 DAI
```

Then Bob sends those instructions to the contract.

The contract simply executes them.

---

# 💡 Huge Realization

The contract never decides:

- Which DEX to use.
- Which token to buy.
- Which token to sell.
- How much to trade.
- What minimum profit is acceptable.

Someone else already made those decisions.

Usually:

- Bob.
- An off-chain bot.
- A backend service.
- An MEV searcher.

The contract is an **executor**, not a strategist.

---

# Understanding Each Field

---

# `router0`

## ❓ Question

We asked:

> "There is no `router0` in Uniswap. Why are there two routers?"

Initially this looked confusing because Uniswap itself doesn't have contracts named:

```text
router0

router1
```

---

## 💡 Realization

These are **just variable names**.

`router0` simply means:

> The router used for the **first swap**.

Example:

```text
router0

↓

Uniswap Router
```

Tomorrow it could be:

```text
router0

↓

Curve Router
```

or

```text
router0

↓

PancakeSwap Router
```

The contract doesn't care.

---

# `router1`

Likewise,

`router1` simply means:

> The router used for the **second swap**.

Example:

```text
router1

↓

SushiSwap Router
```

Tomorrow it could instead be:

```text
router1

↓

Balancer Router
```

The contract simply performs:

Swap #1

↓

Swap #2

It doesn't care which protocols those routers belong to.

---

# 💡 Realization

The contract never says:

> Use Uniswap.

or

> Use SushiSwap.

It simply uses **whatever router addresses Bob supplied inside `SwapParams`.**

---

# `tokenIn`

Initially we asked:

Is this the pair?

The router?

Some address?

---

## 💡 Realization

No.

`tokenIn` is simply the ERC20 token that starts the arbitrage.

Example:

```text
tokenIn

↓

DAI
```

Meaning:

The first swap begins with DAI.

---

# `tokenOut`

Another question we asked was:

Is this the final output of the arbitrage?

---

## 💡 Realization

No.

This was an important realization.

`tokenOut` is **only the output of the first swap.**

Example:

First swap:

```text
DAI

↓

WETH
```

Here:

```text
tokenIn

↓

DAI

tokenOut

↓

WETH
```

The second swap simply reverses it.

```text
WETH

↓

DAI
```

So the complete arbitrage becomes:

```text
DAI

↓

WETH

↓

DAI
```

Notice that the arbitrage returns back to the original token.

---

# ❓ Question

Why not:

```text
DAI

↓

WETH

↓

XYZ
```

Wouldn't that also work?

---

## 💡 Realization

Technically,

yes.

But then how do we calculate profit?

Example:

```text
100 DAI

↓

20 XYZ
```

Did we make money?

The contract doesn't know.

It would need to know the market price of XYZ.

Instead,

the contract intentionally returns to the original asset.

```text
100 DAI

↓

105 DAI
```

Now profit is obvious.

```text
105

-

100

=

5 DAI
```

This is why `Arb1` performs:

```text
tokenIn

↓

tokenOut

↓

tokenIn
```

instead of ending with another token.

---

# `amountIn`

`amountIn` is simply:

> How much of `tokenIn` Bob wants to start the arbitrage with.

Example:

```text
tokenIn

↓

DAI

amountIn

↓

100 DAI
```

Meaning:

Start the arbitrage using exactly **100 DAI**.

---

# `minProfit`

Initially we described it as:

> The amount we'll calculate at the end.

We then corrected ourselves.

---

## 💡 Realization

The contract calculates the **actual profit**.

Bob chooses the **minimum acceptable profit**.

Suppose Bob says:

```text
amountIn

↓

100 DAI

minProfit

↓

5 DAI
```

After arbitrage:

Scenario 1

```text
100 DAI

↓

108 DAI
```

Profit:

```text
8 DAI
```

Comparison:

```text
8 >= 5
```

Trade succeeds.

---

Scenario 2

```text
100 DAI

↓

103 DAI
```

Profit:

```text
3 DAI
```

Comparison:

```text
3 < 5
```

Trade reverts.

Bob doesn't want to execute an arbitrage that makes too little profit.

---

# Final Understanding

By the end of our discussion we realized that `SwapParams` is much more than just a Solidity struct.

It is the complete **execution plan** for one arbitrage.

It answers every important question before the contract begins:

- Which protocol do I buy from?
- Which protocol do I sell on?
- Which token do I start with?
- Which token do I temporarily receive?
- How much should I trade?
- What's the minimum acceptable profit?

The contract itself never answers those questions.

It simply receives `SwapParams` and executes the plan exactly as instructed.

---

# Key Takeaways

- `SwapParams` is **not required by Solidity**; it is a design choice for cleaner and more maintainable code.
- It groups all information required for one arbitrage execution.
- It is created **off-chain** (by Bob, a bot, or another caller), not by the contract.
- `router0` and `router1` are generic router addresses, not specific Uniswap contracts.
- `tokenIn` and `tokenOut` are ERC20 token addresses, **not Pair addresses**.
- This contract intentionally performs:

```text
tokenIn

↓

tokenOut

↓

tokenIn
```

so that profit can be measured in the same asset that the arbitrage started with.
