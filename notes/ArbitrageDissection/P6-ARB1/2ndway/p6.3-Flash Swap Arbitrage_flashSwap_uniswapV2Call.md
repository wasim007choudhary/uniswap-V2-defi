# Part 4 — Flash Swap Arbitrage (`flashSwap()` + `uniswapV2Call()`)

After completely understanding the normal arbitrage flow, we moved to the second style:

**Flash Swap Arbitrage.**

Unlike the first style, Bob does **not** need to provide the starting capital.

Instead, the Uniswap V2 Pair temporarily lends the tokens to the Arb Contract.

---

# Two Types of Arbitrage in this Contract

The contract implements **two different funding mechanisms**.

## Style 1 — Normal Arbitrage

Starting capital comes from Bob.

```text
Bob

↓

Approve Arb Contract

↓

transferFrom()

↓

Arb Contract owns tokenIn

↓

_swap()

↓

Profit Check

↓

transfer()

↓

Bob receives amount + profit
```

---

## Style 2 — Flash Swap Arbitrage

Starting capital comes from the Pair.

```text
Bob

↓

flashSwap()

↓

Pair lends tokenIn

↓

_swap()

↓

Repay Pair

↓

Profit sent to Bob
```

The arbitrage itself is identical.

Only the funding source changes.

---

# Biggest Mental Model

The biggest realization before reading the code was:

`_swap()` **does not care where the money came from.**

It only assumes:

```text
The Arb Contract currently owns tokenIn.
```

Whether those tokens came from:

- Bob
- Flash Swap
- Another contract

doesn't matter.

This is why `_swap()` can be reused by both arbitrage styles.

---

# Entry Function

```solidity
function flashSwap(
    address pair,
    bool isToken0,
    SwapParams calldata params
) external
```

Bob starts everything by calling:

```text
Bob

↓

flashSwap()
```

Therefore:

```text
msg.sender

↓

Bob
```

Unlike the normal arbitrage,

there is **no**:

```solidity
transferFrom(...)
```

No approval is required.

No user funds are transferred.

Immediately we know:

> The capital will come from somewhere else.

---

# Parameters

## pair

```solidity
address pair
```

This is **not** a Router.

It is the Uniswap V2 Pair contract.

Example:

```text
DAI / WETH Pair
```

This contract is where we borrow the tokens from.

---

## isToken0

The Pair API works with:

```text
token0

token1
```

instead of token addresses.

Therefore we must tell the Pair which side to borrow.

Example:

```text
true

↓

Borrow token0
```

```text
false

↓

Borrow token1
```

---

# Important API Observation

One important realization was that this API is mainly educational.

The caller must already know whether:

```text
tokenIn

↓

token0 ?

or

token1 ?
```

Otherwise they cannot correctly pass:

```text
true

or

false
```

A better production implementation would remove the boolean entirely.

Example:

```solidity
address token0 = IUniswapV2Pair(pair).token0();

bool isToken0 = params.tokenIn == token0;
```

This automatically determines whether the borrowed token is token0 or token1.

Therefore,

a production API would simply receive:

```text
pair

params
```

and compute the rest internally.

---

# Pair Always Contains Both Tokens

Initially we wondered whether the Pair only contains `tokenOut`.

The answer is:

**No.**

A Pair always contains both assets.

Example:

```text
DAI / WETH Pair

Contains:

DAI

and

WETH
```

Either token can be borrowed.

Which one depends entirely on the arbitrage strategy.

---

# Arbitrage Layer vs Pair Layer

One major realization was that these are **different concepts.**

---

## Arbitrage Layer

These belong to our strategy.

```text
tokenIn

↓

Asset we begin arbitrage with.
```

```text
tokenOut

↓

Asset received from the first Router swap.
```

---

## Pair Layer

These belong to the Pair API.

```text
amount0Out

↓

Borrow token0
```

```text
amount1Out

↓

Borrow token1
```

The Pair has no idea what:

```text
tokenIn

tokenOut

routers

profit
```

mean.

It only understands:

```text
token0

token1
```

Our contract simply connects the two layers.

---

# Encoding Callback Data

The next line:

```solidity
bytes memory data =
    abi.encode(msg.sender, pair, params);
```

is one of the most important concepts.

---

# Why encode?

Execution is about to leave our contract.

Flow:

```text
Arb Contract

↓

Pair.swap()

↓

Pair calls us back later
```

When the Pair calls us back,

how do we remember:

- Bob
- Pair
- Routers
- Tokens
- Amount
- Profit requirement

We package everything into one byte array.

---

# What is `data`?

Think of it as a backpack.

Before:

```text
Bob

Pair

SwapParams
```

After:

```text
Bytes

↓

Everything packed together.
```

The Pair never reads this data.

It simply stores it and later returns the exact same bytes.

The Pair acts like a courier.

---

# Important Realization

The Pair **does not care** what is inside `data`.

We could have encoded:

```solidity
abi.encode(
    msg.sender,
    pair,
    params,
    "I am happy",
    123,
    true
);
```

Perfectly valid.

The only rule is:

Encoding and decoding must match.

---

# Decoding Rule

If we encode:

```text
address

address

SwapParams
```

we must decode:

```text
(address,address,SwapParams)
```

Exactly the same order and types.

Otherwise decoding fails.

---

# Calling Pair.swap()

```solidity
IUniswapV2Pair(pair).swap({
    amount0Out: isToken0 ? params.amountIn : 0,
    amount1Out: isToken0 ? 0 : params.amountIn,
    to: address(this),
    data: data
});
```

This is where the Flash Swap officially begins.

---

# Borrowing

Suppose:

```text
token0 = DAI

token1 = WETH
```

Borrow DAI:

```text
amount0Out = 100

amount1Out = 0
```

Borrow WETH:

```text
amount0Out = 0

amount1Out = 0.05
```

Only one side is borrowed.

---

# Could We Rewrite It?

Yes.

Instead of the ternary:

```solidity
amount0Out: isToken0 ? amount : 0
```

we could write:

```solidity
uint amount0Out;
uint amount1Out;

if (isToken0) {
    amount0Out = amount;
} else {
    amount1Out = amount;
}

pair.swap(
    amount0Out,
    amount1Out,
    address(this),
    data
);
```

This is purely a readability preference.

---

# Execution Leaves Our Contract

At this point execution jumps into:

```text
Pair.swap()
```

The Pair executes its **entire** swap function.

Important realization:

It **does not** jump directly to the callback.

All checks still execute:

- amount validation
- reserve validation
- optimistic transfer
- reserve update

We skipped those only because we had already studied Pair.sol.

The EVM still executes every line.

---

# Callback Trigger

Eventually Pair reaches:

```solidity
if (data.length > 0) {
    IUniswapV2Callee(to).uniswapV2Call(...);
}
```

---

# Why check data?

This is how Uniswap distinguishes:

Normal Swap

vs

Flash Swap.

Normal Router swaps send:

```text
data = ""
```

Therefore:

```text
data.length == 0
```

No callback.

Flash Swaps send encoded bytes.

Therefore:

```text
data.length > 0
```

Callback executes.

There is no:

```text
isFlashSwap
```

flag anywhere.

The presence of data alone enables the callback.

---

# Callback

Execution becomes:

```text
Pair

↓

ArbContract.uniswapV2Call(...)
```

This is the first callback we encountered during our Uniswap study.

Execution temporarily returns to our contract before Pair.swap() finishes.

---

# msg.sender Changes

Inside:

```text
flashSwap()
```

```text
msg.sender

↓

Bob
```

Inside:

```text
uniswapV2Call()
```

```text
msg.sender

↓

Pair
```

One of the biggest Solidity concepts is that `msg.sender` changes at every external contract call.

---

# Decoding

```solidity
(address caller,address pair,SwapParams memory params)=
abi.decode(...)
```

This simply opens the package we previously encoded.

After decoding:

```text
caller

↓

Bob
```

```text
pair

↓

Pair
```

```text
params

↓

Original SwapParams
```

Nothing new is created.

The original information is simply reconstructed.

---

# `_swap()` Again

The callback executes:

```solidity
uint256 amountOut = _swap(params);
```

This is the exact same arbitrage algorithm used in the normal arbitrage.

The callback doesn't contain arbitrage logic.

It simply reuses `_swap()`.

This demonstrates good software design.

---

# Why Reuse `_swap()`?

`_swap()` only assumes:

```text
The contract owns tokenIn.
```

It doesn't care whether those tokens came from:

- Bob
- Flash Swap

Therefore the exact same arbitrage engine works for both funding methods.

---

# Repayment

After `_swap()`:

Suppose:

```text
Borrowed

100 DAI
```

After arbitrage:

```text
105 DAI
```

The callback computes:

```solidity
fee

↓

Flash Swap fee

amountToRepay

↓

Borrowed + Fee
```

Example:

```text
Borrowed

100

Fee

0.3009

Repay

100.3009
```

---

# Profit

Profit becomes:

```text
105

-

100.3009

=

4.6991
```

Only now does the contract know how much it actually earned.

---

# Minimum Profit

Exactly like the normal arbitrage,

the callback verifies:

```text
profit >= minProfit
```

Otherwise,

the entire transaction reverts.

---

# Repayment

Repayment occurs using:

```solidity
transfer(pair, amountToRepay)
```

One observation we made:

The decoded variable:

```solidity
address pair;
```

is already an address.

Therefore:

```solidity
transfer(pair,...)
```

and

```solidity
transfer(address(pair),...)
```

are identical.

The explicit cast is unnecessary.

---

# Profit Distribution

Finally,

after repaying the Pair,

the remaining profit is sent to Bob.

Flow:

```text
Pair

↓

Borrow

↓

_swap()

↓

Repay Pair

↓

Send Profit

↓

Bob
```

---

# Complete Flash Swap Flow

```text
Bob
 │
 ▼
flashSwap()
 │
 ▼
Encode

(Bob + Pair + Params)

 │
 ▼
Pair.swap()
 │
 ▼
Pair checks

↓

Optimistic Transfer

↓

Borrowed Tokens Arrive

↓

Callback

↓

Decode

↓

_swap()

↓

Calculate Fee

↓

Repay Pair

↓

Profit Check

↓

Transfer Profit to Bob

↓

↓

Pair verifies reserves

↓

Transaction completes
```

---

# Final Understanding

Flash Swaps are **not** a different arbitrage algorithm.

They are simply a different **funding mechanism.**

Normal Arbitrage:

```text
Capital

↓

Bob
```

Flash Swap Arbitrage:

```text
Capital

↓

Uniswap Pair
```

Everything after obtaining the starting capital is almost identical because both paths reuse the exact same `_swap()` implementation.

This separation of responsibilities makes the contract clean, reusable, and easy to reason about.
