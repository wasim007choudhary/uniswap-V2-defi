# `transferFrom()`

# Part 2 — Where Is The Allowance Actually Checked?

---

# A Question That Confused Me

After understanding unlimited approval, I looked at this code again.

```solidity
if (allowance[from][msg.sender] != type(uint256).max) {
    allowance[from][msg.sender] =
        allowance[from][msg.sender] - value;
}
```

Immediately another question came to my mind.

> **"Where is the contract actually checking that I have enough allowance?"**

I expected to see something like:

```solidity
require(
    allowance[from][msg.sender] >= value,
    "Not Enough Allowance"
);
```

But that code simply doesn't exist.

So where does the check happen?

---

# My Initial Assumption

Initially I thought this line:

```solidity
if (allowance[from][msg.sender] != type(uint256).max)
```

was checking whether the transfer was allowed.

It isn't.

This line is only answering one question:

> **"Should I decrease the allowance?"**

Nothing more.

---

# The Real Check

The actual permission check happens here:

```solidity
allowance[from][msg.sender] =
    allowance[from][msg.sender] - value;
```

This surprised me.

There is no explicit:

```solidity
require(...)
```

Instead,

Solidity performs the check automatically.

---

# Example 1 — Enough Allowance

Suppose:

```text
Allowance = 100

Transfer = 40
```

The subtraction becomes:

```text
100 - 40
```

Result:

```text
60
```

Everything succeeds.

The transfer continues.

---

# Example 2 — Not Enough Allowance

Now suppose:

```text
Allowance = 20

Transfer = 50
```

The subtraction becomes:

```text
20 - 50
```

Can a `uint256` store:

```text
-30 ?
```

No.

Starting from Solidity 0.8,

this causes an **underflow**.

The EVM immediately reverts the transaction.

So although we never wrote:

```solidity
require(
    allowance >= value
)
```

the subtraction itself enforces that rule.

---

# Child Analogy

Imagine Bob has permission to borrow:

```text
3 books
```

He now tries to borrow:

```text
5 books
```

The teacher says:

> "You only had permission for three."

The borrowing stops immediately.

The teacher doesn't need another rule.

Trying to go below zero is already impossible.

That is exactly how Solidity behaves with unsigned integers.

---

# Another Question I Asked

Then I asked:

> **"If the allowance is MAX, does the transfer automatically succeed?"**

The answer is:

**Not necessarily.**

Remember,

there are **two completely different checks**.

---

# Check 1 — Permission

```text
Does the spender have permission?
```

This is determined by:

```text
allowance
```

---

# Check 2 — Ownership

```text
Does the owner actually have enough tokens?
```

This is determined by:

```text
balanceOf
```

---

# Two Security Guards

The easiest way to think about this is to imagine two guards standing at a gate.

---

## Guard 1

Looks at:

```text
Allowance
```

He asks:

> "Are you allowed to spend these tokens?"

---

## Guard 2

Looks at:

```text
Balance
```

He asks:

> "Do these tokens actually exist?"

---

Only if **both** guards say "yes" does the transfer happen.

---

# Example

Suppose:

```text
Balance = 100

Allowance = MAX
```

Router transfers:

```text
40
```

Guard 1:

```text
Permission?

YES
```

Guard 2:

```text
Enough tokens?

YES
```

Transfer succeeds.

---

Now suppose:

```text
Balance = 0

Allowance = MAX
```

Router tries to transfer:

```text
1
```

Guard 1:

```text
Permission?

YES
```

Guard 2:

```text
Enough tokens?

NO
```

The transfer fails.

This was the moment I finally understood:

> **Unlimited approval does not bypass balance checks.**

---

# Where Is The Balance Check?

The balance is checked inside:

```solidity
_transfer(from, to, value);
```

Earlier we studied:

```solidity
balanceOf[from] =
    balanceOf[from] - value;
```

Suppose:

```text
Balance = 0

Transfer = 1
```

The subtraction becomes:

```text
0 - 1
```

Again,

this underflows and Solidity immediately reverts.

So `_transfer()` itself guarantees that you cannot transfer more tokens than actually exist.

---

# The Complete Flow

By now the function became much clearer.

```text
transferFrom()

↓

Unlimited Allowance?

↓

YES

↓

Skip allowance update

↓

_transfer()

↓

Balance checked

↓

Transfer succeeds
```

---

Or:

```text
transferFrom()

↓

Unlimited Allowance?

↓

NO

↓

Decrease allowance

↓

Allowance underflow?

↓

YES

↓

Revert
```

---

Or:

```text
transferFrom()

↓

Allowance OK

↓

_transfer()

↓

Balance underflow?

↓

YES

↓

Revert
```

---

# Final Mental Model

By the end of this discussion, I no longer thought of `transferFrom()` as "one check."

Instead, I understood it as **two independent layers of security**:

1. **Allowance Layer**

   * Is the spender authorized to spend this amount?

2. **Balance Layer**

   * Does the owner actually possess this amount?

Even if the spender has **unlimited approval**, the second layer still prevents transferring tokens that do not exist.

That final realization completely removed my confusion about `type(uint256).max`.
