# UniswapV2ERC20.md

# Part 4.7 — `owner`, `spender`, `value`, `nonce`, and `deadline`

---

# Continuing Inside `abi.encode(...)`

At this point our Permit structure looked like:

```solidity
abi.encode(
    PERMIT_TYPEHASH,
    owner,
    spender,
    value,
    nonces[owner]++,
    deadline
)
```

We already understood:

```text
PERMIT_TYPEHASH
```

It identifies the structure of the Permit.

Now we move to the actual values Alice is authorizing.

---

# `owner`

The first real value is:

```solidity
owner
```

This represents:

> **Who owns the LP tokens?**

Example:

```text
Alice owns

250 LP Tokens
```

Then:

```text
owner

↓

Alice
```

Notice something important.

This is **not** automatically:

```text
msg.sender
```

Unlike `approve()`.

Instead,

it is passed into the function.

Initially this felt strange because we were used to ERC20.

---

# Doesn't That Mean Anyone Can Pretend?

One of the first questions that naturally comes up is:

> **What stops me from calling:**

```solidity
permit(
    Alice,
    Router,
    100,
    ...,
    ...
)
```

even though I am not Alice?

The answer is:

**Nothing stops you from calling it.**

Anyone may call `permit()`.

The security does **not** come from restricting who calls the function.

It comes later from verifying the signature.

If the signature was not produced by Alice,

the transaction will fail.

This was a major shift in thinking.

Unlike normal ERC20,

`permit()` does **not** trust:

```text
msg.sender
```

It trusts:

```text
Digital Signature
```

---

# `spender`

Next:

```solidity
spender
```

This simply answers:

> **Who is being allowed to spend the LP tokens?**

In Uniswap's normal flow:

```text
spender

↓

Router
```

The Router will later call:

```text
transferFrom(...)
```

to move Alice's LP tokens into the Pair contract.

---

# `value`

Next comes:

```solidity
value
```

This is the amount being approved.

Example:

```text
100 LP
```

Exactly like ERC20's `approve()`.

Nothing unusual here.

---

# `deadline`

Next is:

```solidity
deadline
```

We already studied this earlier.

It defines:

> **How long this signature remains valid.**

Example:

```text
Deadline

↓

Tomorrow
```

If the Permit is submitted after tomorrow,

the transaction immediately reverts.

---

# The Most Interesting Field

Now we reach:

```solidity
nonces[owner]++
```

This became one of the biggest discussions.

At first glance,

it looks like:

```text
Read nonce

and

increase nonce
```

all at once.

But why?

---

# Current Example

Suppose Alice currently has:

```text
nonces[Alice]

=

7
```

Now the expression is evaluated.

```solidity
nonces[Alice]++
```

What happens?

---

# Post-Increment

This is **post-increment**.

That means:

1.

Use the current value.

↓

7

2.

Then increase it.

↓

8

Conceptually:

```text
Current Nonce

↓

7

↓

Use 7

↓

Store 8
```

---

# Why Not `++nonces[owner]`?

Another excellent question.

That would be **pre-increment**.

Meaning:

```text
Current

↓

7

↓

Increase

↓

8

↓

Use 8
```

But the signature Alice created used:

```text
Nonce = 7
```

not

```text
Nonce = 8
```

So the contract must hash the old nonce first,

then prepare the next nonce.

Therefore post-increment is exactly what we need.

---

# A Very Important Question We Asked

Suppose execution reaches:

```solidity
nonces[owner]++
```

The nonce becomes:

```text
8
```

Later,

suppose:

```text
ecrecover()

↓

fails
```

Does Alice permanently lose her nonce?

Fortunately,

**no.**

---

# Revert Restores Everything

Ethereum transactions are atomic.

If execution later reverts,

every storage modification is rolled back.

That includes:

```text
Nonce

↓

7

(not 8)
```

Even though the increment happened during execution,

the failed transaction never becomes part of the blockchain.

So the nonce is restored automatically.

---

# Child Analogy

Imagine writing with a pencil.

You change:

```text
7

↓

8
```

Then someone says:

> "The exam is invalid."

The teacher tears up the paper.

Everything you wrote disappears.

The blockchain behaves similarly.

A reverted transaction leaves **no permanent storage changes**.

---

# Mental Picture

By the end of this discussion we understood:

```text
Current Nonce

↓

Used inside the hash

↓

Nonce increased

↓

If transaction succeeds

↓

Keep new nonce

----------------------------

If transaction reverts

↓

Rollback

↓

Original nonce restored
```

---

# Why Is This So Important?

Because it guarantees two things simultaneously.

1.

The current signature uses the correct nonce.

2.

The next successful Permit must use a different nonce.

Without this mechanism,

replay protection would not work correctly.
