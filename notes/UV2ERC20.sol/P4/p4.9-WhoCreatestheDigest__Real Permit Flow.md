# Part 4.9 — Who Creates the Digest? (Real Permit Flow)

---

# The Next Big Question

After understanding how the contract constructs:

```solidity
bytes32 digest
```

we stopped and asked a practical question.

> **Who actually creates this digest first?**

Does:

* the smart contract create it?
* Alice create it?
* the wallet create it?
* the frontend create it?

The answer is:

**Both the wallet and the smart contract create exactly the same digest independently.**

---

# Why Do Both Need It?

Think about the purpose of a digital signature.

Alice signs **one specific message**.

Later, the smart contract must verify that she really signed **that exact same message**.

For verification to work,

both sides must compute an identical digest.

If even one byte differs,

the signature becomes invalid.

---

# The Timeline

Instead of thinking only about Solidity,

we drew the complete real-world flow.

```text
Alice

↓

Clicks

"Remove Liquidity"

↓

Frontend prepares Permit data

↓

Wallet builds digest

↓

Wallet signs digest

↓

Signature

(v, r, s)

↓

Frontend receives signature

↓

Frontend sends transaction

↓

Router

↓

Pair.permit(...)

↓

Pair rebuilds digest

↓

ecrecover()

↓

Signature verified
```

Notice something very important.

The digest is built **twice**.

---

# First Time

The wallet builds:

```text
digest
```

using exactly the same inputs:

* DOMAIN_SEPARATOR
* PERMIT_TYPEHASH
* owner
* spender
* value
* nonce
* deadline

Then it signs that digest using Alice's private key.

---

# Second Time

Much later,

after the blockchain transaction arrives,

the Pair contract builds:

```text
digest
```

again.

Using the exact same algorithm.

If both computations match,

the resulting signature verification succeeds.

---

# Child Analogy

Imagine Alice and the teacher both solve the same math problem independently.

Alice gets:

```text
42
```

The teacher also gets:

```text
42
```

The answers match.

Therefore,

the teacher knows Alice solved the correct problem.

Ethereum works similarly.

The wallet computes one digest.

The contract computes the same digest.

Matching digests allow the signature to be verified.

---

# Does Alice Calculate the Digest?

One of the questions we discussed was:

> **Does Alice manually calculate all of this?**

Absolutely not.

Alice never performs:

* `keccak256`
* `abi.encode`
* `DOMAIN_SEPARATOR`
* `PERMIT_TYPEHASH`

herself.

Instead,

her wallet does everything automatically.

---

# What Alice Actually Sees

From Alice's perspective,

the interaction is usually much simpler.

She clicks:

```text
Remove Liquidity
```

Her wallet opens something similar to:

```text
Sign Typed Data

Owner:
Alice

Spender:
Router

Amount:
100 LP

Deadline:
Tomorrow

[Cancel]

[Sign]
```

Alice simply presses:

```text
Sign
```

Behind the scenes,

the wallet performs all the cryptographic work.

---

# What Happens Inside the Wallet?

Internally,

the wallet performs roughly this sequence.

```text
Read Domain

↓

Read Permit Fields

↓

Build Permit Hash

↓

Build Final Digest

↓

Use Alice's Private Key

↓

Generate Signature
```

All of this happens **off-chain**.

No blockchain transaction is sent.

No gas is paid.

---

# The Signature

After signing,

the wallet produces a signature.

Conceptually:

```text
Signature

↓

65 Bytes
```

This signature contains enough information to later prove:

> **Alice approved this exact Permit.**

---

# Does the Wallet Show `v`, `r`, and `s`?

Another question we discussed was:

> **Does MetaMask show me the digest, `v`, `r`, and `s`?**

Normally,

no.

Most wallets simply return one signature.

For example:

```text
0x8c4d9f...
```

The frontend or supporting libraries then split that signature into:

```text
v

r

s
```

before calling the contract.

Alice usually never sees those individual values.

---

# Does Alice Write Them Down?

We also asked:

> **Does Alice calculate these values herself or write them on paper?**

No.

Nothing like that happens.

Everything is automatic.

The wallet software handles:

* digest creation,
* cryptographic signing,
* signature formatting.

The user simply approves the signing request.

---

# Mental Picture

By the end of this discussion,

our understanding became:

```text
Wallet

↓

Creates digest

↓

Signs digest

↓

Produces Signature

----------------------------

Contract

↓

Creates same digest

↓

Verifies Signature
```

Neither side trusts the other.

They independently perform the same calculation.

If both digests match,

the signature verification succeeds.

---

# Important Observation

Notice that the digest itself is **never sent** to the contract.

Instead,

the contract reconstructs it from the Permit fields.

Only the signature is transmitted.

This guarantees that the contract verifies the signature against data it computed itself, rather than trusting data supplied by the caller.
