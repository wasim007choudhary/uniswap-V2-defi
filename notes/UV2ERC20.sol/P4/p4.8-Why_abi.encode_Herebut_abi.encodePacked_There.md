# Part 4.8 — Why `abi.encode()` Here but `abi.encodePacked()` There?

---

# The Question That Started This Discussion

When we reached:

```solidity
bytes32 digest = keccak256(
    abi.encodePacked(
        "\x19\x01",
        DOMAIN_SEPARATOR,
        keccak256(
            abi.encode(
                PERMIT_TYPEHASH,
                owner,
                spender,
                value,
                nonces[owner]++,
                deadline
            )
        )
    )
);
```

we noticed something interesting.

Inside the code we have:

```text
abi.encode(...)
```

but outside we have:

```text
abi.encodePacked(...)
```

Naturally, we asked:

> **Why use two different encoding functions?**

Wouldn't one of them be enough?

---

# The First Question We Asked

Our very first question was:

> **"Isn't `abi.encodePacked()` impossible to decode?"**

The answer is:

**Generally, yes.**

Packed encoding often loses the boundaries between values.

Because of that,

there is frequently no unique way to recover the original inputs.

---

# Example

Imagine writing:

```text
ABC

+

DEF
```

Packed together.

Result:

```text
ABCDEF
```

Now imagine somebody only gives you:

```text
ABCDEF
```

Can you always tell where the split happened?

Was it:

```text
ABC

DEF
```

or

```text
AB

CDEF
```

or

```text
A

BCDEF
```

You cannot know.

The boundaries disappeared.

That is the general idea behind packed encoding.

---

# Then We Asked Again

At this point we asked another question.

> **"So this can never be decoded, right?"**

Again,

the answer is generally:

**Correct.**

The result of `abi.encodePacked()` is not intended to be decoded back into the original values.

In this code,

it never is.

---

# But Then Another Question

Since nothing is ever decoded,

we immediately wondered:

> **If we're never going to decode it anyway, why not just use `abi.encodePacked()` everywhere?**

This was probably the most important question of this discussion.

---

# At First It Seems Reasonable

Imagine this instead:

```solidity
keccak256(
    abi.encodePacked(
        PERMIT_TYPEHASH,
        owner,
        spender,
        value,
        nonce,
        deadline
    )
);
```

Since we're hashing it immediately,

why not?

At first,

this looks perfectly fine.

---

# The Problem

The issue is **not decoding**.

The issue is **ambiguity before hashing**.

`abi.encode()` creates one unique representation for a given set of values.

`abi.encodePacked()` sometimes allows different inputs to produce the same packed bytes.

Those situations are called encoding ambiguities.

Even if you never decode,

ambiguity before hashing is dangerous.

---

# Child Analogy

Imagine writing two words on paper.

Version A:

```text
CAT

DOG
```

Version B:

```text
CA

TDOG
```

Now remove the space.

Both become:

```text
CATDOG
```

You can no longer distinguish which original pair of words was intended.

That is exactly the type of ambiguity packed encoding can create.

---

# Why Doesn't It Matter Here?

Now look at the outer encoding.

```solidity
abi.encodePacked(

    "\x19\x01",

    DOMAIN_SEPARATOR,

    permitHash

)
```

These are **not arbitrary dynamic values**.

They are:

* a fixed 2-byte prefix,
* a fixed 32-byte hash,
* another fixed 32-byte hash.

Conceptually:

```text
2 Bytes

+

32 Bytes

+

32 Bytes
```

Their boundaries are already known.

Nothing is ambiguous.

So packed encoding is perfectly safe.

---

# Why Isn't the Inner Structure Safe?

Inside:

```solidity
abi.encode(

    PERMIT_TYPEHASH,

    owner,

    spender,

    value,

    nonce,

    deadline

)
```

we are building a structured EIP-712 object.

The EIP-712 specification explicitly defines that structured data must be encoded using:

```text
abi.encode()
```

not packed encoding.

This guarantees that every compliant wallet,

library,

and smart contract

produces exactly the same bytes before hashing.

---

# Another Way To Think About It

The inner object is like filling out an official government form.

There is one correct layout.

Everyone must use that layout.

The outer encoding is different.

It is simply joining together three already-complete pieces:

```text
Header

+

Domain Hash

+

Permit Hash
```

No structure needs to be preserved anymore.

---

# Our Important Realization

At the beginning of this discussion we thought:

> **Maybe `abi.encodePacked()` should always be preferred because it's smaller.**

By the end,

our understanding changed completely.

The choice is **not** about saving a few bytes.

The choice is about using the correct encoding for the job.

---

# Final Mental Picture

The inner encoding:

```text
Build a structured object

↓

abi.encode()

↓

Unique, unambiguous representation

↓

Hash
```

The outer encoding:

```text
Already have:

Header

Domain Hash

Permit Hash

↓

Simply concatenate

↓

abi.encodePacked()

↓

Hash
```

The two encoding functions have different responsibilities.

Neither is "better."

They are used in different places because they solve different problems.

---

The final result of all this work is:
