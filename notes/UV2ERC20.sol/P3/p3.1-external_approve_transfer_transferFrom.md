# UniswapV2ERC20.md

# Part 3.1 — `approve()`, `transfer()`, and `transferFrom()`

---

# Moving from Helper Functions to Public Functions

After understanding:

```text
_mint()

_burn()

_approve()

_transfer()
```

we finally reached the functions that users (or other contracts) actually interact with.

```solidity
function approve(address spender, uint value)
    external
    returns (bool)
{
    _approve(msg.sender, spender, value);
    return true;
}

function transfer(address to, uint value)
    external
    returns (bool)
{
    _transfer(msg.sender, to, value);
    return true;
}

function transferFrom(address from, address to, uint value)
    external
    returns (bool)
{
    if (allowance[from][msg.sender] != uint(-1)) {
        allowance[from][msg.sender] =
            allowance[from][msg.sender].sub(value);
    }

    _transfer(from, to, value);

    return true;
}
```

Immediately, something became obvious.

These functions are surprisingly small.

Why?

Because all of the storage-changing work has already been moved into the helper functions we just studied.

---

# `approve()`

The implementation is only:

```solidity
_approve(msg.sender, spender, value);

return true;
```

---

# The Most Important Observation

Notice carefully:

```solidity
msg.sender
```

is passed as the owner.

The function does **not** accept:

```text
owner
```

as a parameter.

Instead, ERC20 says:

> **The person calling `approve()` automatically becomes the owner.**

---

# Example

Alice calls:

```solidity
approve(router, 100);
```

Inside the function:

```text
msg.sender

↓

Alice
```

So it becomes:

```solidity
_approve(
    Alice,
    Router,
    100
);
```

which updates:

```text
allowance[Alice][Router]

=

100
```

---

# Why Is This Secure?

Imagine if ERC20 allowed:

```solidity
approve(
    owner,
    spender,
    value
);
```

Anyone could do:

```text
approve(

Alice,

MyWallet,

1_000_000
)
```

and give themselves permission to spend Alice's tokens.

That would completely break ERC20.

Instead,

the owner is always taken from:

```text
msg.sender
```

which cannot be forged.

---

# This Became Very Important Later

During our Permit discussion we repeatedly came back to this design.

Because:

```text
approve()

↓

owner = msg.sender
```

the Router cannot simply call `approve()` on behalf of Alice.

Inside the Pair contract:

```text
msg.sender

=

Router
```

not Alice.

This became the exact reason that `permit()` had to be invented.

At this point we didn't fully appreciate it yet, but later everything connected together.

---

# `transfer()`

Next we reached:

```solidity
function transfer(
    address to,
    uint value
)
```

The body is:

```solidity
_transfer(
    msg.sender,
    to,
    value
);
```

Again,

the sender comes automatically from:

```text
msg.sender
```

---

# Example

Alice owns:

```text
100 LP
```

She calls:

```solidity
transfer(
    Bob,
    20
);
```

Internally this becomes:

```solidity
_transfer(

Alice,

Bob,

20
);
```

Afterwards:

```text
Alice

100

↓

80

----------------

Bob

50

↓

70
```

Simple.

---

# `transferFrom()`

The final ERC20 function is more interesting.

```solidity
function transferFrom(
    address from,
    address to,
    uint value
)
```

Unlike `transfer()`,

the caller is **not** necessarily the owner.

Instead,

someone else is attempting to spend the owner's tokens.

---

# Example

Alice approves:

```text
Router

↓

100 LP
```

Later,

the Router executes:

```solidity
transferFrom(

Alice,

Router,

100

);
```

The Pair contract checks:

```text
Does Router have enough allowance?
```

If yes,

the transfer proceeds.

---

# The Allowance Check

The first line is:

```solidity
if (
    allowance[from][msg.sender]
    != uint(-1)
)
```

At first this looked strange.

Why compare against:

```solidity
uint(-1)
```

instead of simply subtracting?

---

# Infinite Allowance

`uint(-1)` was the Solidity 0.5 way of representing:

```text
Maximum uint256
```

Conceptually:

```text
2²⁵⁶ − 1
```

This value is used as a special marker meaning:

```text
Infinite Approval
```

---

# Example

Suppose Alice approves:

```text
Maximum uint256
```

instead of:

```text
100 LP
```

Now the allowance is treated as unlimited.

Every call to:

```solidity
transferFrom(...)
```

can spend tokens,

but the allowance itself never decreases.

---

# Why?

Imagine if we decreased it anyway.

Starting with:

```text
Maximum uint256
```

after one transfer:

```text
Maximum uint256

−

20
```

It would no longer be "infinite."

The whole purpose of an infinite approval is that the user approves once and never needs to approve again.

Therefore Uniswap skips decreasing it.

---

# Normal Allowance

Suppose instead:

```text
Alice

↓

Router

↓

100 LP
```

The Router transfers:

```text
20 LP
```

Now:

```text
Allowance

100

↓

80
```

This is exactly what this line does:

```solidity
allowance[from][msg.sender] =
    allowance[from][msg.sender]
        .sub(value);
```

---

# Finally

Once the allowance is handled,

the function executes:

```solidity
_transfer(
    from,
    to,
    value
);
```

Notice something elegant.

The actual balance movement is delegated to the helper function we already studied.

Again,

there is no duplicated transfer logic.

---

# Design Pattern

At this point we could clearly see Uniswap's architecture.

```text
Public Function

↓

Validation

↓

Helper Function

↓

Storage Update
```

For example:

```text
approve()

↓

_approve()

-----------------------

transfer()

↓

_transfer()

-----------------------

transferFrom()

↓

check allowance

↓

_transfer()
```

Each function focuses on its own responsibility.

---

# Looking Ahead

Although these looked like ordinary ERC20 functions,

they prepared us for the biggest discussion that followed.

When we reached:

```solidity
permit(...)
```

we compared it against `approve()` and realized:

```text
approve()

↓

owner = msg.sender

-----------------------

permit()

↓

owner = recovered signature
```

That single difference is what allows Permit to replace a separate approval transaction while remaining secure.

---

# End of Part 3.1

At this point we had completed every standard ERC20 function inside `UniswapV2ERC20`.

Everything that followed was dedicated to one feature that normal ERC20 tokens do not have:

```text
permit()
```

This function became the largest and most in-depth section of our entire study because it introduced EIP-712, digital signatures, cryptographic verification, replay protection, off-chain approvals, and one-transaction workflows.
