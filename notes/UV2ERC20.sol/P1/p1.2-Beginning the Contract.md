# UniswapV2ERC20.md

# Part 1.2 — Beginning the Contract (ERC20 State Variables)

---

# Opening the Contract

After understanding **why `UniswapV2ERC20` exists** and **how every Pair inherits it**, we finally started reading the contract itself.

The beginning of the contract is:

```solidity
contract UniswapV2ERC20 is IUniswapV2ERC20 {
    using SafeMath for uint;

    string public constant name = 'Uniswap V2';
    string public constant symbol = 'UNI-V2';
    uint8 public constant decimals = 18;
    uint public totalSupply;

    mapping(address => uint) public balanceOf;
    mapping(address => mapping(address => uint)) public allowance;
```

Before discussing Permit, signatures, or anything cryptographic, we first understood these ERC20 variables.

---

# `using SafeMath for uint`

```solidity
using SafeMath for uint;
```

At this point we didn't spend much time on `SafeMath` because we were already familiar with it from previous Solidity learning.

This simply allows writing:

```solidity
totalSupply.add(value);
```

instead of:

```solidity
SafeMath.add(totalSupply, value);
```

Since Uniswap V2 uses Solidity **0.5.16**, overflow checks are **not built into the compiler**, so `SafeMath` is necessary.

---

# `name`

```solidity
string public constant name = "Uniswap V2";
```

We already knew this from ERC20.

Every ERC20 token has a human-readable name.

Examples:

```text
USDC

↓

"USD Coin"

--------------------

LINK

↓

"Chainlink"

--------------------

LP Token

↓

"Uniswap V2"
```

Notice something interesting.

Every LP token created by Uniswap V2 shares the same name:

```text
Uniswap V2
```

The pools are distinguished by the Pair contract itself, not by changing the ERC20 name.

---

# `symbol`

```solidity
string public constant symbol = "UNI-V2";
```

Again, this is standard ERC20 metadata.

Just like:

```text
ETH

USDC

WBTC
```

the LP token also has a symbol.

```text
UNI-V2
```

Again,

every Pair uses this same symbol.

---

# `decimals`

```solidity
uint8 public constant decimals = 18;
```

Exactly like most ERC20 tokens,

LP tokens use:

```text
18 decimals
```

Nothing special here.

---

# `totalSupply`

Now we finally reached the variable that originally brought us here.

```solidity
uint public totalSupply;
```

This immediately answered our earlier question.

> **Where does `totalSupply` inside `_mintFee()` come from?**

Answer:

It comes from here.

Because every Pair inherits `UniswapV2ERC20`,

every Pair automatically owns its own:

```text
totalSupply
```

---

# What Does `totalSupply` Mean?

This is **not**

```text
Total ETH

or

Total USDC
```

It is also **not**

```text
Total liquidity value
```

Instead it means:

> **Total LP tokens currently in existence for THIS Pair.**

---

# Example

Suppose the ETH/USDC pool currently has:

```text
Alice

↓

600 LP

----------------

Bob

↓

300 LP

----------------

Charlie

↓

100 LP
```

Then:

```text
totalSupply

=

1000 LP
```

Notice:

This has absolutely nothing to do with:

```text
500 ETH

or

1,000,000 USDC
```

Those are reserves.

`totalSupply` is only counting LP tokens.

---

# Why Is This Important?

Later,

inside `_mintFee()`,

we saw:

```solidity
uint numerator =
    totalSupply.mul(...);
```

Now that line makes much more sense.

It is using:

> **Current LP token supply**

to calculate how many **new LP tokens** should be minted for protocol fees.

Without understanding what `totalSupply` represented,

that formula would have been very confusing.

---

# `balanceOf`

Next we reached:

```solidity
mapping(address => uint)
public balanceOf;
```

At this point you immediately recognized something.

You said:

> **"I know this. This mapping searches the balance of the LP token holders, like how many LP tokens they own."**

Exactly.

Unlike reserve balances,

this mapping stores:

```text
Wallet Address

↓

LP Tokens Owned
```

---

# Example

Suppose:

```text
Alice

↓

250 LP
```

Internally:

```text
balanceOf[Alice]

↓

250
```

Bob:

```text
balanceOf[Bob]

↓

80
```

Charlie:

```text
balanceOf[Charlie]

↓

670
```

Every liquidity provider has an LP token balance.

---

# Very Important Distinction

During our discussion we emphasized something easy to confuse.

This mapping is **NOT** storing:

```text
ETH Balance
```

or

```text
USDC Balance
```

Instead it stores:

```text
LP Token Balance
```

These are completely different assets.

Example:

Alice might own:

```text
20 ETH

5000 USDC

250 LP Tokens
```

Those are three completely different balances.

`balanceOf` only tracks the last one.

---

# `allowance`

The final state variable in this section is:

```solidity
mapping(address =>
mapping(address => uint))
public allowance;
```

You immediately said:

> **"Allowance too, done."**

Since we already understood ERC20,

we only refreshed the concept.

The mapping stores:

```text
Owner

↓

Spender

↓

Maximum amount
```

Example:

```text
Alice

↓

Router

↓

100 LP
```

Internally:

```text
allowance[Alice][Router]

↓

100
```

Notice again,

this is allowing the Router to spend:

```text
LP Tokens
```

not ETH,

not USDC,

not token0,

not token1.

Only LP tokens.

---

# Mental Picture

At this point our understanding of the ERC20 state variables became:

```text
name
↓

Human-readable name

-----------------------

symbol
↓

Ticker

-----------------------

decimals
↓

Precision

-----------------------

totalSupply
↓

Total LP tokens that exist

-----------------------

balanceOf
↓

How many LP tokens each wallet owns

-----------------------

allowance
↓

How many LP tokens someone else may spend
```

---

# End of Part 1.2

By this point,

every ERC20 storage variable at the top of `UniswapV2ERC20` was fully understood.

The next unfamiliar variables we encountered were no longer standard ERC20 variables.

Instead, we entered the world of **Permit**, beginning with:

```solidity
bytes32 public DOMAIN_SEPARATOR;

bytes32 public constant PERMIT_TYPEHASH;

mapping(address => uint) public nonces;
```

These three variables form the foundation of Uniswap V2's signature-based approval system, and they became the next major topic of our discussion.
