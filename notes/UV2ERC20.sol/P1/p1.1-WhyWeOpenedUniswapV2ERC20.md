# Part 1.1 — Why We Opened `UniswapV2ERC20`

---

# Where were we before this?

Before opening `UniswapV2ERC20`, we were studying the `_mintFee()` function inside `UniswapV2Pair`.

At that point we reached the line:

```solidity
uint numerator = totalSupply.mul(rootK.sub(rootKLast));
```

Everything before this line had already been understood:

* Why protocol fees exist.
* LP fees vs protocol fees.
* Why protocol fees are paid by minting LP tokens.
* `feeTo`
* `feeOn`
* `kLast`
* Why `kLast` is only a historical snapshot.
* Why `rootK` uses reserves.
* Why √k is used.
* Why we compare `rootK > rootKLast`.

Now, while reading the formula, one variable immediately stood out:

```solidity
totalSupply
```

At this point we stopped.

Instead of continuing with the formula, we asked:

> **Where does `totalSupply` even come from?**

We searched through `UniswapV2Pair` and noticed something surprising.

There was **no declaration** like:

```solidity
uint public totalSupply;
```

inside `UniswapV2Pair`.

Naturally, this raised the question:

> If `Pair` doesn't define `totalSupply`, how is `_mintFee()` using it?

Rather than guessing, we decided to pause `_mintFee()` and understand the source of `totalSupply` first.

This led us to open another contract:

```text
UniswapV2ERC20.sol
```

---

# The First Realization

After opening the file, the very first important realization was:

> **`totalSupply` does not belong to `UniswapV2Pair` directly.**

Instead, it comes from another contract.

That contract is:

```text
UniswapV2ERC20
```

Then another question naturally followed.

> **Does the whole Uniswap protocol have one `UniswapV2ERC20` contract, or does every Pair have its own?**

This is one of the most important architectural concepts in Uniswap V2.

---

# Does Uniswap Have One LP Token Contract?

Our first intuition was:

```text
Entire Uniswap

↓

One LP Token Contract
```

But that would immediately create a problem.

Imagine three liquidity pools:

```text
ETH / USDC

WBTC / ETH

USDC / DAI
```

Suppose Alice provides liquidity to **ETH/USDC**.

She receives LP tokens.

Now Bob provides liquidity to **WBTC/ETH**.

If there were only **one** LP token contract for the entire protocol, both Alice and Bob would receive exactly the same LP token.

That would make no sense.

Why?

Because Alice owns part of the **ETH/USDC** pool.

Bob owns part of the **WBTC/ETH** pool.

Those ownership claims are completely different.

They cannot be represented by the same LP token.

---

# The Correct Architecture

Instead, Uniswap creates a completely separate Pair contract for every token pair.

For example:

```text
ETH / USDC

↓

ETH-USDC Pair Contract
```

Another pool:

```text
WBTC / ETH

↓

WBTC-ETH Pair Contract
```

Another:

```text
USDC / DAI

↓

USDC-DAI Pair Contract
```

Each Pair represents exactly one liquidity pool.

---

# Then Where Does `UniswapV2ERC20` Fit?

The next realization was even more important.

Each Pair **inherits** from `UniswapV2ERC20`.

Conceptually:

```text
ETH-USDC Pair
        │
        │ inherits
        ▼
UniswapV2ERC20
```

Another Pair:

```text
WBTC-ETH Pair
        │
        │ inherits
        ▼
UniswapV2ERC20
```

Notice something very important.

There is only **one contract definition** of `UniswapV2ERC20` in the source code.

However, every deployed Pair has **its own copy of the inherited storage**.

This is standard Solidity inheritance.

The code is shared.

The storage is **not**.

---

# What Does This Mean?

Suppose the ETH/USDC Pair has:

```text
totalSupply = 5000 LP
```

That value belongs only to the ETH/USDC Pair.

Meanwhile,

the WBTC/ETH Pair might have:

```text
totalSupply = 1200 LP
```

Completely independent.

Changing one Pair's LP supply does **not** affect another Pair.

Each Pair tracks:

* its own LP token supply,
* its own balances,
* its own allowances,
* its own nonces,
* its own permit data.

Nothing is shared between pools except the contract **logic**.

---

# Mental Picture

Instead of imagining:

```text
Entire Uniswap

↓

One LP Token
```

the correct picture is:

```text
ETH/USDC Pair
    ├── Own totalSupply
    ├── Own balanceOf
    ├── Own allowance
    └── Own LP Token

WBTC/ETH Pair
    ├── Own totalSupply
    ├── Own balanceOf
    ├── Own allowance
    └── Own LP Token

USDC/DAI Pair
    ├── Own totalSupply
    ├── Own balanceOf
    ├── Own allowance
    └── Own LP Token
```

Every Pair behaves as its own ERC20 LP token.

---

# Why Didn't Uniswap Put These Variables Directly Inside `Pair`?

At this point we also discussed why Uniswap didn't simply place:

```solidity
uint public totalSupply;

mapping(address => uint) public balanceOf;

mapping(address => mapping(address => uint)) public allowance;
```

inside `UniswapV2Pair`.

The answer is code reuse.

Every Pair needs identical ERC20 functionality.

Instead of copying hundreds of ERC20 lines into every Pair implementation, Uniswap extracted that logic into a reusable parent contract:

```text
UniswapV2ERC20
```

Then every Pair inherits it.

This provides:

* identical ERC20 behavior,
* less duplicated code,
* easier maintenance,
* cleaner architecture.

---

# End of Part 1.1

At this point we understood **why** we opened `UniswapV2ERC20` and exactly where `totalSupply` comes from.

Only after understanding this architecture did we begin reading the actual `UniswapV2ERC20` source code line by line.
