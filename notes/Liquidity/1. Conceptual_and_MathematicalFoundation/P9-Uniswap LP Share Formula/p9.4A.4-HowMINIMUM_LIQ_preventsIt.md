# 9.4 — The Inflation Attack & Why `MINIMUM_LIQUIDITY` Exists

## 9.4A.4 — Why `MINIMUM_LIQUIDITY` Stops The Attack

---

# 🎯 Goal

In the previous section we successfully derived the complete Inflation Attack.

The attack required all four ingredients:

```text
1. Microscopic LP Supply

+

2. Extremely Valuable LP Tokens

+

3. Bob Deposits Normally

+

4. Solidity Integer Rounding
```

The obvious question now is:

> **How does burning only 1000 LP Tokens stop this entire attack?**

At first glance,

```solidity
MINIMUM_LIQUIDITY = 1000;
```

looks completely arbitrary.

It isn't.

---

# 📌 First, Let's Eliminate A Common Misunderstanding

At this point we naturally asked:

> **"What if Alice simply creates a normal pool like 1 ETH + 1000 USDC and then donates another 1 ETH? Doesn't the attack still work?"**

This is an excellent question.

The answer is:

> **No.**

Let's prove it.

---

# Example 1 — A Normal Pool (Without `MINIMUM_LIQUIDITY`)

Assume Uniswap never implemented

```solidity
MINIMUM_LIQUIDITY.
```

Alice creates a perfectly normal pool.

```text
1 ETH

1000 USDC
```

---

# Step 1 — Raw On-Chain Values

Remember,

the Pair contract never sees

```text
1 ETH
```

It actually receives

```text
1 × 10¹⁸ wei
```

Similarly,

```text
1000 USDC
```

becomes

```text
1000 × 10⁶
```

base units.

So the liquidity calculation becomes

```text
√(10¹⁸ × 10⁹)

≈

31,622,776,601,683
```

LP Tokens.

Alice owns

```text
31,622,776,601,683 LP Tokens.
```

---

# Step 2 — Alice Donates

Now Alice transfers

```text
1 ETH
```

directly into the Pair.

No LP Tokens are minted.

The pool becomes larger.

Each LP Token becomes more valuable.

---

# Step 3 — Bob Arrives

Bob deposits using the correct ratio.

The Pair computes

```text
liquidity

=

amount × totalSupply

/

reserve
```

Since

```text
Total Supply

≈

31,622,776,601,683
```

the result is still an enormous integer.

Bob receives billions or trillions of LP Tokens.

Nothing rounds to zero.

Nothing breaks.

The attack completely fails.

---

# 🧠 Huge Realization

Even **without**

```solidity
MINIMUM_LIQUIDITY
```

this pool was already safe.

Why?

Because the LP Supply is already enormous.

Integer rounding is economically insignificant.

---

# So What Kind Of Pool Is Dangerous?

The attacker does **not** create a normal pool.

Instead,

they intentionally create a pathological pool.

---

# Example 2 — The Actual Attack Pool

The attacker does **not** deposit

```text
1 ETH

+

1 USDC.
```

Instead,

they use the smallest possible on-chain units.

```text
1 wei ETH

+

1 base-unit USDC
```

Notice the difference.

For ETH,

```text
1 wei

=

0.000000000000000001 ETH
```

For USDC,

```text
1 base unit

=

0.000001 USDC
```

These are microscopic amounts.

---

# Initial LP Supply

The Pair computes

```text
√(1 × 1)

=

1
```

Ignoring

```solidity
MINIMUM_LIQUIDITY
```

Alice receives

```text
1 LP Token.
```

Now the attacker has exactly what they wanted.

```text
Tiny LP Supply.
```

---

# Step 2 — Donate Assets

Alice now donates

```text
1,000,000 USDC
```

directly into the Pair.

No LP Tokens are minted.

LP Supply remains

```text
1.
```

That single LP Token now represents ownership of an enormous amount of assets.

---

# Step 3 — Bob Arrives

Bob behaves perfectly.

He deposits using the correct reserve ratio.

The Pair calculates

```text
liquidity

=

amount × totalSupply

/

reserve
```

Suppose

```text
amount

=

10
```

```text
totalSupply

=

1
```

```text
reserve

=

100
```

The calculation becomes

```text
10 × 1 / 100

=

0.1
```

Mathematically,

Bob deserves

```text
0.1 LP Tokens.
```

But Solidity cannot mint fractional integers.

Instead,

```text
0.1

↓

0
```

Bob receives

```text
0 LP Tokens.
```

Yet his assets are already inside the Pair.

Alice still owns

```text
100%
```

of the LP Supply.

She therefore owns

```text
100%
```

of the enlarged pool.

This is the Inflation Attack.

---

# 🚨 Where Does `MINIMUM_LIQUIDITY` Stop It?

Now replay exactly the same attack.

This time,

Uniswap contains

```solidity
uint public constant MINIMUM_LIQUIDITY = 1000;
```

During the first mint,

the Pair computes

```solidity
liquidity =
    sqrt(amount0 * amount1)
    - MINIMUM_LIQUIDITY;
```

Substitute the attacker's values.

```text
√(1 × 1)

=

1
```

Now

```text
1

-

1000
```

is impossible.

The transaction immediately reverts.

The pool is never created.

The attacker never obtains

```text
1 LP Token.
```

The attack dies before it even begins.

---

```text
+

```

```text
31,622,776,601,683
```

LP Tokens.

```text
1000
```

LP Tokens.

Ownership sacrificed is therefore

```text
1000
```

```text

```

For a


Essentially zero.

---

# 🧠 Biggest Realization

Notice what Uniswap achieved.

For honest users,

the cost is essentially invisible.

For attackers,

creating the microscopic LP Supply required for the attack becomes impossible.

That is an extraordinary tradeoff.

---

# What `MINIMUM_LIQUIDITY` Does NOT Do

It does **not**

- stop donations,
- stop price manipulation,
- stop arbitrage,
- change swap pricing,
- modify the constant product formula.

Those mechanisms remain exactly the same.

---

# What `MINIMUM_LIQUIDITY` Actually Does

It guarantees that attackers cannot create pools whose LP Supply is so small that Solidity's integer rounding becomes economically exploitable.

That single guarantee is enough to eliminate the Inflation Attack.

---

# Questions We Resolved

### ❓ Does donating assets defeat `MINIMUM_LIQUIDITY`?

No.

Donations still increase LP Token value.

The protection is against microscopic LP Supplies, not donations themselves.

---

### ❓ Would a normal pool like 1 ETH + 1000 USDC be vulnerable without `MINIMUM_LIQUIDITY`?

No.

The LP Supply is already enormous because calculations use raw on-chain units.

Bob's mint calculation never rounds to zero.

---

### ❓ What kind of pool is actually vulnerable?

A deliberately engineered microscopic pool using the smallest possible token units, such as

```text
1 wei

+

1 base-unit USDC.
```

---

### ❓ Why burn exactly 1000 LP Tokens?

Not because 1000 is mathematically special.

It is a practical engineering choice that prevents microscopic LP Supplies while imposing a negligible cost on honest users.

---

- `MINIMUM_LIQUIDITY` prevents those pathological pools from ever existing.

What initially looked like an arbitrary subtraction,

```solidity

    sqrt(amount0 * amount1)
    - MINIMUM_LIQUIDITY;
```

is actually a carefully engineered security mechanism.

Rather than preventing donations or price manipulation,

it prevents the creation of dangerously small LP Supplies where Solidity's integer rounding could be exploited.

A single constant—

```solidity
MINIMUM_LIQUIDITY = 1000;
```

eliminates an entire class of economic attacks while costing honest liquidity providers almost nothing.
