# 9.4 — The Inflation Attack & Why `MINIMUM_LIQUIDITY` Exists

## 9.4A.3 — The Actual Inflation Attack

---

# 🎯 Goal

In the previous sections we proved:

- Donations alone do not create profit.
- Bob behaves correctly.
- Integer rounding only becomes dangerous when the LP Supply is extremely small.

Now we'll combine those ideas into the complete Inflation Attack.

---

# 📌 The Attacker's Goal

The attacker is **not** trying to manipulate prices.

The attacker is **not** trying to steal tokens directly.

Instead, the attacker wants to create this situation:

```text
Bob deposits assets

↓

Bob deserves LP Tokens

↓

Mint calculation rounds to zero

↓

Bob receives no LP Tokens

↓

Attacker keeps the ownership
```

To make this possible, four ingredients are required.

```text
Tiny LP Supply

+

Very Valuable LP Tokens

+

Bob Deposits

+

Integer Rounding
```

---

# Step 1 — Create A Microscopic Pool

Earlier we proved

```text
Initial LP Supply

=

√(amount0 × amount1)
```

The attacker intentionally wants this value to be as small as possible.

---

# ⚠️ Important Clarification

When we write

```text
amount0 = 1

amount1 = 1
```

we do **not** mean

```text
1 ETH

+

1 USDC
```

We mean the smallest possible on-chain units.

For ETH:

```text
1

=

1 wei

=

0.000000000000000001 ETH
```

For USDC:

```text
1

=

1 base unit

=

0.000001 USDC
```

The Pair contract only works with these raw integer values.

# Step 3 — Bob Arrives

Bob wants to become a liquidity provider.

He uses the Router.

He deposits the correct ratio.

Bob makes absolutely no mistakes.

---

# Step 4 — Pair Calculates Bob's LP Tokens

Earlier we derived the mint formula.

```solidity
liquidity = Math.min(
    amount0 * totalSupply / reserve0,
    amount1 * totalSupply / reserve1
);
```

Now substitute actual values.

Suppose

```text
Total Supply

=

1
```

Current reserves

```text
100

100
```

Bob deposits

```text
10

10
```

The calculation becomes

```text
10 × 1 / 100

=

0.1
```

Mathematically,

Bob deserves

```text
0.1 LP Tokens.
```

However,

Solidity cannot mint fractional integers.

It truncates

```text
0.1

↓

0
```

The second calculation produces the same result.

Therefore

```text
Math.min(0,0)

=

0
```

Finally,

```text
liquidity

=

0
```

---

# 🚨 What Just Happened?

Bob deposited

```text
Real Assets.
```

But he received

```text
0 LP Tokens.
```

# 💬 Our Discussion

At this point you immediately realised what happened and said:

> **"Alice kept all Bob's money because of the rounding problem."**

That is exactly the right intuition.

A slightly more precise statement is:

> **Alice effectively keeps the ownership that Bob should have received.**

---

# Why?

Bob's assets entered the Pair.

Ownership is determined entirely by LP Tokens.

Bob received

```text
0 LP Tokens.
```

Therefore

```text
Bob owns

0%
```

Alice still owns

```text
100%
```

of the LP Supply.

Therefore,

she effectively owns Bob's deposited assets as well.

---

# Why Doesn't This Work On A Normal Pool?

Suppose instead Alice initially created

```text
1 ETH

+

1000 USDC
```

Even **without** `MINIMUM_LIQUIDITY`,

the initial LP Supply would be approximately

```text
31,622,776,601,683
```

LP Tokens because the calculation uses raw token units.

Now Bob's mint becomes

```text
amount × 31,622,776,601,683

/

reserve
```

The result is still an enormous integer.

It cannot round to zero.

The attack fails.

---

# Why Doesn't This Work On Real Uniswap?

Because Uniswap immediately performs

```solidity
liquidity =
    Math.sqrt(amount0 * amount1)
    - MINIMUM_LIQUIDITY;
```

Using the attacker's microscopic pool,

```text
√(1 × 1)

=

1
```

Therefore

```text
1

-

1000
```

is impossible.

The transaction reverts.

The attacker never succeeds in creating a pool with

```text
1 LP Token.
```

---

# 🧠 Biggest Realizations

- The attacker intentionally uses the **smallest possible on-chain units**, not whole tokens.
- Donations alone do not create the exploit.
- The exploit only appears when the LP Supply is microscopic.
- Bob follows the protocol correctly.
- Integer rounding causes Bob to receive zero LP Tokens.
- `MINIMUM_LIQUIDITY` prevents the attacker from ever creating this pathological pool.
