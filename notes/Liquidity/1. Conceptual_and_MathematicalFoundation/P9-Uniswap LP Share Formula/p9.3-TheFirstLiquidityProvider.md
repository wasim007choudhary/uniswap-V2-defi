

# Part 3 — The First Liquidity Provider

# 📌 A Completely Different Situation

Everything we've derived so far assumes one important thing.

> **LP tokens already exist.**

Our formula is:

```text
LP Tokens to Mint

=

(dx/x)

×

T
```

where:

```text
T
```

is the existing LP Token Supply.

This works perfectly...

As long as:

```text
T > 0
```

---

Suppose a brand new pool is being created.

Current state:

```text

=

0
```




> **How many LP Tokens should Alice receive?**

# Can We Use Our Existing Formula?

Let's try.

```text
S

=

(dx/x)

×

T
```

Substitute:

```text
dx

=

100
```

```text
x

=

0
```

```text
T
```

Immediately we get:

```text

```

Impossible.

Division by zero.

---

Even if we ignored that...

```text
Anything

×

0

=

0
```

Alice would receive:

```text
0 LP Tokens.
```

Clearly impossible.

She owns:

```text
100%
```

of the pool.

---

# 🧠 Huge Realization

Our proportional minting formula assumes:

> **There is already an ownership system.**

But the first LP creates that ownership system.

There is nothing to be proportional to.

A completely different formula is required.

---

```text

```

```text
100 / 100

```

Now suppose instead she receives:



Likewise:

```text
1,000,000

1,000,000

```

---

# 🧠 Huge Realization

The absolute number of LP Tokens is arbitrary.

Ownership percentages remain identical.

Exactly like company shares.

A company can have:

```text
100 shares
```

or

```text
1 billion shares.
```

Ownership percentages don't change.

---

# Then Why Doesn't Uniswap Pick A Random Number?

Because there is a beautiful choice available.

Remember our liquidity definition.

```text
L

=

√(xy)
```

Suppose Alice deposits:

```text
100 ETH

100,000 USDC
```

Initial liquidity becomes:

```text
√(100 × 100,000)

=

√10,000,000

≈3162
```

Instead of inventing an arbitrary LP supply...

Uniswap simply defines:

```text
Initial LP Supply

=

Initial Liquidity
```

or

```text
LP Supply

=

√(xy)
```

---

# Why Is This Elegant?

Now:

```text
Liquidity

=

LP Supply
```

at pool creation.

No scaling factor.

No conversion constant.

One unit of liquidity corresponds directly to one LP Token.

---

# This Gives The Famous Formula

```solidity
liquidity = sqrt(amount0 * amount1);
```

Notice something important.

This formula wasn't guessed.

It follows directly from the liquidity definition we spent the previous chapter deriving.

```text
L

=

√(xy)

↓

Initial LP Supply

=

√(xy)
```

The Solidity implementation is simply using the mathematical definition of liquidity.

---

# 💬 Another Discussion

At this point another question naturally appeared.

> **If the first LP receives exactly `√(xy)` LP Tokens...**

Then why does the code immediately subtract:

```solidity
MINIMUM_LIQUIDITY
```

resulting in:

```solidity
liquidity = sqrt(amount0 * amount1) - MINIMUM_LIQUIDITY;
```

and permanently mint those LP Tokens to:

```text
address(0)
```

instead?

Surely that just hurts the first LP.

---

# Clarification

It does slightly reduce the first LP's ownership.

But it isn't done for gas optimization.

It isn't done for rounding.

It isn't done for convenience.

It is a deliberate security mechanism.

# 🧠 Biggest Realizations

- The proportional LP minting formula cannot be used for the first LP.
- The first LP creates the ownership system itself.
- The initial LP supply could technically be any positive number.
- Uniswap chooses the most elegant option:
  ```text
  Initial LP Supply = Initial Liquidity = √(xy)
  ```
