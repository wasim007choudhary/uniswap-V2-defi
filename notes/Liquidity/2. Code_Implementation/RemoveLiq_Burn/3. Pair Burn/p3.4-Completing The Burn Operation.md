# Part III — Completing The Burn Operation

At this point, the Pair has already determined exactly how many tokens
the Liquidity Provider should receive.

```solidity
amount0
amount1
```

have already been calculated using the Burn Share Formula.

However, before returning those assets, the Pair still has a few very
important jobs to perform.

It must:

- verify that the redemption is meaningful,
- destroy the LP tokens,
- transfer the underlying assets,
- synchronize its internal accounting,
- update the protocol fee snapshot,
- and finally record the burn event.

---

# 1. Verifying The Redemption Amount

The function first executes:

```solidity
require(
    amount0 > 0 && amount1 > 0,
    "UniswapV2: INSUFFICIENT_LIQUIDITY_BURNED"
);
```

(or in this implementation)

```solidity
if (amount0 == 0 || amount1 == 0) {
    revert UV2Pair__burn__InsufficientLiquidityBurned();
}
```

## Why Is This Check Needed?

Before destroying the LP tokens, the Pair first verifies that the
Liquidity Provider will actually receive a meaningful amount of both
underlying assets.

If either amount becomes:

```text
0
```

the redemption is rejected.

---

## Example

Suppose the pool contains:

```text
100 ETH

200,000 USDC
```

Total LP Supply:

```text
1,000,000,000 LP
```

Alex burns:

```text
1 LP
```

The calculations become:

```text
amount0

=

1 × 100

/

1,000,000,000

=

0
```

and

```text
amount1

=

1 × 200,000

/

1,000,000,000

=

0
```

Notice something important.

The mathematical answers are **not actually zero**.

They are simply **very tiny decimal values**.

However, Solidity does not support floating point numbers.

Integer division always rounds down.

So values like:

```text
0.0000001
```

become:

```text
0
```

If the Pair continued, Alex would lose LP tokens while receiving
nothing.

Instead, the transaction immediately reverts.

---

## Why Check Both Amounts?

Notice the condition:

```solidity
amount0 > 0 && amount1 > 0
```

Both tokens must be greater than zero.

The Pair refuses situations like:

```text
Token0

↓

0

Token1

↓

15
```

or

```text
Token0

↓

15

Token1

↓

0
```

because an LP token represents ownership of the **entire liquidity
position**, not just one side of the pool.

Redeeming LP tokens should always redeem **both assets**.

---

## Child Analogy

Imagine you own a gift box containing:

```text
Chocolate

+

Candy
```

Your ownership certificate represents a share of the **entire gift
box**.

When you redeem it, the shop should not say:

```text
Here is

0 Chocolate

15 Candy
```

or

```text
15 Chocolate

0 Candy
```

That would not be a proper redemption of your share.

The Pair follows the same idea.

If one side becomes zero, the redemption is rejected.

# 2. Burning The LP Tokens

Next the Pair executes:

```solidity
_burn(address(this), liquidity);
```


## Summary

Now the Pair destroys the LP tokens that it currently owns.

Remember earlier:

```solidity
uint256 liquidity = balanceOf[address(this)];
```

The Router (or a developer calling the Pair directly) already
transferred those LP tokens into the Pair contract.

Those LP tokens represent ownership of the pool.

Before returning the underlying assets, that ownership must first be
removed.

Think of selling a house.

The order is:

```text
Transfer Ownership

↓

Receive Payment
```

Not:

```text
Receive Payment

↓

Transfer Ownership
```

The Pair follows exactly the same principle.

First the ownership disappears.

Only afterwards are the underlying assets returned.

---

## Child Analogy

Imagine you have a movie ticket.

The ticket proves that you have the right to enter the cinema.

When you arrive:

```text
Give Ticket

↓

Ticket Gets Torn

↓

Enter Cinema
```

The staff do not let you enter first and tear the ticket afterwards.

Once the ticket is torn, it no longer exists.

LP tokens work exactly the same way.

```text
LP Tokens

↓

Destroyed

↓

Ownership Removed

↓

Ready To Receive Assets
```
## Understanding How `_burn()` Destroys LP Tokens

While studying `_burn()`, one question naturally comes up.

> **"Why aren't the burned LP tokens transferred to `address(0)`?"**

Many ERC-20 implementations use one of two different burn mechanisms.

Understanding the difference is important.

---

# Burn Method 1 — Transfer Tokens To `address(0)`

Some token contracts implement burning like this:

```solidity
_transfer(msg.sender, address(0), amount);
```

Internally, this becomes something similar to:

```solidity
balanceOf[msg.sender] -= amount;
balanceOf[address(0)] += amount;
```

The result is:

```text
Alice

100 Tokens
        │
        ▼
Transfer

100 Tokens

↓

address(0)
```

Now the zero address owns those tokens.

If someone queried:

```solidity
token.balanceOf(address(0));
```

it would return:

```text
100
```

because `address(0)` is simply another key inside the `balanceOf`
mapping.

It is a perfectly valid Ethereum address:

```text
0x0000000000000000000000000000000000000000
```

The only difference is that nobody owns its private key.

Therefore, tokens sent there are practically unrecoverable.

Notice something important.

The tokens still exist.

Ownership simply changed.

```text
Alice

↓

address(0)
```

The tokens were moved.

They were **not destroyed**.

---

# Burn Method 2 — Destroy The Tokens (Used By Uniswap)

Uniswap's LP token does **not** transfer tokens to the zero address.

Instead, `_burn()` performs something conceptually similar to:

```solidity
balanceOf[from] -= amount;

totalSupply -= amount;

emit Transfer(from, address(0), amount);
```

Notice what is missing.

There is **no** line like:

```solidity
balanceOf[address(0)] += amount;
```

The zero address never receives the tokens.

Instead:

- the owner's balance decreases,
- the total supply decreases.

The tokens simply cease to exist.

---

# So Where Do The Tokens Go?

They go **nowhere**.

They are removed from existence.

Imagine there are:

```text
Total Supply

1,000 LP
```

Alice burns:

```text
100 LP
```

After burning:

```text
Alice

0 LP

↓

Total Supply

900 LP
```

Those 100 LP tokens are not sitting inside another wallet.

They simply no longer exist.

---

# Then Why Does `_burn()` Emit

```solidity
Transfer(from, address(0), amount);
```

Good question.

This often confuses developers.

The answer is:

> **That is only an event.**

It is **not** updating storage.

It is simply notifying wallets, block explorers, analytics platforms,
and indexers that a burn occurred.

It does **not** mean:

```solidity
balanceOf[address(0)] += amount;
```

The event follows the ERC-20 convention for signalling burned tokens.

---

# Child Analogy

Imagine you have:

```text
10 Balloons
```

You pop one.

Now you have:

```text
9 Balloons
```

Where did the popped balloon go?

It didn't move into another room.

It simply no longer exists.

That is exactly how Uniswap's `_burn()` works.

The LP tokens are destroyed rather than transferred.

---

# Why Did `mint()` Use `address(0)` Then?

Earlier in `Pair.mint()` we saw:

```solidity
_mint(address(0), MINIMUM_LIQUIDITY_LOCKED);
```

This is different.

Here, the zero address is passed as the **recipient**.

Inside `_mint()`:

```solidity
balanceOf[address(0)] += MINIMUM_LIQUIDITY_LOCKED;
```

The zero address genuinely receives those LP tokens.

Because nobody controls `address(0)`, those LP tokens are permanently
locked and can never be redeemed.

This is how Uniswap permanently locks the initial
`MINIMUM_LIQUIDITY_LOCKED`.

---

# Key Difference

| UniswapLocking `_mint(address(0), amount)` | UniswapCustom `_burn(from, amount)` |
|-----------------------------|-----------------------|
| `balanceOf(address(0))` increases | `balanceOf(address(0))` does **not** change |
| The zero address actually owns the tokens | Nobody owns the burned tokens |
| Tokens remain in existence but are permanently locked | Tokens are completely removed from existence |
| `totalSupply` increases | `totalSupply` decreases |

---

# Summary

Uniswap's LP token uses the **true burn** approach.

The LP tokens are **not transferred** to `address(0)`.

Instead:

```text
Owner Balance

↓

Decrease

↓

Total Supply

↓

Decrease

↓

Tokens No Longer Exist
```

The only mention of `address(0)` inside `_burn()` is the emitted
`Transfer` event, which simply tells external applications that a burn
has occurred.

Unlike `_mint(address(0), ...)`, the zero address never actually
receives ownership of the burned LP tokens.

---

# 3. Returning The Underlying Assets

Next the Pair executes:

```solidity
_safeTransfer(_token0, to, amount0);

_safeTransfer(_token1, to, amount1);
```

## Summary

Now that the LP tokens have been destroyed, the Pair finally transfers
the underlying assets to the recipient.

Remember:

```text
amount0
```

was already calculated as the Liquidity Provider's proportional share of
Token0.

Likewise,

```text
amount1
```

represents the proportional share of Token1.

These two transfers complete the actual redemption.

The Liquidity Provider has now exchanged:

```text
LP Tokens

↓

Underlying Tokens
```

which is exactly the opposite of what happened during `Pair.mint()`.

---

# 4. Reading The Updated Balances

Next:

```solidity
balance0 = IERC20(_token0).balanceOf(address(this));

balance1 = IERC20(_token1).balanceOf(address(this));
```

## Why Read The Balances Again?

Earlier in the function we already read:

```text
balance0

balance1
```

However, those balances represented the Pair **before** the assets were
transferred.

Since then:

- Token0 has been transferred.
- Token1 has been transferred.

The Pair now owns fewer tokens.

The previous balances are therefore no longer correct.

The Pair simply asks again:

> **"After paying the Liquidity Provider, how many Token0 and Token1 do I actually own now?"**

These new balances represent the Pair's latest holdings.

---

## Child Analogy

Imagine a piggy bank.

Before giving money to your friend, it contains:

```text
100 Coins
```

After giving away:

```text
20 Coins
```

the piggy bank now contains:

```text
80 Coins
```

If you want to know how many coins remain, you count again.

The Pair follows exactly the same idea.

# 6. Updating `kLast`

Next:

```solidity
if (feeOn)
    kLast = uint256(reserve0).mul(reserve1);
```

Notice the comment:

```solidity
// reserve0 and reserve1 are up-to-date
```

This comment is important.

Immediately before this line, `_update()` synchronized the reserves.

That means:

```text
reserve0

and

reserve1
```

now contain the latest values.

The Pair therefore records a new protocol fee snapshot using the updated
reserves.

This only happens when protocol fees are enabled.

If protocol fees are disabled, `kLast` is not used, so there is no need

# 7. Emitting The Burn Event

Finally:

```solidity
emit Burn(msg.sender, amount0, amount1, to);
```

## Summary

The burn operation has already completed.

This event does **not** change the contract's state.

Instead, it informs the outside world that a liquidity removal has
occurred.

Wallets, block explorers, analytics platforms, and indexers can all
listen for this event and display information such as:

- who initiated the burn,
- how much Token0 was redeemed,
- how much Token1 was redeemed,
- who received the underlying assets.

The event serves as an on-chain record of the completed liquidity
removal.

# Chapter Summary

`Pair.burn()` is the complete inverse of `Pair.mint()`.

During `Pair.mint()`:

```text
Underlying Tokens

↓

LP Tokens
```

During `Pair.burn()`:

```text
LP Tokens

↓

Underlying Tokens
```

The Pair first calculates the Liquidity Provider's proportional
(**pro-rata**) share of the pool, verifies that the redemption is
meaningful, destroys the LP tokens representing ownership, returns the
underlying assets, synchronizes its accounting, updates the protocol fee
snapshot when necessary, and finally records the completed liquidity
removal through the `Burn` event.

The result is a fair redemption process where every Liquidity Provider
receives their proportional share of the Pair's **actual current
holdings**.
