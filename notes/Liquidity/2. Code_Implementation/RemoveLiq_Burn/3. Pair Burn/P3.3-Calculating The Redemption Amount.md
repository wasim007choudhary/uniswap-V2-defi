After gathering everything needed:

- the previous reserves,
- the current token balances,
- the LP tokens being redeemed,
- whether protocol fees are enabled,
- and the latest LP token supply,

the Pair is finally ready to answer the most important question.

> **"Given these LP tokens, exactly how many Token0 and Token1 should be returned to the Liquidity Provider?"**

This is the heart of `Pair.burn()`.

---

# The Code

```solidity
amount0 = liquidity * balance0 / _totalSupply; // using balances ensures pro-rata distribution
amount1 = liquidity * balance1 / _totalSupply; // using balances ensures pro-rata distribution
```

These two lines calculate the exact amount of each underlying token that
the Liquidity Provider should receive.

---

# This Formula Should Look Familiar

Earlier, we derived the Burn Share Formula.

> **Complete Mathematical Derivation**
>
> See:
>
> ```text
> notes/Liquidity/
> └── 1. Conceptual_and_MathematicalFoundation/
>     └── 5. Burn Share Formula.md
> ```

There we proved:

```text
Liquidity Returned

=

(S / T)

×

Pool Value
```

Where:

| Symbol | Meaning |
|---------|----------|
| **S** | Pool Shares (LP Tokens) being burned |
| **T** | Total Pool Shares (Total LP Supply) |
| **Pool Value** | Everything currently owned by the pool |

The Solidity code is simply implementing this exact formula.

---

# Mapping The Formula To Solidity

Our mathematical variables become:

| Mathematical Formula | Solidity |
|----------------------|----------|
| **S** | `liquidity` |
| **T** | `_totalSupply` |
| **Pool Value** | `balance0` or `balance1` |

For Token0:

```text
amount0

=

(S / T)

×

Token0 Balance
```

which becomes:

```solidity
amount0 = liquidity * balance0 / _totalSupply;
```

For Token1:

```text
amount1

=

(S / T)

×

Token1 Balance
```

which becomes:

```solidity
amount1 = liquidity * balance1 / _totalSupply;
```

Nothing new is happening mathematically.

The Pair is simply applying the Burn Share Formula separately to each
token.

---

# Why Are There Two Formulas?

Earlier, while deriving the mathematics, we intentionally used a
single-asset pool.

```text
USDC Only
```

That allowed us to focus entirely on ownership.

However, Uniswap V2 pools always contain **two assets**.

For example:

```text
ETH

+

USDC
```

Instead of calculating one redemption amount, the Pair must calculate
two.

One for:

```text
Token0
```

and another for:

```text
Token1
```

The ownership percentage remains exactly the same.

Only the assets are different.

The total LP supply is:

```text
1,000 LP
```

Alex burns:

```text
100 LP
```

Alex is surrendering:

```text
100 / 1000

10%
```

of the ownership.

Therefore Alex should receive:

For Token0:

```text
10%

10 ETH
```

For Token1:

```text
10%

=

20,000 USDC
```

Notice something beautiful.

The Pair never changes the ownership percentage.

It simply applies the same percentage to each asset individually.

---

# Why Doesn't The Pair Calculate One Total Pool Value?

Someone might ask:

> **"Why not calculate the total value of the pool first?"**

For example:

```text
100 ETH

+

200,000 USDC

↓

Total Pool Value
```

Then divide that value proportionally.

The answer is simple.

The Pair does **not** redeem dollar value.

It redeems actual tokens.

The Liquidity Provider expects to receive:

```text
ETH

and

USDC
```

Not:

```text
"$50,000"
```

Each asset must therefore be calculated independently.

---

# Why Does The Pair Use `balance0` And `balance1`?

Notice something important.

The code uses:

```solidity
balance0
balance1
```

It does **not** use:

```solidity
_reserve0
_reserve1
```

At first this may seem strange.

We just read the reserves earlier.

Why ignore them now?

Because a Liquidity Provider owns a percentage of **everything the Pair
currently owns**.

Not a percentage of what the Pair owned during the previous reserve
snapshot.

The balances represent the Pair's **actual current holdings**.

That is what should be redeemed.

---

# Example

Suppose:

```text
Stored Reserves

```

Since the last reserve update, trading fees accumulated.

The Pair now actually owns:

```text

```

Alex should receive:

If the Pair instead used the reserves:

```text

20,000 USDC
```

```text
2,000 USDC
```

That would violate proportional ownership.

---

# What Does

```solidity
// using balances ensures pro-rata distribution
```

Mean?

The comment is saying:

> **"Use the Pair's actual current balances so every Liquidity Provider receives their fair proportional share of everything the Pair currently owns."**

The phrase:

```text
Pro-rata
```

simply means:

> **According to ownership percentage.**

If someone owns:

```text
10%
```

of the LP tokens,

they should receive:

```text
10%

of Token0

and

10%

of Token1
```

No more.

No less.

# Mental Summary

These two lines perform one simple calculation.

```text
Calculate

Ownership Being Redeemed

↓

liquidity

/

totalSupply

↓

Apply That Ownership

To Token0

↓

amount0

↓

Apply The Same Ownership

To Token1

↓

amount1
```

The mathematics are exactly the same as the Burn Share Formula derived
earlier.

The only difference is that Uniswap V2 contains **two assets instead of
one**, so the Pair applies the same ownership percentage separately to
each token.

This guarantees that every Liquidity Provider receives their fair,
proportional (**pro-rata**) share of everything the Pair currently owns.
