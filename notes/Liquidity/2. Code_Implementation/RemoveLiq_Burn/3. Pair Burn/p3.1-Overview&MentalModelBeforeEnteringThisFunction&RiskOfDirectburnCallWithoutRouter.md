# The Function

```solidity
function burn(address to)
```

This function is responsible for **redeeming LP tokens**.

During `Pair.mint()`, the Pair answered the question:

> **"Given these deposited assets, how many LP tokens should I mint?"**

`Pair.burn()` answers the exact opposite question.

> **"Given these LP tokens, how many underlying assets should I return?"**

Think of it like this:

```text
Pair.mint()

Assets

↓

LP Tokens
```

Now we are simply reversing the process.

```text
Pair.burn()

LP Tokens

↓

Underlying Assets
```

Everything inside this function exists to perform that redemption safely
and proportionally.

---

# Before `burn()` Begins

One of the most important things to understand is this:

**`Pair.burn()` does not pull LP tokens from the user.**

By the time this function starts executing, the Pair contract already
owns the LP tokens that are about to be redeemed.

This is exactly the same idea we saw with `Pair.mint()`.

Just as `mint()` assumed the Pair already owned the deposited tokens,
`burn()` assumes the Pair already owns the LP tokens.

---

# Normal Execution Flow (Through the Router)

For almost every user interaction, the execution flow is:

```text
User
        │
        ▼
Router.removeLiquidity()
        │
        ▼
Router computes Pair
        │
        ▼
Router transfers LP Tokens
User ─────────► Pair
        │
        ▼
Pair now owns LP Tokens
        │
        ▼
Router calls

Pair.burn(to)
```

Notice something extremely important.

By the time `burn()` starts executing:

```text
Pair

Already Owns

↓

LP Tokens
```

That fact explains one of the most confusing lines later in the function.

```solidity
uint256 liquidity =
    balanceOf[address(this)];
```

The Pair is simply asking:

> **"How many LP tokens do I currently own?"**

Those are the LP tokens that will be redeemed.

---

# Can `Pair.burn()` Be Called Directly?

Yes.

Many people assume it cannot because the Router is normally used.

However, `Pair.burn()` is a public external function and can absolutely
be called directly.

The only requirement is:

> **The Pair must already own the LP tokens that should be redeemed.**

The Router merely automates that process.

---

# Direct Pair Flow

A developer may completely bypass the Router.

The workflow becomes:

```text
Transfer LP Tokens

↓

Pair

Now Owns LP Tokens

↓

Call

Pair.burn(receiver)
```

As long as the Pair already owns redeemable LP tokens, the redemption
logic works exactly the same.

The Router simply automates:

- approvals,
- token transfers,
- slippage protection,
- user-friendly APIs.

The Pair contains the real redemption logic.

---

# The First Question Everyone Asks

Most developers immediately notice something strange.

The function signature is:

```solidity
burn(address to)
```

There is only one parameter.

Naturally, the first question becomes:

> **"Where do I specify how many LP tokens I want to burn?"**

Surprisingly...

You don't.

There is no parameter like:

```solidity
burn(
    liquidity,
    receiver
);
```

or:

```solidity
burn(
    100 LP,
    receiver
);
```

The only parameter is:

```solidity
to
```

which simply tells the Pair:

> **"Who should receive the redeemed underlying assets?"**

It has absolutely nothing to do with determining how many LP tokens are
burned.

# Alex Example

Suppose Alex owns:

```text
100 LP Tokens
```

Alex decides:

> "I don't want to use the Router. I'll call `Pair.burn()` directly."

Alex opens the function and sees:

```solidity
burn(address to)
```

Immediately Alex wonders:

> **"Where do I tell the Pair to burn my 100 LP?"**

The answer is:

You can't.

The Pair never asks the caller:

```text
How many LP tokens should I burn?
```

Instead, the Pair determines that itself.

Alex must first transfer the LP tokens into the Pair.

```text
Alex

↓

Transfer

100 LP

↓

Pair

Now Owns

100 LP
```

Only after that does Alex call:

```solidity
pair.burn(Alex);
```

Notice what Alex passed.

```text
Alex
```

is simply the recipient of the redeemed assets.

It is **not** telling the Pair:

> Burn Alex's LP tokens.

Those are two completely different concepts.

The Pair never even checks Alex's wallet.

Instead, it asks itself:

> **"How many LP tokens do I currently own?"**

That question is answered by:

```solidity
balanceOf[address(this)]
```

If the answer is:

```text
100 LP
```

then:

```text
100 LP

↓

Burned

↓

Underlying Assets

↓

Returned To Alex
```

No liquidity parameter is needed because the Pair already knows exactly
how many LP tokens it owns.

---

# Why Doesn't `burn()` Need a Liquidity Parameter?

Imagine the API looked like this instead.

```solidity
burn(
    100 LP,
    Alex
);
```

The Pair would now have to trust whatever number the caller supplied.

Instead, Uniswap uses a much safer design.

The Pair ignores user input and simply asks:

> **"What do I actually own right now?"**

Reality is always more trustworthy than a user-supplied number.

Instead of trusting:

```text
Caller Says

↓

100 LP
```

the Pair reads:

```solidity
balanceOf[address(this)]
```

which represents the actual LP token balance currently owned by the Pair.

That concludes the mental model required before reading the first line
of code.

The next section begins with:

```solidity

```

where the actual redemption process starts.

`Pair.burn()` behaves exactly the same way.

```text
LP Tokens

↓

Transferred Into Pair
```

```text
Pair.burn()
```

> **"How many LP tokens do I currently own?"**

# A Very Important Question

At this point, many people ask:

> **"What if I only wanted to burn 50 LP, but someone else transferred LP
> tokens into the Pair before I called `burn()`?"**

This is an excellent question.

Let's walk through it.

---

# Example — Alex Wants To Burn 50 LP

Suppose Alex owns:

```text
50 LP
```

Alex decides not to use the Router.

Instead, Alex performs the steps manually.

First:

```text
Alex

↓

Transfer

50 LP

↓

Pair
```

Now the Pair owns:

```text
50 LP
```

So far everything is fine.

---

# Bob Appears

Before Alex calls:

```solidity
pair.burn(Alex);
```

Bob also transfers:

```text
60 LP
```

to the Pair.

The Pair now owns:

```text
Alex

50 LP

+

Bob

60 LP

=

110 LP
```

Notice something important.

Nothing has been burned yet.

The Pair is simply holding:

```text
110 LP
```

---

# Alex Finally Calls `burn()`

Alex now executes:

```solidity
pair.burn(Alex);
```

Inside the function the Pair executes:

```solidity
uint256 liquidity =
    balanceOf[address(this)];
```

What does it return?

Not:

```text
50 LP
```

Instead it returns:

```text
110 LP
```

because that is the Pair's current LP token balance.

The Pair does **not** know:

- Alex originally transferred 50 LP.
- Bob later transferred 60 LP.

It simply knows:

> **"I currently own 110 LP tokens."**

---

# What Happens Next?

Since the Pair believes it owns:

```text
110 LP
```

it performs redemption for:

```text
110 LP
```

The redemption formulas calculate:

```text
amount0

↓

Value Of 110 LP

----------------------

amount1

↓

Value Of 110 LP
```

Then the Pair burns:

```text
110 LP
```

Finally it transfers the redeemed assets to:

```text
Alex
```

because Alex called:

```solidity
pair.burn(Alex);
```

Alex therefore receives the underlying assets represented by all:

```text
110 LP
```

Bob's LP tokens are also redeemed.

---

# Doesn't That Sound Dangerous?

Yes.

If users manually split the workflow into multiple blockchain
transactions, this situation can happen.

That is exactly why almost every user interacts through the Router.

---

# Why Doesn't This Happen Through The Router?

The Router performs:

```text
transferFrom()

↓

burn()
```

inside **one single blockchain transaction**.

The workflow looks like this:

```text
Router

↓

transferFrom(
    Bob,
    Pair,
    50 LP
)

↓

Pair

50 LP

↓

Immediately

↓

Pair.burn()
```

Notice there is **no gap** between those two operations.

---

# Ethereum Executes Transactions Sequentially

Another important concept is how Ethereum executes transactions.

Ethereum does **not** execute multiple transactions simultaneously.

Instead, validators execute them one by one.

For example:

```text
Transaction 1

↓

Completely Finished

↓

Transaction 2

↓

Completely Finished

↓

Transaction 3
```

There is no situation where one transaction pauses halfway while another
transaction runs.

---

# Atomic Transactions

A transaction is **atomic**.

That means:

> Either the entire transaction succeeds...

or:

> The entire transaction reverts.

Nothing can interrupt a transaction while it is executing.

Therefore, during the Router workflow:

```text
transferFrom()

↓

burn()
```

no other transaction can suddenly insert additional LP tokens into the
Pair.

The Pair therefore reads exactly the LP tokens transferred by the
Router.

---

# Direct Calls Are A Low-Level Primitive

`Pair.burn()` is intentionally a very low-level function.

It assumes only one thing.

> **Whatever LP tokens I currently own are the LP tokens that should be
> redeemed.**

It does **not** attempt to determine:

- who transferred them,
- when they were transferred,
- why they were transferred,
- who originally owned them.

Those concerns are handled by higher-level contracts like the Router.

---

# High-Level Execution Flow

Ignoring the mathematical details, `Pair.burn()` performs the following
steps.

```text
Read Previous Reserves
        │
        ▼
Read Current Token Balances
        │
        ▼
Read Pair's LP Balance
        │
        ▼
Recognize Protocol Fees
        │
        ▼
Read Current LP Supply
        │
        ▼
Calculate Redemption Amounts
        │
        ▼
Verify Redemption
        │
        ▼
Burn LP Tokens
        │
        ▼
Transfer Token0
        │
        ▼
Transfer Token1
        │
        ▼
Read Updated Balances
        │
        ▼
Update Reserves
        │
        ▼
Update kLast
        │
        ▼
Emit Burn Event
```

Every line inside the function belongs to one of these stages.

# Summary

Before studying the implementation, remember the following ideas.

- `Pair.burn()` is the inverse of `Pair.mint()`.
- The Pair does **not** pull LP tokens from users.
- By the time `burn()` starts, the Pair already owns the LP tokens that
  will be redeemed.
- The `to` parameter only specifies who receives the underlying assets.
  It does **not** specify how many LP tokens to burn.
- The Pair determines the burn amount by reading its own LP token
  balance using `balanceOf(address(this))`.
- If `burn()` is called directly across multiple transactions, the Pair
  redeems **all** LP tokens it currently owns.
- The Router avoids this issue by performing `transferFrom()` and
  `burn()` atomically within a single transaction.

With this mental model established, the remaining lines inside
`Pair.burn()` become much easier to understand.
