# Part I — Reading the Current State

Before `Pair.burn()` can calculate how many tokens should be returned to
the Liquidity Provider, it must first understand the **current state of
the pool**.

It begins by gathering all of the information it will need for the
redemption process.

Those pieces of information are:

- the previous reserve snapshot,
- the token addresses,
- the Pair's current token balances,
- the Pair's current LP token balance,
- whether protocol fees should be recognized,
- and the latest total LP supply.

Only after collecting all of this information does the Pair calculate
how many underlying tokens should be returned.

---

# 1. Reading The Previous Reserve Snapshot

The function begins with:

```solidity
(uint112 _reserve0, uint112 _reserve1,) = getReserves();
```

>

>
> ```text

## Summary

Before changing anything, the Pair first remembers what the pool looked
like before the burn operation started.

Think of the reserves as a snapshot taken during the previous pool
interaction.

```text
Previous Pool State

↓

reserve0

reserve1
```

The Pair will later use these stored reserves when:

- recognizing protocol fees,
- updating the reserves,
- recording the pool's new state.

Think of a teacher taking attendance.

Before students leave the classroom, the teacher first writes down:

```text
30 Students
```

Later, after students leave:

```text
Before

30 Students

↓

After

25 Students
```

The teacher now knows exactly what changed.

The Pair follows the same idea.

Before anything changes, it first remembers the previous state of the
pool.

The reserves are simply the pool's previous snapshot.

# 2. Cache `token0` And `token1`

Next the Pair executes:

```solidity
address _token0 = token0;
address _token1 = token1;
```

## Summary

These lines exist only to save gas.

`token0` and `token1` are stored in contract storage.

Reading storage repeatedly is more expensive than reading a local
variable.

Instead of repeatedly accessing:

```solidity
token0
```

throughout the function, the Pair copies it once into:

```solidity
_token0
```

The same happens for:

```solidity
_token1
```

The rest of the function now reuses these local variables.

Nothing about the pool changes here.

No balances change.

No reserves change.

No LP tokens change.

The Pair is simply preparing local copies for cheaper access later.

### Child Analogy

Imagine your favorite toy is inside a cupboard.

Instead of opening the cupboard every time you want the toy, you simply
take it out once and keep it beside you until you're finished.

```text
Open Cupboard

↓

Take Toy

↓

Keep It Beside You

↓

Use It Whenever Needed
```

That is exactly what these two lines are doing.

# 3. Reading The Current Token Balances

Next the Pair executes:

```solidity
uint256 balance0 =
    IERC20(_token0).balanceOf(address(this));

uint256 balance1 =
    IERC20(_token1).balanceOf(address(this));
```

## Summary

The Pair now checks how many tokens it actually owns right now.

It asks:

> **"How many Token0 do I currently have?"**

and

> **"How many Token1 do I currently have?"**

These values are the Pair's **actual ERC-20 balances**.

Notice these are **not** the stored reserves.

The reserves represent the previous recorded snapshot.

The balances represent what the Pair physically owns at this exact
moment.

Those values will later be used to determine how many tokens should be
returned to the Liquidity Provider.

### Child Analogy

Imagine a piggy bank.

Yesterday you counted:

```text
100 Coins
```

Today, before giving some coins to your friend, you count again.

```text
Current Coins

↓

105 Coins
```

You always use what is actually inside the piggy bank today—not
yesterday's count.

The Pair follows exactly the same idea.

---

# 4. Reading The Pair's LP Token Balance

Next:

```solidity
uint256 liquidity = balanceOf[address(this)];
```

## Summary

This line asks one very simple question.

> **"How many LP tokens do I currently own?"**

Remember what happened before `burn()` was called.

The Router (or a developer calling the Pair directly) already
transferred LP tokens into the Pair contract.

So if the Pair currently owns:

```text
75 LP
```

then:

```text
liquidity

=

75 LP
```

Those are the LP tokens that will be redeemed.

Notice something important.

The function never asks the caller:

> **"How many LP tokens should I burn?"**

Instead, the Pair simply checks what it already owns.

That is why `burn()` does **not** require a `liquidity` parameter.

Reality is always more trustworthy than user input.

### Child Analogy

Imagine a recycling machine.

People first place bottles inside it.

When the recycle button is pressed, the machine doesn't ask:

> "How many bottles should I recycle?"

Instead it simply looks inside.

```text
Machine

↓

15 Bottles

↓

Recycle All 15
```

The Pair behaves exactly the same way.

It simply redeems the LP tokens it currently owns.

# 5. Recognizing Protocol Fees

Next:

```solidity
bool feeOn = _mintFee(_reserve0, _reserve1);
```

## Summary

Before calculating how many tokens should be returned, the Pair first
checks whether protocol fees should be recognized.

If protocol fees are enabled, `_mintFee()` may mint new LP tokens to the
protocol.

If that happens:

```text
totalSupply
```

changes.

That is why protocol fees must always be recognized **before** any
redemption calculations begin.

Think of a pizza.

Before dividing the pizza among everyone, your mom says:

> "Wait... Grandma gets one slice first."

Only after Grandma receives her slice do you divide the remaining pizza.

The Pair follows the same idea.

First recognize the protocol's ownership.

Then calculate everyone else's share correctly.

---

# 6. Reading The Latest LP Supply

Next:

```solidity
uint256 _totalSupply = totalSupply;
```

## Summary

Now that `_mintFee()` has finished, the Pair reads the latest LP token
supply.

This is important because `_mintFee()` may have minted additional LP
tokens.

If the Pair had read `totalSupply` before `_mintFee()`, the value could
already be outdated.

Instead the Pair waits until `_mintFee()` finishes, then stores the
latest value.

It also caches the value locally to save gas, since it will be used
multiple times in the upcoming calculations.

### Child Analogy

Imagine there are:

```text
100 Pizza Coupons
```

Just before dividing the pizza, the manager prints:

```text
10 New Coupons
```

There are now:

```text
110 Coupons
```

If you still divide the pizza using:

```text
100 Coupons
```

everyone's share becomes incorrect.

So first count the latest number of coupons.

Then divide the pizza.

The Pair follows exactly the same idea.

---

# Summary

Before performing any redemption calculations, `Pair.burn()` first
collects everything it needs.

```text
Read Previous Reserves
        │
        ▼
Cache Token Addresses
        │
        ▼
Read Current Token Balances
        │
        ▼
Read Pair's LP Balance
        │
        ▼
Recognize Protocol Fees
        │
        ▼
Read Latest LP Supply
```

Only after gathering all of this information is the Pair ready to
calculate:

```solidity
amount0

amount1
```

which determine exactly how many underlying tokens should be returned to
the Liquidity Provider.
