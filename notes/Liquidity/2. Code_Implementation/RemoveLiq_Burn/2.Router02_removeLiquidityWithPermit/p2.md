# Who Is Actually Spending The LP Tokens?

One of the most common questions when first reading this function is:

> **"Who is actually spending the LP tokens? The Router or the Pair?"**

The answer is:

> **The Router is the spender.**
>
> **The Pair only receives the LP tokens and later burns them.**

This distinction is extremely important.

---

# Why Doesn't The Pair Need Approval?

Let's follow the execution flow.

Suppose Bob owns:

```text
100 LP Tokens
```

Bob calls:

```solidity
removeLiquidityWithPermit(...)
```

Inside the Router, the first important call is:

```solidity
permit(
    owner,
    spender,
    ...
);
```

Notice the values.

```text
Owner

↓

Bob

---------------------

Spender

↓

Router
```

The Pair is **not** the spender.

Why?

Because the Pair never takes LP tokens from Bob's wallet.

The Router does.

---

# The Router Executes `transferFrom()`

After Permit succeeds, the Router executes:

```solidity
transferFrom(
    Bob,
    Pair,
    100 LP
);
```

Notice carefully.

The Router is saying:

> **"Move Bob's LP tokens into the Pair."**

Since the Router is performing `transferFrom()`, the Router must have an
allowance.

That is exactly what Permit provides.

---

# The Pair Does Not Spend Anything

After the transfer completes:

```text
Bob

0 LP

---------------------

Pair

100 LP
```

Now the Pair already owns the LP tokens.

The Pair no longer needs anyone's permission.

It simply burns **its own** LP tokens.

```text
Pair

100 LP

↓

_burn()

↓

0 LP
```

Then it returns the underlying assets.

---

# Why Is The Router The Spender?

Think about a bank.

Bob owns money inside his account.

Someone must withdraw that money.

The Router acts like the bank employee.

```text
Bob

↓

Signs Authorization

↓

Router

↓

Withdraws LP Tokens

↓

Hands Them To Pair

↓

Pair Burns Them

↓

Returns Assets
```

Who needed permission?

The Router.

The Pair simply receives whatever the Router hands over.

---

# Understanding `msg.sender`

Suppose Bob owns the LP tokens.

Bob calls:

```solidity
removeLiquidityWithPermit(...)
```

Inside the Router:

```text
msg.sender

↓

Bob
```

Therefore Permit becomes:

```text
permit(

owner   = Bob,

spender = Router

)
```

This tells the Pair's ERC-20 implementation:

> **"Bob authorizes the Router to spend Bob's LP tokens."**

Immediately afterward, the Router executes:

```text
transferFrom(

Bob,

Pair,

100 LP

)
```

Since Permit has just granted allowance, this transfer succeeds.

---

# What If Alice Calls Instead?

Suppose Bob owns the LP tokens.

Bob signs the Permit message.

Now Alice tries calling:

```solidity
removeLiquidityWithPermit(...)
```

Inside the Router:

```text
msg.sender

↓

Alice
```

The Router therefore calls:

```text
permit(

owner = Alice,

spender = Router

)
```

However, the submitted signature belongs to Bob.

It does **not** belong to Alice.

During signature verification:

```text
Alice

≠

Bob
```

The signature check fails.

The transaction immediately reverts.

Therefore, in this helper function:

```text
msg.sender

↓

LP Token Owner
```

are expected to be the same address.

---

# `approveMax`

The function computes:

```solidity
uint256 value =
    approveMax
        ? type(uint256).max
        : liquidity;
```

There are two possible behaviours.

## `approveMax = false`

Only the specified LP amount is approved.

```text
Approve

100 LP

↓

Spend

100 LP

↓

Allowance Exhausted
```

The next liquidity removal requires another approval (or another Permit).

---

## `approveMax = true`

The Router receives an effectively unlimited allowance.

```text
Approve

Unlimited

↓

Spend

100 LP

↓

Remaining Allowance

Still Unlimited
```

Future liquidity removals can reuse the existing allowance without
requiring another Permit signature.

---

# Complete Execution Flow

Ignoring the internal implementation of `removeLiquidity()`, the overall
workflow is:

```text
Bob Signs Permit
(No Blockchain Transaction)
        │
        ▼
Bob Calls
removeLiquidityWithPermit()
        │
        ▼
Router Calls permit()
        │
        ▼
Router Receives Allowance
        │
        ▼
Router Calls
removeLiquidity()
        │
        ▼
Router Transfers LP Tokens
to Pair
        │
        ▼
Pair.burn()
        │
        ▼
LP Tokens Burned
        │
        ▼
Underlying Assets Returned
```

---

# Summary

`removeLiquidityWithPermit()` introduces **no new liquidity removal
logic**.

Its only purpose is to eliminate the separate on-chain approval
transaction.

Instead of:

```text
approve()

↓

removeLiquidity()
```

it performs:

```text
permit()

↓

removeLiquidity()
```

inside a single blockchain transaction.

The Router is the contract that spends the user's LP tokens.

The Pair simply receives those LP tokens, burns them, and returns the
underlying assets.

After `permit()` completes, execution follows the exact same
`removeLiquidity()` workflow that has already been fully dissected.
