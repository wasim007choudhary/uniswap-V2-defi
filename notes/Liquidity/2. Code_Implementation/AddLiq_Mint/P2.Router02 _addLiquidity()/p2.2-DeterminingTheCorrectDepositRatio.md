#  Router._addLiquidity()

# Part 2 — Determining The Correct Deposit Ratio

---

# Picking Up Where We Left Off

In Part 1, the Router finished its preparation.

At this point it knows:

- The Pair exists.
- The current reserves.
- The user's desired amounts.
- The user's minimum acceptable amounts.

Now it must answer the most important question inside `_addLiquidity()`:

> **Given the current pool ratio, how many tokens should actually be deposited?**

---

# First Decision

The Router immediately checks:

```solidity
if (reserveA == 0 && reserveB == 0) {
    (amountA, amountB) = (amountADesired, amountBDesired);
} else {
    ...
}
```

Notice what the Router is asking.

Not:

> "Is this a newly created Pair?"

Instead:

> **"Does this Pair currently have any liquidity?"**

Those are different questions.

A Pair contract may exist...

Yet still have zero liquidity.

---

# Why Check Both Reserves?

The Router checks:

```solidity
reserveA == 0

&&

reserveB == 0
```

instead of checking only one reserve.

Why?

Because a valid Uniswap V2 pool can only exist in two meaningful states.

---

## State 1

Brand-new pool.

```text
Reserve A = 0

Reserve B = 0
```

---

## State 2

Existing liquidity.

```text
Reserve A > 0

Reserve B > 0
```

---

A state like

```text
Reserve A = 0

Reserve B = 1000
```

should never occur during normal operation.

Liquidity is always added and removed proportionally.

---

# If Both Reserves Are Zero

The Router executes:

```solidity
(amountA, amountB) =
(
    amountADesired,
    amountBDesired
);
```

Notice something surprising.

There is

- no calculation,
- no quote(),
- no ratio adjustment,
- no optimization.

The Router simply accepts exactly what the user requested.

---

# Why?

Because there is no existing price yet.

No previous LP exists.

No reserve ratio exists.

Nothing has to be preserved.

---

# Connection To Part I

Earlier we spent an entire chapter proving:

> **The first liquidity provider chooses the initial price.**

Suppose Alice creates the pool with:

```text
100 ETH

100,000 USDC
```

The initial price becomes:

```text
1 ETH = 1000 USDC
```

But Alice could also choose:

```text
100 ETH

200,000 USDC
```

Now the initial price becomes:

```text
1 ETH = 2000 USDC
```

Or:

```text
100 ETH

50,000 USDC
```

giving

```text
1 ETH = 500 USDC
```

All of these are perfectly valid.

Because no market price exists inside the pool yet.

The first liquidity provider defines it.

---

# Our Earlier Discussion

This directly matches what we derived conceptually.

For the first LP:

- there are no reserves,
- no LP Tokens,
- no ownership to preserve,
- and no existing price.

Therefore there is **no ratio** that must be maintained.

The Router simply trusts the user's desired amounts.

---

# What About `amountAMin` And `amountBMin`?

You might wonder:

> **Why aren't the minimum amounts checked here?**

Because the Router isn't modifying anything.

Suppose the user submits:

```text
Desired

10 ETH

10,000 USDC
```

The Router immediately returns:

```text
10 ETH

10,000 USDC
```

Nothing changed.

Since the Router never adjusted the deposit,

the minimum checks cannot possibly fail.

Those checks only become meaningful when the Router begins changing the user's requested amounts.

That happens inside the `else` branch.

---

# Existing Liquidity

If the pool already contains liquidity,

execution enters:

```solidity
else {
    ...
}
```

Now the Router has a completely different job.

Instead of allowing any ratio,

it must preserve the existing reserve ratio.

---

# First Calculation

The first line is:

```solidity
uint256 amountBOptimal =
    UniswapV2Library.quote(
        amountADesired,
        reserveA,
        reserveB
    );
```

Notice something interesting.

The Router starts with only **one** token.

Specifically:

```text
Token A
```

It asks:

> **"If I accept all of Token A, how much Token B should accompany it?"**

---

# Why Start With Token A?

At first glance this seems arbitrary.

Why not start with Token B?

The answer is simple.

The Router has to start somewhere.

Its first attempt is:

```text
Accept all of Token A

↓

Compute the required Token B
```

If that attempt fails,

it simply flips the calculation later.

---

# We Already Know `quote()`

Earlier we derived the formula used by `quote()`.

```text
amountBOptimal

=

amountADesired × reserveB

────────────────────────

reserveA
```

The purpose of this formula is simply:

> **Maintain the existing reserve ratio.**

>Visit ***notes/Periphery/Library/Library/UV2Library--quote.md*** for complete dissection of the function. Also see the ***natspec*** too

---

# Example

Suppose the pool currently contains:

```text
100 ETH

100,000 USDC
```

The user wants to deposit:

```text
10 ETH

15,000 USDC
```

The Router asks:

> "If I accept all 10 ETH, how much USDC should I actually accept?"

Using `quote()`:

```text
10 × 100,000

────────────

100

=

10,000 USDC
```

Therefore:

```text
amountBOptimal

=

10,000 USDC
```

Notice something important.

The user offered:

```text
15,000 USDC
```

The Router replies:

> **"I only need 10,000."**

---

# Huge Realization

This is the first point where the Router disagrees with the user.

The user says:

```text
I want to deposit

10 ETH

15,000 USDC
```

The Router replies:

```text
No.

To preserve the existing pool price,

I'll only accept

10 ETH

10,000 USDC.
```

This is the entire reason `_addLiquidity()` exists.

---

# Why Can't The Router Accept Everything?

Suppose it accepted:

```text
10 ETH

15,000 USDC
```

The reserves would become:

```text
110 ETH

115,000 USDC
```

The pool price changes from:

```text
1000 USDC / ETH
```

to

```text
115000 / 110

≈1045 USDC / ETH
```

Simply adding liquidity would change the exchange rate.

Earlier we proved that this must never happen.

Liquidity additions should preserve the reserve ratio.

---

# Current Execution Flow

```text
Existing Pool

↓

User Chooses Deposit

↓

Router Accepts All Of Token A

↓

Uses quote()

↓

Computes Required Token B

↓

Checks Whether That Deposit Is Possible
```

---

# Biggest Realizations

- A Pair can exist while still containing zero liquidity.
- The first liquidity provider defines the initial price.
- No reserve ratio exists for the first LP.
- Therefore the Router accepts the desired amounts directly.
- Once liquidity already exists, every new deposit must preserve the reserve ratio.
- The Router begins by accepting all of Token A and computing the required amount of Token B using `quote()`.
- `_addLiquidity()` exists solely to ensure that liquidity additions do not move the pool price.
