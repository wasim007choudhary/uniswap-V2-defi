# 💬 Important Question

One natural question that may arise while reading `_addLiquidity()` is:

> **What if both branches fail?**

The Router first tries to use all of **Token A**.

If that fails...

It switches to using all of **Token B**.

But what if **that** also fails?

Wouldn't the function simply reach the end without assigning `amountA` and `amountB`?

Surprisingly...

**This situation is mathematically impossible.**

---

# Example

Suppose the current pool contains:

```text
100 ETH

100,000 USDC
```

A user wants to deposit:

```text
10 ETH

5,000 USDC
```

---

# First Attempt

The Router begins by assuming:

> **"Let's use all of Token A."**

It computes:

```text
amountBOptimal

=

10,000 USDC
```

Now it checks:

```text
amountBOptimal

≤

amountBDesired
```

Substituting:

```text
10,000

≤

5,000
```

This is:

```text
False
```

So the first attempt fails.

---

# Second Attempt

Now the Router flips the calculation.

Instead of using all of Token A,

it uses all of Token B.

Using `quote()` again:

```text
amountAOptimal

=

5 ETH
```

Now notice the assertion:

```solidity
assert(amountAOptimal <= amountADesired);
```

Substituting:

```text
5

≤

10
```

This is always true.

---

# Why Is This Guaranteed?

Think about what happened.

The first attempt failed because:

> **The user did not have enough Token B to support all of Token A.**

So instead the Router says:

> **"Fine. I'll simply use all of Token B instead."**

Since we are now using **less Token B** than before,

the corresponding amount of Token A required also becomes smaller.

Therefore:

```text
amountAOptimal

≤

amountADesired
```

must always hold.

---

# Visual Intuition

```text
Attempt 1

Use all Token A

↓

Need too much Token B

↓

Impossible
```

So the Router switches strategy.

```text
Attempt 2

Use all Token B

↓

Need less Token A

↓

Always possible
```

The second branch is not simply another random attempt.

It is the mathematically guaranteed fallback.

---

# Then Why Use `assert()`?

Notice that the Router writes:

```solidity
assert(amountAOptimal <= amountADesired);
```

instead of

```solidity
require(...);
```

This is a very deliberate choice.

The Router is **not validating user input**.

Instead, it is validating **its own mathematical assumption**.

The developers are effectively saying:

> **"If this assertion ever fails, then our algorithm is wrong."**

It is an internal correctness check.

---

# What Errors Can Actually Occur?

Only three realistic failures exist inside `_addLiquidity()`.

---

## 1. Token B Falls Below The User's Minimum

```solidity
require(
    amountBOptimal >= amountBMin,
    "UniswapV2Router: INSUFFICIENT_B_AMOUNT"
);
```

Error:

```text
UniswapV2Router: INSUFFICIENT_B_AMOUNT
```

---

## 2. Token A Falls Below The User's Minimum

```solidity
require(
    amountAOptimal >= amountAMin,
    "UniswapV2Router: INSUFFICIENT_A_AMOUNT"
);
```

Error:

```text
UniswapV2Router: INSUFFICIENT_A_AMOUNT
```

---

## 3. The Assertion Fails

```solidity
assert(amountAOptimal <= amountADesired);
```

This should **never** happen.

If it does,

the transaction panics because one of Uniswap's own mathematical invariants has been violated.

This would indicate a bug in the implementation rather than an invalid user input.

---

# 🧠 Biggest Realization

At first glance it appears that `_addLiquidity()` contains two independent branches that could both fail.

It doesn't.

The algorithm is actually:

```text
Try using all Token A
        │
        ├── Success
        │       │
        │       ▼
        │     Done
        │
        └── Failure
                │
                ▼
        Use all Token B
                │
                ▼
        This is mathematically guaranteed
                │
                ▼
        assert() verifies the invariant
```

The `else` branch is **not another attempt that might fail**.

It is the mathematically guaranteed fallback once the first attempt becomes impossible.
