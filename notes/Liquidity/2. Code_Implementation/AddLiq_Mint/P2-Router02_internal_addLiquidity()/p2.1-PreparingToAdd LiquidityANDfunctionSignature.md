#  Router._addLiquidity()

# Part 1 — Preparing To Add Liquidity

---

# 🎯 Goal

In the previous chapter we learned the overall execution flow of adding liquidity.

```text
User

↓

Router.addLiquidity()

↓

Router._addLiquidity()

↓

Transfer Tokens

↓

Pair.mint()

↓

LP Tokens Minted
```

We now zoom into the first function that performs the actual work.

```solidity
function _addLiquidity(...)
```

Surprisingly...

This function **does not add liquidity.**

It doesn't:

- transfer ERC20 tokens,
- mint LP Tokens,
- update reserves,
- or interact with the Pair contract (other than creating it if necessary).

Instead, it solves one very specific problem:

> **Given the user's desired deposit, what amounts should actually be deposited?**

Everything inside this function is simply preparation.

The actual liquidity addition happens later.

---

# Function Signature

```solidity
function _addLiquidity(
    address tokenA,
    address tokenB,
    uint256 amountADesired,
    uint256 amountBDesired,
    uint256 amountAMin,
    uint256 amountBMin
) internal virtual returns (uint256 amountA, uint256 amountB)
```

---

# First Observation

Notice something immediately.

This function is:

```solidity
internal
```

not

```solidity
public
```

or

```solidity
external
```

---

# Why Is It Internal?

Users are never supposed to call this function directly.

Instead, the intended execution flow is:

```text
User

↓

Router.addLiquidity()

↓

Router._addLiquidity()
```

This makes `_addLiquidity()` an internal helper function.

---

# Why Split The Logic?

A natural question is:

> **Why didn't Uniswap simply place all of this logic inside `addLiquidity()`?**

Because the two functions solve completely different problems.

---

## `Router.addLiquidity()`

Responsible for:

- receiving user input,
- transferring ERC20 tokens,
- calling the Pair contract.

It performs the actual interaction.

---

## `Router._addLiquidity()`

Responsible for only one thing:

> **Determining the correct amounts of Token A and Token B that should actually be deposited.**

Notice what it deliberately does **not** do.

It does not:

- transfer tokens,
- mint LP Tokens,
- update reserves,
- calculate LP ownership.

It is purely a planning function.

---

# Looking At The Parameters

```solidity
address tokenA,
address tokenB,
```

These identify the two ERC20 tokens that form the liquidity pair.

Example:

```text
tokenA = WETH

tokenB = USDC
```

---

```solidity
uint256 amountADesired,
uint256 amountBDesired,
```

The important word is:

> **Desired**

These values represent what the user **would like** to deposit.

They are **not guarantees**.

For example:

```text
User wants

10 ETH

15,000 USDC
```

This does **not** mean all of those tokens will actually be deposited.

The Router may later decide to use smaller amounts.

---

# Our Discussion

During our discussion we clarified an important misconception.

Initially it is tempting to think:

> **Desired = the amount that will definitely be deposited.**

That is incorrect.

A better interpretation is:

> **Desired = the maximum amount the user is willing to contribute.**

The Router will never deposit **more** than this amount.

However, it may deposit **less**.

---

# `amountAMin` And `amountBMin`

```solidity
uint256 amountAMin,
uint256 amountBMin,
```

These parameters answer a completely different question.

Not:

> **"How much do I want to deposit?"**

Instead:

> **"What is the smallest actual deposit I am willing to accept after the Router finishes its calculations?"**

These values are provided by the user.

They are **slippage protection**.

---

# Our Discussion

You asked:

> **"These are set by the user, right?"**

Yes.

The user chooses these values.

After the Router computes the actual deposit amounts...

If either amount falls below the user's minimum...

The transaction immediately reverts.

---

# Another Discussion

You summarized it as:

> **"So in short I must deposit this much at least or revert?"**

That is an excellent way to think about it.

A slightly more precise statement is:

> **The Router computes the actual deposit first.**

Then it checks:

```text
Actual Deposit

≥

Minimum Deposit
```

If not...

The transaction reverts.

---

# Relationship Between Desired, Minimum And Actual

During our discussion another question came up.

Suppose:

```text
Desired = 10 ETH

Minimum = 9 ETH
```

What values can the Router actually choose?

The answer is:

```text
9 ETH

↓

9.5 ETH

↓

9.9 ETH

↓

10 ETH
```

All of these are valid.

But it can **never** return:

```text
11 ETH
```

---

# Why?

You asked:

> **"Say desired is 10. If it goes to 11 then revert?"**

The answer is:

It can never happen.

`amountADesired` is the **maximum** amount the Router is allowed to use.

Therefore the actual deposit always satisfies:

```text
amountAMin

≤

amountA

≤

amountADesired
```

The Router never deposits more than the user authorized.

---

# Return Values

The function returns:

```solidity
returns (
    uint256 amountA,
    uint256 amountB
)
```

Notice what is **not** returned.

```solidity
liquidity
```

is absent.

---

# Why Doesn't It Return LP Tokens?

Because this function has absolutely no idea how many LP Tokens should be minted.

Only the Pair contract can determine LP ownership.

Remember one of the biggest realizations from Part I:

> **The Router never computes LP ownership.**

Instead...

The Router only determines:

```text
How many Token A?

How many Token B?
```

The Pair later determines:

```text
How many LP Tokens?
```

---

# First Executable Block

The first thing the Router checks is:

```solidity
if (IUniswapV2Factory(factory).getPair(tokenA, tokenB) == address(0)) {
    IUniswapV2Factory(factory).createPair(tokenA, tokenB);
}
```

Notice something interesting.

The Router doesn't immediately calculate deposit amounts.

Instead it first asks:

> **"Does a Pair contract even exist?"**

If not...

It creates one.

>Visit ***notes/Factory/2.Factory Storage/P2-getPair&allPair.md*** for Complete dissection of ***getPair()*** function

>Visit ***notes/Factory/4.Functions/P2-createPair.md*** for complete dissection of the Factory function ***createPair()*** and also read its natspec!
---

# Why Compare Against `address(0)`?

`getPair()` returns the Pair contract address.

If no Pair has been created yet...

It returns:

```text
address(0)
```

which simply means:

> **No Pair exists yet.**

---

# If No Pair Exists

The Router tells the Factory:

```text
Create a brand-new Pair contract for these two tokens.
```

Notice something important.

The Router itself never deploys Pair contracts.

Only the Factory is allowed to do that.

The execution flow becomes:

```text
Router

↓

Factory.createPair(...)

↓

New Pair Contract
```

---

# Why Doesn't The User Create The Pair First?

You could imagine requiring users to perform two separate transactions.

```text
Create Pair

↓

Add Liquidity
```

That would work.

But it would provide a poor user experience.

Instead...

The Router automatically creates the Pair when necessary and immediately continues.

From the user's perspective everything happens inside one transaction.

---

# Reading The Current Reserves

After ensuring the Pair exists...

The Router reads the reserves.

```solidity
(uint256 reserveA, uint256 reserveB) =
    UniswapV2Library.getReserves(
        factory,
        tokenA,
        tokenB
    );
```

We already studied `getReserves()` earlier, so we won't repeat its internal implementation.
>Visit ***notes/Periphery/Library/Library/UV2library--getResrerves.md*** and ***notes/Core/UV2Pair-getReserves.md*** for complte dissection of it. Also see the ***function natspecs***!

The important point is that the Router now knows the pool's current reserve ratio.

For example:

```text
100 ETH

100,000 USDC
```

These reserves become the mathematical reference for the remainder of `_addLiquidity()`.

---

# Why Read Reserves?

Because we already proved in Part I that liquidity additions must preserve the existing reserve ratio.

The Router needs the current reserves so it can determine the correct deposit ratio for the new liquidity provider.

Without these values...

It would have no way to compute the optimal deposit amounts.

---

# Current Execution Flow

```text
User

↓

Router.addLiquidity()

↓

Router._addLiquidity()

↓

Pair Exists?

↓

Create Pair (if needed)

↓

Read Current Reserves

↓

Determine Optimal Deposit Amounts
```

---

# Biggest Realizations

- `_addLiquidity()` does **not** add liquidity.
- It is a planning function.
- `amountADesired` and `amountBDesired` are the **maximum** amounts the user is willing to contribute.
- `amountAMin` and `amountBMin` are the **minimum acceptable actual deposits** after the Router performs its calculations.
- The Router can reduce the deposited amount, but it can never exceed the user's desired amounts.
- `_addLiquidity()` never computes LP Tokens.
- The Router automatically creates the Pair if it does not already exist.
- Before performing any calculations, the Router first retrieves the pool's current reserves.
