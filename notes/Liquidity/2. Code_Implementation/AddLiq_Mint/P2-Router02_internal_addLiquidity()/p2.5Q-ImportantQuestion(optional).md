# 💬 Important Question

# Why Does Uniswap Use `assert()` In The Second Branch But Not In The First?

While reading `_addLiquidity()`, one detail immediately stands out.

The first branch uses a normal `if` statement.

```solidity
uint256 amountBOptimal =
    quote(amountADesired, reserveA, reserveB);

if (amountBOptimal <= amountBDesired) {
    ...
}
```

But later, in the second branch, Uniswap suddenly writes:

```solidity
uint256 amountAOptimal =
    quote(amountBDesired, reserveB, reserveA);

assert(amountAOptimal <= amountADesired);
```

This naturally raises two questions:

> **Why isn't the first comparison also an `assert()`?**

and

> **Why isn't the second comparison simply another `if` statement?**

The answer lies in understanding the difference between **decision making** and **verifying an invariant**.

---

# First Branch — A Business Decision

Consider the first comparison.

```solidity
if (amountBOptimal <= amountBDesired)
```

The Router is asking:

> **"Can I use all of Token A?"**

The answer genuinely depends on the user's inputs.

Sometimes the answer is:

```text
Yes
```

Sometimes it is:

```text
No
```

Both outcomes are completely valid.

---

# Example 1 — Success

Current Pool

```text
100 ETH

100,000 USDC
```

User wants

```text
10 ETH

15,000 USDC
```

The Router computes:

```text
amountBOptimal

=

10,000 USDC
```

Now compare:

```text
10,000

≤

15,000
```

Result:

```text
True
```

The Router can safely use all of Token A.

---

# Example 2 — Failure

Current Pool

```text
100 ETH

100,000 USDC
```

User wants

```text
10 ETH

5,000 USDC
```

The Router computes:

```text
amountBOptimal

=

10,000 USDC
```

Now compare:

```text
10,000

≤

5,000
```

Result:

```text
False
```

Nothing is wrong.

Nothing unexpected happened.

The user simply didn't offer enough Token B.

The Router now switches to another strategy.

---

# Why Doesn't It Use `assert()` Here?

Because this comparison is **not checking correctness**.

It is simply making a decision.

The Router is asking:

```text
Can Plan A work?
```

Possible outcomes:

```text
Yes

↓

Continue
```

or

```text
No

↓

Try Plan B
```

Both outcomes are expected.

Therefore this comparison belongs inside a normal:

```solidity
if (...)
```

---

# Second Branch — Verifying An Invariant

Now consider the second comparison.

```solidity
assert(amountAOptimal <= amountADesired);
```

Notice where we are.

We only reach this code because the first attempt already failed.

That means:

```text
The user could not support all of Token A.
```

So the Router intentionally changes its strategy.

Instead of using all of Token A...

It now uses all of Token B.

---

# What Happens Next?

The Router computes:

```text
amountAOptimal
```

This represents:

> **The amount of Token A required if we use all of the user's Token B.**

Now think carefully.

Since we are using **less Token B** than the first attempt required...

The required Token A must also become smaller.

Therefore:

```text
amountAOptimal

≤

amountADesired
```

must always be true.

---

# Why Is This Guaranteed?

Imagine the two execution paths.

---

## First Attempt

```text
Use All Token A

↓

Need Too Much Token B

↓

Impossible
```

So the Router changes strategy.

---

## Second Attempt

```text
Use All Token B

↓

Need Less Token A

↓

Always Possible
```

This is no longer a decision.

It is a mathematical consequence of the previous branch.

---

# Why Use `assert()`?

The Router is no longer asking:

> **"Can this work?"**

Instead it is saying:

> **"This must work."**

If it doesn't...

Something inside the Router itself is broken.

Possible causes include:

- a bug in `quote()`,
- incorrect arithmetic,
- incorrect branching logic,
- or a programming mistake.

It is **not** caused by invalid user input.

Therefore Uniswap uses:

```solidity
assert(...)
```

instead of:

```solidity
require(...)
```

or

```solidity
if (...)
```

---

# Why Not Write Another `if`?

Imagine the code were written like this.

```solidity
if (amountAOptimal <= amountADesired) {
    ...
}
```

This would imply:

> **"Sometimes this condition is true, and sometimes it is false."**

But according to the mathematics...

That is incorrect.

The condition should never legitimately evaluate to `false`.

Writing another `if` would incorrectly suggest that both outcomes are normal program flow.

They are not.

---

# Why Not Use `require()`?

`require()` is typically used to validate:

- user input,
- external contract behavior,
- runtime conditions.

For example:

```solidity
require(
    amountBOptimal >= amountBMin,
    ...
);
```

This can legitimately fail because the user chose the minimum deposit amount.

The user controls that input.

The assertion is different.

The user cannot legitimately make it fail.

If it ever fails...

The Router's own mathematical assumptions have been violated.

That is precisely what `assert()` is designed for.

---

# Decision vs Invariant

The easiest way to remember the difference is:

## First Comparison

```solidity
if (amountBOptimal <= amountBDesired)
```

asks:

> **Can Plan A work?**

Possible answers:

```text
Yes

or

No
```

Both are valid.

---

## Second Comparison

```solidity
assert(amountAOptimal <= amountADesired)
```

states:

> **Plan B must work.**

Possible answers:

```text
True
```

Only.

If it is ever:

```text
False
```

the Router contains a bug.

---

---

# 🧠 Biggest Realization

Although both statements compare values, they serve completely different purposes.

The first comparison is **business logic**.

It determines which execution path the Router should take.

The second comparison is an **internal mathematical invariant**.

It verifies that the Router's own algorithm is behaving exactly as the mathematics predicts.

That difference is why Uniswap intentionally uses:

```solidity
if (...)
```

for the first comparison,

and

```solidity
assert(...)
```

for the second.
