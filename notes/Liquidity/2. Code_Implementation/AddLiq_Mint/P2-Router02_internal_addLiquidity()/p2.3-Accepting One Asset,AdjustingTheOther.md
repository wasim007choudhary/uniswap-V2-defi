# Part 3 — Accepting One Asset, Adjusting The Other

# Picking Up Where We Left Off

At the end of Part 2, the Router had calculated:

```text
amountBOptimal
```

This value answers the question:

> **"If I use all of Token A, how much Token B should accompany it to preserve the current pool ratio?"**

One important question still remains.

> **Can the user actually satisfy that amount?**

That is exactly what the next branch determines.

---

# The Decision

```solidity
if (amountBOptimal <= amountBDesired) {
```

Notice what the Router is comparing.

```text
Required Token B

vs

Maximum Token B The User Was Willing To Deposit
```

It is asking:

> **"Is the required amount of Token B less than or equal to what the user offered?"**

---

# Why Use `<=`?

Suppose the current pool contains:

```text
100 ETH

100,000 USDC
```

The user submits:

```text
10 ETH

15,000 USDC
```

Earlier we calculated:

```text
amountBOptimal

=

10,000 USDC
```

Now the Router checks:

```text
10,000

≤

15,000
```

The answer is:

```text
True
```

The user offered more than enough USDC.

So the Router accepts the calculation.

---

# What If They Are Exactly Equal?

Suppose the user instead submits:

```text
10 ETH

10,000 USDC
```

Now the comparison becomes:

```text
10,000

≤

10,000
```

Still true.

This simply means the user already supplied the perfect ratio.

Nothing needs adjusting.

---

# Why Doesn't The Router Use `==`?

During our discussion this naturally comes up.

Why not require:

```text
Optimal == Desired
```

instead?

Because users frequently provide extra tokens intentionally.

For example:

```text
Desired

10 ETH

20,000 USDC
```

The Router should not reject this transaction.

Instead it should simply ignore the excess.

That is exactly why the comparison is:

```text
Optimal

≤

Desired
```

instead of

```text
Optimal

==

Desired
```

---

# The Slippage Check

If the comparison succeeds,

the Router immediately performs another check.

```solidity
require(
    amountBOptimal >= amountBMin,
    "UniswapV2Router: INSUFFICIENT_B_AMOUNT"
);
```

Notice what is being compared.

Not:

```text
Desired
```

We already know:

```text
Optimal

≤

Desired
```

Instead the Router asks:

> **"Is the actual amount I'm about to use still acceptable to the user?"**

---

# Example

Suppose the user submits:

```text
Desired

10 ETH

15,000 USDC
```

along with

```text
Minimum

9 ETH

11,000 USDC
```

The Router calculates:

```text
amountBOptimal

=

10,000 USDC
```

Now it checks:

```text
10,000

≥

11,000
```

This is false.

The transaction immediately reverts.

---

# Why Revert?

The Router is effectively saying:

> **"The correct reserve ratio requires only 10,000 USDC, but you told me you refuse to deposit less than 11,000 USDC."**

Rather than silently proceeding,

it aborts the transaction.

This is exactly the slippage protection that the user requested.

---

# Our Earlier Discussion

During our conversation we clarified the meaning of the minimum values.

Initially it sounds like:

> **"I must deposit this amount."**

The more precise interpretation is:

> **"If the Router's calculated deposit falls below this amount, revert the transaction."**

The Router computes the optimal deposit first.

Only then does it compare the result against the user's minimum acceptable values.

---

# Final Decision

If every check succeeds,

the Router finally decides:

```solidity
(amountA, amountB) =
(
    amountADesired,
    amountBOptimal
);
```

This is the final answer produced by `_addLiquidity()`.

Notice what happened.

The Router accepted:

- all of Token A,

while reducing:

- Token B.

---

User submits:

```text
10 ETH

```

The Router returns:

```text

amountB

=
```

The remaining

# Huge Realization

The Router follows a simple philosophy.

> **Accept one asset completely.**

Then:

> **Adjust the other asset to preserve the reserve ratio.**

It does **not** partially reduce both assets.

Instead it chooses one asset as the reference and computes the exact amount required for the other.

---

# What If The First Attempt Fails?

Suppose the comparison fails.

```solidity
amountBOptimal <= amountBDesired
```

becomes false.

This means:

```text
Required Token B

>

Desired Token B
```

The user does **not** have enough Token B to support all of Token A.

So the Router simply changes strategy.

Instead of saying:

> **"Use all of Token A."**

it now says:

> **"Fine. I'll use all of Token B instead."**

---

# The Mirror Branch

The `else` block performs exactly the same algorithm with the roles reversed.

```solidity
uint256 amountAOptimal =
    quote(
        amountBDesired,
        reserveB,
        reserveA
    );

assert(amountAOptimal <= amountADesired);

require(
    amountAOptimal >= amountAMin,
    ...
);

(amountA, amountB) =
(
    amountAOptimal,
    amountBDesired
);
```

During our discussion you immediately noticed something.

You said:

> **"The else branch is exact same just for A."**

Exactly.

It is almost a perfect mirror image.

---

# Side-By-Side Comparison

| First Branch | Second Branch |
|--------------|---------------|
| Accept all Token A | Accept all Token B |
| Compute Token B | Compute Token A |
| Check `amountBMin` | Check `amountAMin` |
| Return `(amountADesired, amountBOptimal)` | Return `(amountAOptimal, amountBDesired)` |

The algorithm itself never changes.

Only the roles of the two tokens are swapped.

---

# The Only New Line

The only line that is not simply mirrored is:

```solidity
assert(amountAOptimal <= amountADesired);
```

Unlike `require()`,

this uses `assert()`.

We'll discuss why Uniswap chose `assert()` instead of `require()` when we examine the implementation details, since that is the only genuinely new concept inside the mirrored branch.

# Questions We Resolved

### ❓ Why compare `amountBOptimal <= amountBDesired`?

Because the Router is checking whether the user offered enough Token B to support all of Token A.

---

### ❓ Why use `<=` instead of `==`?

Because users are allowed to offer more than required.

The Router simply ignores the excess.

---

### ❓ Why check against `amountBMin`?

To ensure the Router's calculated deposit is still acceptable to the user.

---

### ❓ What happens to the extra tokens?

Nothing.

They never leave the user's wallet because only the returned amounts are later transferred.

---

### ❓ Is the `else` branch a different algorithm?

No.

It is the exact same algorithm with the roles of Token A and Token B reversed.

This was one of the key observations we made during our discussion.

---

# Biggest Realizations

- The Router first attempts to accept all of Token A.
- If that succeeds, it trims Token B.
- If that fails, it flips the calculation and accepts all of Token B instead.
- The Router never forces the user to deposit more than their desired amounts.
- The Router never deposits less than the user's minimum acceptable amounts.
- Only the final returned amounts are later transferred.
- The `else` branch is simply a mirror image of the first branch.
