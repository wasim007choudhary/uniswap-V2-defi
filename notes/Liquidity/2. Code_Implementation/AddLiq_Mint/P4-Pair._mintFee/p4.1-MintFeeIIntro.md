# `_mintFee()` --- (Part 1)

> **Scope of this chapter**
>
> This chapter covers everything from **why `_mintFee()` exists** up to
> the motivation for `feeTo`. It intentionally **stops before** the
> mathematical fee formula.

------------------------------------------------------------------------

# The `_mintFee()` Function

``` solidity
function _mintFee(uint112 _reserve0, uint112 _reserve1) private returns (bool feeOn)
```

When developers first see this function, the natural assumption is:

> "\_mintFee() collects trading fees."

That is **not** what it does.

Understanding this distinction is essential because the trading fees
have **already been collected** long before `_mintFee()` executes.

The real job of `_mintFee()` is to determine whether the **protocol
itself** should receive a share of the fee-generated growth that has
accumulated inside the pool since the previous liquidity event.

------------------------------------------------------------------------

# Before Looking at the Code

Before reading even the first line of `_mintFee()`, answer one question:

> **What problem is `_mintFee()` trying to solve?**

If you understand the problem, the code becomes much easier to follow.

------------------------------------------------------------------------

# Every Swap Pays a Fee

Suppose a pool initially contains:

  Token     Reserve
  ------- ---------
  ETH           100
  USDC      200,000

A trader swaps **1 ETH** for USDC.

Uniswap V2 charges a **0.30%** trading fee.

The trader still transfers the full **1 ETH** into the pair contract.

However, for pricing purposes, only **0.997 ETH** participates in the
swap calculation.

The remaining:

    0.003 ETH

stays inside the liquidity pool.

After many swaps, these tiny fees accumulate.

Eventually the pool may contain something like:

  Token      Before   After Many Swaps
  ------- --------- ------------------
  ETH           100                105
  USDC      200,000            210,000

No liquidity provider deposited these extra assets.

They came entirely from trading fees.

------------------------------------------------------------------------

# Who Owns Those Extra Assets?

Imagine there are only two liquidity providers.

  LP        Ownership
  ------- -----------
  Alice           50%
  Bob             50%

Initially:

-   Pool: 100 ETH + 200,000 USDC

Each LP effectively owns:

-   50 ETH
-   100,000 USDC

After thousands of swaps:

-   Pool: 105 ETH + 210,000 USDC

Nobody sent these extra assets to Alice or Bob.

Instead, because each still owns **50%** of the pool, they now
effectively own:

-   52.5 ETH
-   105,000 USDC

Their ownership percentage never changed.

The pool simply became more valuable.

This is how LPs earn fees in Uniswap V2.

------------------------------------------------------------------------

# LP Fees Are Automatic

This is one of the most elegant design decisions in Uniswap.

Uniswap does **not** distribute fees after every trade.

Instead:

    Trader swaps
          │
          ▼
    Fee remains inside the pool
          │
          ▼
    Pool reserves increase
          │
          ▼
    Each LP token becomes worth more

There is:

-   No reward distribution.
-   No periodic payout.
-   No per-user accounting.

The LP token itself appreciates because it represents ownership of a
larger pool.

------------------------------------------------------------------------

# Then Why Does `_mintFee()` Exist?

At first glance it seems unnecessary.

If LPs already receive all trading fees automatically, why is another
fee mechanism needed?

Because Uniswap distinguishes between:

1.  **Liquidity Provider Fees**
2.  **Protocol Fees**

------------------------------------------------------------------------

# LP Fees

Every swap pays **0.30%**.

By default:

    100% of that 0.30%
            │
            ▼
    Liquidity Providers

------------------------------------------------------------------------

# Protocol Fees

Governance may decide that the protocol itself should also earn revenue.

Instead of increasing the trading fee, Uniswap simply shares the
existing fee.

The trader still pays:

    0.30%

But now the split becomes:

    Trading Fee
         │
         ▼
    ┌───────────────┐
    │ LPs      5/6  │
    │ Protocol 1/6  │
    └───────────────┘

Which is equivalent to:

-   LPs effectively receive **0.25%**
-   Protocol effectively receives **0.05%**

The trader still pays exactly **0.30%**.

------------------------------------------------------------------------

# Why Doesn't Uniswap Transfer Tokens to the Protocol?

An obvious idea would be:

``` solidity
transfer(protocol, feeAmount);
```

Unfortunately that creates several problems.

Every swap would need extra token transfers.

Removing assets from the pool would also change reserves immediately,
affecting pricing and increasing gas costs.

Instead, Uniswap keeps **every asset inside the pool**.

Nothing leaves.

The protocol is compensated differently.

------------------------------------------------------------------------

# The Key Idea

Instead of taking ETH or USDC directly:

    Pool grows
          │
          ▼
    Protocol receives LP tokens
          │
          ▼
    Protocol now owns part of the pool

Those LP tokens can later be burned to redeem the protocol's
proportional share of the underlying assets.

No reserves have to leave the pool when the fees are recognized.

This keeps the accounting simple, preserves the pool's assets, and
avoids additional token transfers during swaps.

------------------------------------------------------------------------

# Where Is `_mintFee()` Called?

Inside `mint()`:

``` solidity
bool feeOn = _mintFee(_reserve0, _reserve1);
```

Notice something important.

It executes **before** the user's LP tokens are minted.

Why?

Because if protocol fees need to be minted, they must be minted first.

Otherwise the incoming liquidity provider would incorrectly receive part
of the protocol's ownership.

The order is:

    Previous reserves
            │
            ▼
    _mintFee()
            │
            ▼
    Mint protocol LP (if required)
            │
            ▼
    Mint user's LP tokens
            │
            ▼
    Update reserves

This ordering is fundamental to the correctness of the protocol fee
mechanism.
