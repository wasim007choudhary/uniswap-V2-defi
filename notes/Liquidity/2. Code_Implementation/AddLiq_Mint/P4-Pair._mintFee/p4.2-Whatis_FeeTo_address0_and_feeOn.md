# `_mintFee()` --- Part 2: `feeTo`, `address(0)`, and `feeOn`

``` solidity
address feeTo = IUniswapV2Factory(factory).feeTo();
feeOn = feeTo != address(0);
```

------------------------------------------------------------------------

# Reading the First Line

``` solidity
address feeTo = IUniswapV2Factory(factory).feeTo();
```

At first glance this looks like an ordinary variable assignment.

In reality, this line asks a very important question:

> **"If protocol fees are enabled, who should receive them?"**

Notice that the Pair contract does **not** decide this itself.

------------------------------------------------------------------------

# What is `factory`?

Every Pair contract stores:

``` solidity
address public factory;
```

When the Pair is deployed:

``` solidity
constructor() public {
    factory = msg.sender;
}
```

The deployer is always the Factory contract.

Therefore:

    Pair
     │
     ▼
    factory = Factory Address

------------------------------------------------------------------------

# Why Ask the Factory?

Imagine Uniswap has:

-   ETH/USDC
-   WBTC/ETH
-   LINK/ETH
-   UNI/ETH

and thousands more.

If every Pair stored its own treasury address:

    ETH/USDC  -> Treasury A
    WBTC/ETH  -> Treasury A
    ...

Changing the treasury would require updating every Pair.

Instead:

    Factory
       │
       ├── feeTo
       └── feeToSetter
            │
            ▼
    Every Pair simply asks the Factory.

One update inside the Factory automatically affects every Pair.

The Factory becomes the **single source of truth**.

------------------------------------------------------------------------

# Interface Casting

`factory` is only an address.

Addresses do not expose functions by themselves.

So Solidity is told:

``` solidity
IUniswapV2Factory(factory)
```

meaning:

> "Treat this address as an IUniswapV2Factory."

Now the compiler knows the address has a `feeTo()` function.

------------------------------------------------------------------------

# What Does `feeTo()` Return?

The Factory stores:

``` solidity
address public feeTo;
```

Because it is public, Solidity generates:

``` solidity
function feeTo() external view returns (address);
```

So the Pair simply reads that value.

Example:

    Factory

    feeTo

    ↓

    0xABCD...

The local variable becomes:

``` solidity
address feeTo = 0xABCD...;
```

------------------------------------------------------------------------

# What Is `feeTo`?

`feeTo` is **the address that should receive protocol LP tokens**
whenever protocol fees are minted.

It could be:

-   DAO treasury
-   Governance contract
-   Multisig
-   Treasury wallet

The Pair never decides where protocol fees go.

It only asks the Factory.

------------------------------------------------------------------------

# The Next Line

``` solidity
feeOn = feeTo != address(0);
```

------------------------------------------------------------------------

# What is `feeOn`?

Look at the function signature:

``` solidity
function _mintFee(...) private returns (bool feeOn)
```

`feeOn` is the **named return variable**.

Conceptually, Solidity treats this like:

``` solidity
bool feeOn;

// ...

return feeOn;
```

Assigning to `feeOn` determines both the local value and the value
returned to `mint()`.

------------------------------------------------------------------------

# What is `address(0)`?

The zero address is:

    0x0000000000000000000000000000000000000000

It is commonly used as a sentinel value meaning:

-   no address
-   disabled
-   empty
-   not configured

Examples:

``` solidity
owner != address(0)
```

means an owner exists.

Likewise:

``` solidity
feeTo == address(0)
```

means no protocol fee recipient has been configured.

------------------------------------------------------------------------

# Why Use the Zero Address?

Instead of storing:

``` solidity
bool protocolFeeEnabled;
address protocolTreasury;
```

Uniswap stores only:

``` solidity
address feeTo;
```

If:

    feeTo = 0xABCD...

then:

    Protocol Fees = ON

If:

    feeTo = address(0)

then:

    Protocol Fees = OFF

One variable acts as both the configuration and the on/off switch.

------------------------------------------------------------------------

# Examples

## Fees Enabled

    feeTo = 0x1234...

    ↓

    feeOn = true

Protocol fee calculations will execute.

------------------------------------------------------------------------

## Fees Disabled

    feeTo = address(0)

    ↓

    feeOn = false

The protocol fee calculations are skipped.

------------------------------------------------------------------------

# Important Clarification

This does **not** disable trading fees.

Trading fees are always collected.

When protocol fees are disabled:

    Trader pays 0.30%
            │
            ▼
    Pool grows
            │
            ▼
    100% belongs to LPs

When protocol fees are enabled:

    Trader pays 0.30%
            │
            ▼
    Pool grows
            │
            ▼
    LPs keep 5/6
    Protocol receives 1/6

The trader always pays exactly the same amount.

------------------------------------------------------------------------

# Why Return `feeOn`?

Inside `mint()`:

``` solidity
bool feeOn = _mintFee(_reserve0, _reserve1);
```

Later:

``` solidity
if (feeOn)
    kLast = uint(reserve0).mul(reserve1);
```

Rather than querying the Factory a second time, `_mintFee()` simply
returns the value it already computed.

This avoids another external call and keeps the logic centralized.

------------------------------------------------------------------------

# Summary

These two lines do **not** calculate fees.

They simply determine:

1.  Who should receive protocol LP tokens (`feeTo`).
2.  Whether protocol fee accounting is enabled (`feeOn`).

Only after those questions are answered does `_mintFee()` proceed to
examine `kLast` and determine whether any protocol LP tokens should
actually be minted.
