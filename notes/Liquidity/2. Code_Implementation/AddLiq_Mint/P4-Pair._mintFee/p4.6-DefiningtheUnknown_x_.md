# `_mintFee()` — Chapter 6: Defining the Unknown

Up to this point, we have answered every conceptual question.

We now understand:

- Why `_mintFee()` exists.
- Why the protocol receives LP tokens instead of withdrawing assets from the pool.
- Why `kLast` is stored.
- Why Uniswap compares `rootK` and `rootKLast`.
- Why the protocol cannot simply mint one-sixth of the liquidity growth.

The only question remaining is:

> **Exactly how many LP tokens should the protocol mint?**

Before deriving the formula, we first need to define the quantities involved.

------------------------------------------------------------------------

# Step 1 — What Do We Already Know?

From the previous chapter we know that the pool's liquidity has grown.

Previous measured liquidity:

```text
rootKLast
```

Let's denote this as:

```text
L₀
```

Current measured liquidity:

```text
rootK
```

Let's denote this as:

```text
L₁
```

Therefore:

```text
Previous Liquidity

↓

L₀

Current Liquidity

↓

L₁
```

The total liquidity growth is simply:

```text
L₁ − L₀
```

Notice something important.

We are **not guessing** this growth.

We already measured it in the previous chapter using:

```solidity
uint256 rootK = Math.sqrt(uint256(_reserve0).mul(_reserve1));
uint256 rootKLast = Math.sqrt(_kLast);
```

So the liquidity growth is already known.

The only unknown is how many LP tokens correspond to that growth.

------------------------------------------------------------------------

# Step 2 — Existing LP Tokens

Before minting anything, the pool already has LP tokens.

The current supply is simply:

```solidity
totalSupply
```

To make the mathematics easier, let's call it:

```text
S
```

So:

```text
S

=

Current LP Token Supply
```

This represents every LP token that currently exists before the protocol receives any new LP tokens.

------------------------------------------------------------------------

# Step 3 — The Unknown

Now imagine the protocol is about to mint LP tokens.

How many?

We don't know.

That is exactly what we are trying to calculate.

Let's call this unknown amount:

```text
x
```

where:

```text
x

=

LP Tokens To Mint
```

Notice something interesting.

This unknown `x` is exactly what eventually becomes the Solidity variable:

```solidity
uint256 liquidity;
```

Inside `_mintFee()`.

At this point in our derivation:

```text
liquidity

=

x
```

We simply do not know its value yet.

That is the purpose of the derivation.

------------------------------------------------------------------------

# Step 4 — What Happens After Minting?

Before minting:

```text
LP Supply

↓

S
```

After minting:

```text
LP Supply

↓

S + x
```

This is where the problem becomes interesting.

The unknown:

```text
x
```

does not simply appear in the final answer.

It also changes:

```text
totalSupply
```

which changes:

```text
Every LP Holder's Ownership Percentage
```

This is exactly why we could not simply mint:

```text
(L₁ − L₀) ÷ 6
```

LP tokens.

The act of minting changes the ownership percentages that we are trying to calculate.

------------------------------------------------------------------------

# The Goal

At this point we already know:

- Previous liquidity (`L₀`)
- Current liquidity (`L₁`)
- Current LP token supply (`S`)

The only unknown remaining is:

```text
x
```

such that:

> **After minting `x` LP tokens, the protocol owns exactly one-sixth of the fee-generated liquidity growth.**

Everything that follows is simply solving for:

```text
x
```

using mathematics.

------------------------------------------------------------------------

- liquidity,
- LP token ownership,
- and total LP supply,

until the equation naturally transforms into the Solidity implementation.
