# `_mintFee()` — Chapter 7 (Part 1): Setting Up the Equation

Up to this point, we have established the following:

- The protocol fee is enabled.
- The pool has grown since the previous liquidity event.
- The protocol deserves **one-sixth of the fee-generated liquidity growth**.
- The protocol receives **ordinary LP tokens**, not ETH or USDC.
- We cannot simply mint `growth / 6` LP tokens because minting itself changes `totalSupply`, which changes everyone's ownership percentage.

The only question remaining is:

> **Exactly how many LP tokens should the protocol mint?**

This chapter answers that question by constructing the mathematical equation that Uniswap needs to solve.

We will **not** solve the equation yet.

First, we need to understand why this equation is correct.

------------------------------------------------------------------------

# Defining Our Variables

To make the derivation easier to read, we'll introduce four symbols.

Previous measured liquidity:

```text
L₀ = rootKLast
```

Current measured liquidity:

```text
L₁ = rootK
```

Current LP token supply:

```text
S = totalSupply
```

Unknown LP tokens to mint:

```text
x
```

where

```text
x = LP Tokens To Mint
```

Notice something interesting.

This unknown `x` is exactly what eventually becomes:

```solidity
uint256 liquidity;
```

inside `_mintFee()`.

At this point, however, we do **not** know its value.

That is exactly what we are trying to calculate.

------------------------------------------------------------------------

# What Does The Protocol Actually Want?

It is very easy to think:

> "The protocol wants some LP tokens."

That is **not** the real goal.

The protocol does **not** actually care about the number of LP tokens.

Instead, it cares about **ownership**.

Its goal is:

> **"After minting new LP tokens, I should own exactly one-sixth of the fee-generated liquidity growth."**

Notice the wording carefully.

It does **not** say:

- Mint me one-sixth of the LP tokens.
- Mint me one-sixth of the pool.
- Mint me one-sixth of all liquidity.

Instead, it says:

> **"Mint enough LP tokens so that, after minting, my ownership corresponds to one-sixth of the liquidity growth."**

That is a completely different problem.

------------------------------------------------------------------------

# How Much Of The Pool Will The Protocol Own?

Suppose the current LP token supply is:

```text
S
```

Before minting:

```text
Protocol LP Tokens

↓

0
```

Now suppose Uniswap mints:

```text
x LP Tokens
```

for the protocol.

After minting:

```text
Protocol LP Tokens

↓

x
```

The total LP supply also changes.

Instead of:

```text
S
```

it becomes:

```text
S + x
```

Therefore, after minting, the protocol owns:

```text
        x
────────────────
     S + x
```

of all LP tokens.

This fraction represents the protocol's ownership percentage of the pool.

Nothing magical happened.

It is simply:

```text
Your LP Tokens
──────────────────
Total LP Tokens
```

------------------------------------------------------------------------

# A Very Important Question

At this point I had the following confusion:

> **"The protocol only deserves one-sixth of the liquidity growth, not one-sixth of the current pool."**

That observation is **completely correct**.

The protocol does **not** deserve:

```text
Current Pool / 6
```

Instead, it deserves:

```text
(Current Liquidity − Previous Liquidity)

───────────────────────────────────────
                  6
```

or

```text
(L₁ − L₀) / 6
```

That is the protocol's reward.

Nothing more.

Nothing less.

------------------------------------------------------------------------

# Another Confusion

The next question naturally follows.

> **"Then why is the left side describing ownership of the entire pool?"**

This is one of the most important ideas in the entire derivation.

The answer is:

Because the protocol receives **ordinary LP tokens**.

------------------------------------------------------------------------

# LP Tokens Do Not Own Only The Growth

Imagine the pool looked like this.

Before swaps:

```text
Measured Liquidity

↓

600
```

After many swaps:

```text
Measured Liquidity

↓

660
```

Therefore:

```text
Liquidity Growth

↓

60
```

The protocol deserves:

```text
60 / 6

=

10 Units Of Liquidity
```

Now imagine there were two different kinds of LP tokens.

Type A:

> Owns only the original liquidity.

Type B:

> Owns only the newly generated fee growth.

If Uniswap had Type B tokens, then life would be easy.

The protocol could simply receive "growth-only" tokens.

But Uniswap does **not** work like that.

There is only **one** type of LP token.

Every LP token represents ownership of:

> **The entire current pool.**

Not just:

- the original liquidity,
- or the fee-generated growth.

It owns everything.

------------------------------------------------------------------------

# Company Analogy

Think of a company.

Initially:

```text
Company Value

↓

₹600

Outstanding Shares

↓

600
```

Therefore:

```text
1 Share

=

₹1
```

Later, the company becomes more valuable.

```text
Company Value

↓

₹660
```

Suppose someone says:

> "Let's print 10 new shares."

Now ask yourself.

Does each share still represent:

```text
₹1 ?
```

No.

Now:

```text
Company Value

↓

₹660

Shares

↓

610
```

Each share is now worth approximately:

```text
₹660 / 610

≈

₹1.082
```

Therefore:

```text
10 Shares

≠

₹10
```

Instead:

```text
10 × 1.082

≈

₹10.82
```

Printing new shares changed the relationship between:

- company value,
- and total shares.

Exactly the same thing happens in Uniswap.

LP tokens are ownership shares.

They do **not** represent a fixed amount of liquidity.

------------------------------------------------------------------------

# The Big Realization

This finally answers the earlier confusion.

The protocol deserves:

```text
10 Units Of Liquidity
```

But it cannot receive:

> "10 Units Of Growth Tokens"

because those tokens do not exist.

Instead, it receives **ordinary LP tokens**, and ordinary LP tokens always own a percentage of:

> **The entire current pool.**

Therefore the correct question is **not**:

> "What percentage of the growth should the protocol own?"

Instead it is:

> **"What percentage of the entire current pool is equal in value to one-sixth of the liquidity growth?"**

That is a completely different question.

------------------------------------------------------------------------

# Converting The Reward Into A Percentage

Current pool liquidity:

```text
L₁
```

Protocol reward:

```text
(L₁ − L₀) / 6
```

Therefore, the protocol's desired ownership percentage is:

```text
      Protocol Reward
────────────────────────────
Current Pool Liquidity
```

Substituting the variables gives:

```text
       (L₁ − L₀)/6
──────────────────────────
          L₁
```

Notice something important.

The protocol is **not** receiving one-sixth of the entire pool.

The numerator is **only** the protocol's share of the liquidity growth.

We divide by the current pool liquidity because ownership percentages are always measured relative to the **entire pool**, and LP tokens always represent ownership of the entire pool.

------------------------------------------------------------------------

# The Left Side

Earlier we established that after minting:

```text
Protocol LP Tokens

↓

x
```

Total LP supply becomes:

```text
S + x
```

Therefore the protocol's ownership according to LP tokens is:

```text
        x
────────────────
     S + x
```

------------------------------------------------------------------------

# The Right Side

From the liquidity perspective, the protocol should own:

```text
       (L₁ − L₀)/6
──────────────────────────
          L₁
```

of the current pool.

------------------------------------------------------------------------

# The Key Observation

These two fractions are describing **the exact same ownership**.

The left side measures ownership using:

- LP tokens.

The right side measures ownership using:

- liquidity.

Since they describe the same ownership, they must be equal.

Therefore:

```text
            x                (L₁ − L₀)/6
      ─────────────   =   ─────────────────
          S + x                 L₁
```

This is the most important equation in the entire derivation.

Everything that follows is simply solving this equation for:

```text
x
```

------------------------------------------------------------------------

# Summary

This chapter did **not** derive the formula.

Instead, it constructed the mathematical equation that Uniswap must solve.

The next chapter performs the algebra step by step until this equation naturally transforms into the Solidity implementation:

```solidity
uint256 numerator = totalSupply.mul(rootK.sub(rootKLast));
uint256 denominator = rootK.mul(5).add(rootKLast);
uint256 liquidity = numerator / denominator;
```
