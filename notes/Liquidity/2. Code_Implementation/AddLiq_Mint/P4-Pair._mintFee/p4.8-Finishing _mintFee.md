# Finishing `_mintFee()`

At this point, we have already derived the protocol fee formula from first principles.

The remaining lines of `_mintFee()` simply apply that result and handle a few edge cases.

```solidity
uint256 liquidity = numerator / denominator;

if (liquidity > 0)
    _mint(feeTo, liquidity);

} else if (_kLast != 0) {
    kLast = 0;
}
```

Unlike the previous chapters, there is very little mathematics left.

Most of the remaining logic exists for correctness and gas efficiency.

------------------------------------------------------------------------

# Computing `liquidity`

```solidity
uint256 liquidity = numerator / denominator;
```

This line directly implements the mathematical formula derived in the previous chapter.

```text
                      totalSupply(rootK − rootKLast)
liquidity

=

──────────────────────────────────────────────────────
               5(rootK) + rootKLast
```

Nothing new is happening here.

The code is simply translating the mathematics into Solidity.

The value stored in:

```solidity
liquidity
```

represents the **exact number of LP tokens** that the protocol should receive.

It does **not** represent:

- ETH
- USDC
- WBTC
- Trading fees

Instead, it represents:

> **The exact number of LP tokens that must be minted so that, after increasing `totalSupply`, the protocol owns exactly one-sixth of the fee-generated liquidity growth.**

------------------------------------------------------------------------

# Why Check `liquidity > 0`?

```solidity
if (liquidity > 0)
    _mint(feeTo, liquidity);
```

A natural question is:

> **Why not simply call `_mint(feeTo, liquidity)` every time?**

The answer lies in Solidity's integer arithmetic.

Suppose the calculation produces:

```text
numerator

↓

5

denominator

↓

10
```

Mathematically:

```text
5 / 10

=

0.5
```

However, Solidity performs integer division.

Therefore:

```text
5 / 10

=

0
```

So the calculated liquidity becomes:

```solidity
uint256 liquidity = 0;
```

If Uniswap immediately executed:

```solidity
_mint(feeTo, 0);
```

nothing useful would happen.

The protocol would receive:

```text
0 LP Tokens
```

`totalSupply` would increase by:

```text
0
```

and the transaction would simply waste gas performing a function call that changes nothing.

Therefore, Uniswap first checks:

```solidity
if (liquidity > 0)
```

and only mints LP tokens when there is actually something to mint.

------------------------------------------------------------------------

# Are Tiny Protocol Fees Lost?

At first, it may seem that rounding down causes the protocol to lose fees.

Fortunately, that is **not** what happens.

Suppose today's calculation produces:

```text
0.4 LP Tokens
```

Because Solidity rounds down:

```text
liquidity = 0
```

No LP tokens are minted during this liquidity event.

However, the trading fees themselves never leave the pool.

More swaps continue increasing the pool's liquidity.

Later, another liquidity event may produce:

```text
1.3 LP Tokens
```

which Solidity rounds down to:

```text
1 LP Token
```

Now the protocol finally receives an LP token.

In other words, the tiny fractional amounts are **not lost**.

They remain inside the pool and continue accumulating until the calculated liquidity becomes large enough to mint at least one whole LP token.

------------------------------------------------------------------------

# Minting The LP Tokens

```solidity
_mint(feeTo, liquidity);
```

We already studied `_mint()` earlier, so there is very little new to explain here.

This function does **not** mint:

- ETH
- USDC
- WBTC
- Trading fees

Instead, it mints:

> **LP tokens.**

Those LP tokens are sent directly to:

```solidity
feeTo
```

which is the protocol's treasury address.

From that point onward, the protocol becomes another liquidity provider.

Its LP tokens represent ownership of the pool, allowing it to later redeem its proportional share of the underlying assets by burning those LP tokens.

------------------------------------------------------------------------

# When Protocol Fees Are Disabled

The final branch is:

```solidity
} else if (_kLast != 0) {
    kLast = 0;
}
```

This branch executes only when:

```text
feeOn = false
```

meaning protocol fees have been disabled.

The obvious question is:

> **Why does Uniswap reset `kLast`?**

Remember that `kLast` is simply a historical snapshot of the pool's liquidity.

Its purpose is to measure:

```text
Current k

−

Previous k
```

to determine how much liquidity has grown.

------------------------------------------------------------------------

# Timeline

Imagine the following sequence.

### Day 1

Protocol fees are enabled.

```text
feeOn = true

kLast = 20,000,000
```

This becomes the protocol's baseline.

------------------------------------------------------------------------

### Days 2–10

Thousands of swaps occur.

The pool grows.

Whenever liquidity events occur, protocol LP tokens are minted and `kLast` is updated.

Everything works normally.

------------------------------------------------------------------------

### Day 11

Governance disables protocol fees.

```text
feeOn = false
```

Now this branch executes.

```solidity
kLast = 0;
```

The previous baseline is intentionally discarded.

------------------------------------------------------------------------

### Days 12–30

Protocol fees remain disabled.

Thousands more swaps occur.

The pool continues growing.

During this period, the protocol earns **nothing**.

All trading fees remain with the liquidity providers.

------------------------------------------------------------------------

### Day 31

Governance enables protocol fees again.

The next liquidity event executes `_mintFee()`.

Since:

```text
kLast = 0
```

there is no previous baseline to compare against.

Therefore, no protocol LP tokens are minted.

Instead, a brand-new baseline is established after the liquidity operation finishes.

From this point onward, the protocol only earns fees generated **after** protocol fees were re-enabled.

------------------------------------------------------------------------

# Why Reset `kLast`?

Suppose Uniswap **did not** reset `kLast`.

Imagine the old value remained:

```text
kLast = 20,500,000
```

while protocol fees were disabled.

Weeks later, the pool grows to:

```text
25,000,000
```

If protocol fees are enabled again, Uniswap would compare:

```text
Current k

↓

25,000,000

Old kLast

↓

20,500,000
```

It would incorrectly conclude that the protocol deserves LP tokens for all of that growth.

However, protocol fees were disabled during that period.

The protocol should **not** retroactively claim fees that accumulated while fee collection was turned off.

Resetting:

```solidity
kLast = 0;
```

prevents this.

When protocol fees are enabled again, Uniswap starts measuring from a completely fresh baseline.

------------------------------------------------------------------------

# An Interesting Edge Case

A natural question is:

> **What if protocol fees are disabled before the protocol has collected its accumulated fees?**

Suppose:

```text
Protocol Fees

↓

ON
```

The pool grows significantly.

However, no liquidity event (`mint()` or `burn()`) occurs, so `_mintFee()` is never executed.

Before the next liquidity event, governance disables protocol fees.

Now this branch executes:

```solidity
kLast = 0;
```

without minting any protocol LP tokens.

This means the protocol simply forfeits those unclaimed fees.

Those fees remain inside the pool and continue benefiting the existing liquidity providers.

This is intentional.

Uniswap chooses **not** to retroactively collect protocol fees after governance has decided to disable them.

------------------------------------------------------------------------

# Another Edge Case

Suppose:

```text
feeOn = false

kLast = 0
```

Now `_mintFee()` executes.

The code reaches:

```solidity
else if (_kLast != 0)
```

Since:

```text
_kLast == 0
```

the condition is false.

Nothing happens.

Execution simply leaves the function.

This is correct because:

- protocol fees are already disabled,
- there is already no stored baseline,
- and there is nothing left to reset.

------------------------------------------------------------------------

# Summary

The remaining portion of `_mintFee()` is primarily responsible for:

- implementing the derived fee formula,
- avoiding unnecessary gas usage by skipping zero-value mints,
- minting protocol LP tokens only when appropriate,
- resetting the historical liquidity snapshot when protocol fees are disabled,
- preventing retroactive protocol fee collection, and
- establishing a fresh baseline whenever protocol fees are enabled again.

With this, the entire `_mintFee()` function has been fully dissected—from its motivation and mathematical derivation all the way to its final implementation and edge cases.
