# `_mintFee()` --- Chapter 4: `rootK`, `rootKLast`, and Detecting Liquidity Growth

At this point in `_mintFee()` we have passed the following checks:

``` solidity
if (feeOn) {
    if (_kLast != 0) {
```

This means:

-   Protocol fees are enabled.
-   A previous liquidity snapshot exists.

Now Uniswap measures the current liquidity and compares it with the
previous snapshot.

``` solidity
uint rootK = Math.sqrt(uint(_reserve0).mul(_reserve1));
uint rootKLast = Math.sqrt(_kLast);

if (rootK > rootKLast) {
    // Fee formula begins here...
}
```

This chapter stops **before** the fee formula.

------------------------------------------------------------------------

# Computing `rootK`

``` solidity
uint rootK = Math.sqrt(uint(_reserve0).mul(_reserve1));
```

This happens in three steps.

## Step 1 --- Compute the current `k`

Example reserves:

  Token      Amount
  ------- ---------
  ETH           102
  USDC      198,500

    k = 102 × 198,500
      = 20,247,000

Notice that `_reserve0` and `_reserve1` are the **stored reserves**, not
the ERC-20 balances.

------------------------------------------------------------------------

## Step 2 --- Convert to `uint256`

The reserves are stored as `uint112`.

`Math.sqrt()` expects a `uint256`.

Therefore:

``` solidity
uint(_reserve0)
```

simply widens the integer type.

The value does not change.

------------------------------------------------------------------------

## Step 3 --- Take the square root

    √20,247,000 ≈ 4499

Therefore:

    rootK = 4499

Because Solidity uses integers, the fractional part is discarded.

------------------------------------------------------------------------

# Why `√k` Instead of `k`?

Earlier in the Pair contract, the very first LP mint used:

``` solidity
liquidity = Math.sqrt(amount0 * amount1);
```

That was not an accident.

Uniswap measures liquidity using the **geometric mean**.

Initial liquidity:

    √(amount0 × amount1)

Current pool liquidity:

    √(reserve0 × reserve1)

Both use the same liquidity metric.

LP ownership scales with this liquidity measure rather than with `k`
itself.

------------------------------------------------------------------------

# Computing `rootKLast`

``` solidity
uint rootKLast = Math.sqrt(_kLast);
```

Suppose:

    kLast = 20,000,000

Then:

    √20,000,000 ≈ 4472

Therefore:

    rootKLast = 4472

This is simply the previous liquidity measurement obtained from the
stored snapshot.

------------------------------------------------------------------------

# A Common Question

**Why is `rootK` computed from reserves instead of balances?**

This is one of the most important details in the entire function.

Remember the beginning of `mint()`:

``` solidity
(uint112 _reserve0, uint112 _reserve1,) = getReserves();

uint balance0 = IERC20(token0).balanceOf(address(this));
uint balance1 = IERC20(token1).balanceOf(address(this));
```

Immediately before `_mintFee()` runs:

    Balances already include the user's deposit.

    Reserves do not.

Example:

Before deposit:

    Reserve = 100 ETH / 200,000 USDC
    Balance = 100 ETH / 200,000 USDC

User transfers liquidity:

    Reserve = 100 ETH / 200,000 USDC

    Balance = 110 ETH / 220,000 USDC

Now `_mintFee()` executes.

It intentionally uses:

    100 ETH / 200,000 USDC

instead of

    110 ETH / 220,000 USDC

------------------------------------------------------------------------

# Why?

Because `_mintFee()` wants to answer:

> **"How much did the pool grow because of swaps since the previous
> liquidity event?"**

It does **not** want to include the liquidity being deposited in the
current transaction.

If balances were used:

    110 × 220,000

the protocol would mistake the user's new capital for fee-generated
growth.

That would be incorrect.

------------------------------------------------------------------------

# Timeline

    Previous reserves

    ↓

    User transfers tokens

    ↓

    Balances increase

    ↓

    Reserves unchanged

    ↓

    _mintFee()

    ↓

    Uses reserves

    NOT balances

    ↓

    User LP tokens minted

    ↓

    _update()

    ↓

    Reserves catch up to balances

    ↓

    kLast updated

This ordering prevents the protocol from charging itself fees on newly
deposited liquidity.

------------------------------------------------------------------------

# Relationship Between Reserves, Balances, `rootK`, and `kLast`

A useful mental model is:

    kLast

    ↓

    Previous checkpoint


    Current reserves

    ↓

    rootK

    ↓

    Current checkpoint


    Balances

    ↓

    One step ahead

    (only during mint()/burn() before _update())

Usually:

    Reserve == Balance

They temporarily differ only after tokens have moved but before
`_update()` executes.

------------------------------------------------------------------------

# Comparing the Snapshots

``` solidity
if (rootK > rootKLast) {
```

Now Uniswap asks:

> **"Has the pool's measured liquidity increased since the last
> snapshot?"**

Example:

    rootKLast = 4472

    rootK = 4499

Comparison:

    4499 > 4472

    ↓

    true

The pool has grown.

------------------------------------------------------------------------

# If They Are Equal

    rootK = 4472

    rootKLast = 4472

No growth has occurred.

No protocol fee calculation is needed.

------------------------------------------------------------------------

# If `rootK` Is Smaller

    4460 > 4472

    ↓

    false

The protocol never mints negative LP tokens.

If measured liquidity has not increased, it simply skips the fee
calculation.

------------------------------------------------------------------------

# What Is This Comparison Actually Detecting?

It is **not** asking:

-   "Did a swap happen?"
-   "How many swaps happened?"

Instead it asks:

> **"Has the measured liquidity (`√k`) of the pool increased since the
> previous checkpoint?"**

If the answer is yes, additional value exists inside the pool.

That growth usually comes from accumulated trading fees, although direct
token donations can also increase the measured liquidity.

The algorithm measures growth itself, not its cause.

------------------------------------------------------------------------

# Another Common Misconception

Suppose:

    rootK = 4499

    rootKLast = 4472

Then:

    4499 - 4472 = 27

**This does NOT mean:**

-   27 LP tokens
-   27 ETH
-   27 USDC

It means:

> The liquidity metric (`√k`) increased by **27 units**.

The protocol still needs to determine:

> **"How many LP tokens correspond to that liquidity growth?"**

That is the purpose of the fee formula, which begins immediately after
this comparison.

The next chapter derives that formula from first principles.
