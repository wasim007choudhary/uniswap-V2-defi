# `_mintFee()` --- Chapter 3: `kLast`, Snapshots, and the Fee Accounting Baseline

The next section of `_mintFee()` is:

``` solidity
uint _kLast = kLast; // gas savings

if (feeOn) {
    if (_kLast != 0) {
        ...
    }
}
```

At first glance this appears to be a simple cache and two `if`
statements.

In reality, this is where the protocol fee accounting model begins.

------------------------------------------------------------------------

# The Line

``` solidity
uint _kLast = kLast; // gas savings
```

There are two questions to answer:

1.  What is `kLast`?
2.  Why copy it into `_kLast`?

The first question is vastly more important.

------------------------------------------------------------------------

# First, What is `k`?

Throughout Uniswap V2 you have already seen:

    x × y = k

where

-   `x = reserve0`
-   `y = reserve1`

Example:

  Reserve      Amount
  --------- ---------
  ETH             100
  USDC        200,000

Therefore:

    k = 100 × 200,000
      = 20,000,000

This is called the **constant product**.

------------------------------------------------------------------------

# Does `k` Stay Constant?

People often say:

> "Uniswap keeps `k` constant."

That statement is only approximately true.

Ignoring fees, swaps preserve the invariant.

With trading fees, the fee remains inside the pool.

Therefore the product of reserves gradually grows.

Example:

Before:

    100 ETH
    200,000 USDC

    k = 20,000,000

After many swaps:

    101.2 ETH
    198,430 USDC

    k ≈ 20,077,116

The pool became slightly larger because fees accumulated inside it.

------------------------------------------------------------------------

# Then What is `kLast`?

`kLast` is simply a **stored snapshot** of `k`.

Imagine taking a photograph of the pool.

    Current Pool

    ↓

    100 ETH
    200,000 USDC

    ↓

    k = 20,000,000

    ↓

    Store Snapshot

    ↓

    kLast = 20,000,000

It is not a live value.

It does not automatically update.

It is simply a historical checkpoint.

------------------------------------------------------------------------

# Why Store a Snapshot?

Because later the protocol wants to answer:

> **"How much has the pool grown since the previous liquidity event?"**

Growth always requires two measurements.

Think of a car odometer.

Yesterday:

    15,000 km

Today:

    15,300 km

Distance travelled:

    300 km

Without yesterday's reading, the distance travelled cannot be
determined.

`kLast` serves exactly the same purpose.

------------------------------------------------------------------------

# Timeline

Time A

    100 ETH
    200,000 USDC

    ↓

    k = 20,000,000

    ↓

    kLast = 20,000,000

Thousands of swaps occur.

Fees accumulate.

Current reserves become:

    102 ETH
    196,500 USDC

Current:

    k = 20,043,000

But:

    kLast

    still equals

    20,000,000

Now the protocol can compare:

    Current k

    vs

    Previous k

------------------------------------------------------------------------

# Why Not Compare Individual Reserves?

Old:

    100 ETH
    200,000 USDC

Current:

    102 ETH
    196,500 USDC

One reserve increased.

One decreased.

Looking at either reserve individually does not measure the overall pool
growth.

The product combines both reserves into one quantity.

------------------------------------------------------------------------

# Is `kLast` Updated After Every Swap?

No.

This is one of the most common misconceptions.

Swaps update the reserves.

They therefore change the current value of `k`.

They **do not** update `kLast`.

Visualize it as:

    Liquidity Event

    ↓

    Take Snapshot

    ↓

    kLast fixed

    ↓

    Swap

    Swap

    Swap

    Swap

    ↓

    Current reserves change

    ↓

    kLast never changes

This is intentional.

If `kLast` changed after every swap, the protocol would lose its
historical baseline and could no longer measure accumulated growth.

------------------------------------------------------------------------

# Why Cache It?

``` solidity
uint _kLast = kLast;
```

Reading storage is expensive.

The function uses this value multiple times.

So it copies it into a local variable once.

This is the reason for:

``` solidity
// gas savings
```

The optimization is straightforward.

The concept behind `kLast` is what matters.

------------------------------------------------------------------------

# The First Condition

``` solidity
if (feeOn) {
```

This simply asks:

> **"Are protocol fees enabled?"**

If `feeOn` is false, all protocol fee calculations are skipped.

Trading fees still exist.

Only protocol fee minting is bypassed.

------------------------------------------------------------------------

# The Second Condition

``` solidity
if (_kLast != 0) {
```

This condition asks:

> **"Do we already have a historical snapshot?"**

If:

    kLast = 0

there is nothing to compare the current pool against.

No comparison means no growth measurement.

No growth measurement means no protocol fee calculation.

------------------------------------------------------------------------

# Why Not Compare Against Zero?

Suppose:

    Current k = 20,000,000

If Uniswap treated:

    Previous k = 0

then it would conclude:

    Growth = 20,000,000

which is obviously incorrect.

The pool already existed.

The protocol simply was not tracking it yet.

------------------------------------------------------------------------

# What Happens When Fees Are First Enabled?

Imagine governance enables protocol fees today.

    feeOn = true

    kLast = 0

First liquidity event:

    if (_kLast != 0)

    ↓

    false

No protocol LP tokens are minted.

Later, after the liquidity operation finishes:

``` solidity
if (feeOn)
    kLast = uint(reserve0).mul(reserve1);
```

Now the protocol has established its first baseline.

Future liquidity events can compare against it.

------------------------------------------------------------------------

# Timeline

    Protocol Fees Disabled

    ↓

    kLast = 0

    ↓

    Governance Enables Fees

    ↓

    First mint()

    ↓

    No previous snapshot

    ↓

    Store snapshot

    ↓

    kLast = Current k

    ↓

    Thousands of swaps

    ↓

    Current k grows

    ↓

    Second mint()

    ↓

    Comparison is now possible

This guarantees the protocol only earns fees generated **after**
protocol fees were enabled.

It does not retroactively claim fees accumulated during earlier periods.

------------------------------------------------------------------------

# Summary

By the time execution reaches:

``` solidity
if (_kLast != 0)
```

the protocol has already determined:

-   whether protocol fees are enabled, and
-   whether a valid historical baseline exists.

Only if **both** conditions are satisfied does `_mintFee()` proceed to
compute the current liquidity measure (`rootK`) and compare it against
the previous snapshot.
