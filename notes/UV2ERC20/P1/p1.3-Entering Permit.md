# UniswapV2ERC20.md

# Part 1.3 — Entering Permit (`DOMAIN_SEPARATOR`, `PERMIT_TYPEHASH`, `nonces`)

---

# Leaving Standard ERC20

Up until this point, everything we had seen was familiar ERC20 functionality:

```text
name
symbol
decimals
totalSupply
balanceOf
allowance
```

Then suddenly, we encountered three variables that do **not** exist in a normal ERC20 implementation.

```solidity
bytes32 public DOMAIN_SEPARATOR;

// keccak256("Permit(address owner,address spender,uint256 value,uint256 nonce,uint256 deadline)");
bytes32 public constant PERMIT_TYPEHASH =
    0x6e71edae12b1b97f4d1f60370fef10105fa2faae0126114a169c64845d6126c9;

mapping(address => uint) public nonces;
```

At this point, we were no longer studying ERC20.

We had entered the **Permit** system.

---

# First Observation

Before reading any functions, we asked ourselves:

> **Why does an ERC20 token suddenly need cryptography?**

A normal ERC20 only needs:

* balances,
* allowances,
* transfers.

Why would it suddenly need:

```text
DOMAIN_SEPARATOR

PERMIT_TYPEHASH

nonces
```

The answer is:

Because Permit allows approvals using **digital signatures** instead of a traditional on-chain `approve()` transaction.

Everything from this point onward exists to support that feature.

---

# `DOMAIN_SEPARATOR`

The first new variable was:

```solidity
bytes32 public DOMAIN_SEPARATOR;
```

Immediately, two questions arose.

1.

> What is a Domain?

2.

> Why does it need a separator?

---

# First Intuition

Rather than jumping into cryptography, we started with a simpler mental picture.

Imagine several applications.

```text
Application A

Application B

Application C
```

Suppose Alice signs a message for Application A.

Question:

Should Application B also be able to use that same signature?

Obviously not.

That would be a massive security problem.

Therefore, every application needs its own identity.

That identity is called the:

```text
Domain
```

---

# Child Analogy

Imagine three schools.

```text
School A

School B

School C
```

Alice receives a hall pass from School A.

Could she walk into School B and say:

> "I already have a pass."

No.

School B would reply:

> "That pass belongs to a different school."

Exactly the same idea exists here.

A Permit signature must belong to one specific application.

---

# What Information Identifies an Application?

Later, inside the constructor, we discovered that the Domain contains:

```text
Application Name

Version

Chain ID

Contract Address
```

Together, these uniquely identify one deployed contract.

---

# Why Is It Called a Separator?

We discussed the name.

A better way to think about it is:

> It separates one application's signatures from another application's signatures.

Even if two contracts have identical Permit data,

their Domain is different.

Therefore,

their signatures become different.

---

# Example

Imagine Alice signs:

```text
Approve Router

100 LP
```

Without a Domain,

someone might try to reuse that signature elsewhere.

With a Domain,

the signature effectively becomes:

```text
Approve Router

100 LP

FOR THIS CONTRACT
```

That last part makes all the difference.

---

# `PERMIT_TYPEHASH`

The next variable was:

```solidity
bytes32 public constant PERMIT_TYPEHASH;
```

Above it was a comment.

```solidity
keccak256(
"Permit(address owner,address spender,uint256 value,uint256 nonce,uint256 deadline)"
)
```

Our first reaction was:

> Why hash a string?

---

# What Is This String?

Notice carefully.

It is **not data**.

It is describing a structure.

```text
Permit(

owner,

spender,

value,

nonce,

deadline

)
```

Think of it as a blueprint.

It tells every wallet and contract:

> A Permit consists of exactly these five fields in exactly this order.

---

# Child Analogy

Imagine an exam answer sheet.

Before students fill it out,

the sheet already says:

```text
Name:

Roll Number:

Subject:

Marks:
```

That empty form is the blueprint.

Students only fill the blanks later.

Similarly,

`PERMIT_TYPEHASH` describes the layout of a Permit before actual values are inserted.

---

# Why Hash the Blueprint?

Instead of storing a long string every time,

Uniswap hashes it once.

Result:

```text
Blueprint

↓

keccak256

↓

32-byte constant
```

This constant never changes.

Every Permit uses the exact same blueprint.

---

# `nonces`

Finally,

we reached:

```solidity
mapping(address => uint)
public nonces;
```

Question:

Why does every address need a nonce?

---

# Replay Attack

Imagine Alice signs:

```text
Approve Router

100 LP
```

Suppose the Router successfully uses it.

Without a nonce,

someone could simply reuse the exact same signature again.

And again.

And again.

The signature would never expire.

---

# Solution

Every successful Permit consumes one nonce.

Example:

Initially:

```text
Alice

↓

Nonce = 7
```

Alice signs using:

```text
Nonce = 7
```

Permit succeeds.

Now:

```text
Nonce

↓

8
```

Next time,

the contract expects:

```text
Nonce = 8
```

If somebody replays the old signature,

it still contains:

```text
Nonce = 7
```

The newly generated digest will use:

```text
Nonce = 8
```

Those digests no longer match.

The signature becomes invalid.

Replay prevented.

---

# Mental Picture

At this stage,

before reading a single Permit function,

our understanding became:

```text
DOMAIN_SEPARATOR

↓

Which application?

-----------------------

PERMIT_TYPEHASH

↓

What kind of message?

-----------------------

nonces

↓

Has this signature already been used?
```

Everything else inside `permit()` would later build upon these three concepts.

---

# Why We Didn't Dive Deeper Yet

At this point in our learning journey,

we intentionally stopped.

Although these variables looked cryptographic,

their true purpose would only become obvious once we reached:

```solidity
permit(...)
```

and especially these lines:

```solidity
DOMAIN_SEPARATOR

PERMIT_TYPEHASH

nonces[owner]++
```

Therefore,

we continued reading the contract first,

planning to return to each concept in much greater depth while studying the Permit function line by line.

That decision turned out to be the right one, because later every one of these variables naturally fit into the larger EIP-712 signing process.
