# Part 1.5 — Building the `DOMAIN_SEPARATOR` (Line by Line)

---

# The Complete Statement

Inside the constructor we reached:

```solidity id="8sdgq2"
DOMAIN_SEPARATOR = keccak256(
    abi.encode(
        keccak256(
            "EIP712Domain(string name,string version,uint256 chainId,address verifyingContract)"
        ),
        keccak256(bytes(name)),
        keccak256(bytes("1")),
        chainId,
        address(this)
    )
);
```

At first glance, this looks extremely intimidating.

There are:

* multiple `keccak256` calls,
* an `abi.encode`,
* strings,
* hashes,
* addresses,
* chain IDs.

Instead of trying to understand everything at once, we decided to go one piece at a time.

---

# Big Picture First

Before understanding the individual arguments, we first asked:

> **What is this entire expression trying to produce?**

The answer is:

```text id="0cr3rx"
DOMAIN_SEPARATOR
```

Think of it as creating a permanent identity card for this deployed Pair contract.

Instead of storing several pieces of information separately, they are combined and hashed into one fixed 32-byte value.

---

# Visual Overview

Conceptually:

```text id="bme83f"
Name

+

Version

+

Chain ID

+

Contract Address

↓

abi.encode(...)

↓

Bytes

↓

keccak256(...)

↓

DOMAIN_SEPARATOR
```

Notice that the hash is the **last** step.

Before hashing, everything is first encoded into a deterministic byte representation.

---

# Why `abi.encode()`?

One of the first questions we discussed was:

> Why not simply hash each value separately?

Because hashing independent values would lose their relationship.

Instead, EIP-712 wants to hash **one structured object**.

So we first build that object:

```text id="nhys1o"
Domain

↓

Encode

↓

Hash
```

This is exactly what `abi.encode()` is doing.

---

# First Argument

```solidity id="ebrypq"
keccak256(
    "EIP712Domain(
        string name,
        string version,
        uint256 chainId,
        address verifyingContract
    )"
)
```

This was probably the strangest line at first.

The natural question was:

> **Why hash another string?**

---

# This Is NOT Data

Very important.

This string is **not**:

```text id="g1r7hr"
Uniswap V2

or

Version 1

or

Chain ID
```

Instead, it describes the **structure** of the Domain.

Think of it as the blueprint.

---

# Child Analogy

Imagine a school admission form.

Before anyone fills it in, the paper already contains:

```text id="jx1c91"
Name:

Age:

Class:

Address:
```

That empty form is the structure.

Students only fill in the values later.

The EIP-712 domain works exactly the same way.

The string says:

```text id="8mz5y0"
A Domain consists of:

name

version

chainId

verifyingContract
```

It doesn't yet specify the actual values.

---

# Why Hash the Blueprint?

Instead of carrying around this long string forever, EIP-712 hashes it once.

Conceptually:

```text id="m7spga"
Blueprint

↓

keccak256

↓

32-byte constant
```

Now every implementation of EIP-712 agrees on exactly the same blueprint hash.

---

# Second Argument

```solidity id="ecg0ng"
keccak256(bytes(name))
```

Remember:

Earlier we saw:

```solidity id="j4grdw"
string public constant name =
    "Uniswap V2";
```

Here we are taking:

```text id="4h8fs4"
"Uniswap V2"
```

converting it into bytes:

```text id="t4h2s4"
bytes("Uniswap V2")
```

then hashing it:

```text id="s8g5uk"
keccak256(...)

↓

32-byte hash
```

---

# Why Hash the Name?

A natural question is:

> Why not pass the string directly?

Again, this follows the EIP-712 specification.

Dynamic types such as:

```text id="h6br0b"
string

bytes
```

are represented by their hash when building the domain.

This ensures every implementation computes exactly the same final value.

---

# Third Argument

```solidity id="6z4g5m"
keccak256(bytes("1"))
```

This represents the version.

At this point we paused and asked:

> **Version of what?**

The answer:

This is **not**:

* Ethereum version,
* Solidity version,
* Pair version.

Instead it is the version of the **EIP-712 signing domain** for this contract.

For Uniswap V2, that version is:

```text id="j6ztgu"
"1"
```

---

# What If It Became Version 2?

We discussed an interesting question.

Suppose developers later changed:

```solidity id="3m7f2f"
"1"
```

to

```solidity id="yq8tpn"
"2"
```

Would old signatures still work?

No.

Because the version contributes to the `DOMAIN_SEPARATOR`.

Changing the version changes the hash.

Changing the hash changes every digest.

Changing the digest invalidates every old signature.

This is intentional.

Different versions should not share signatures.

---

# Fourth Argument

```solidity id="mndjhh"
chainId
```

Earlier in the constructor we obtained:

```solidity id="3g9z6g"
assembly {
    chainId := chainid
}
```

Now that value becomes part of the domain.

Example:

```text id="y4hm7r"
Ethereum Mainnet

↓

1
```

or

```text id="y0wq2x"
Sepolia

↓

11155111
```

Including the chain ID ensures signatures cannot be replayed across different blockchains.

---

# Fifth Argument

```solidity id="6e93mo"
address(this)
```

Another excellent question we discussed was:

> **What exactly is `address(this)`?**

Here,

`this` refers to the current deployed Pair contract.

For example:

```text id="d9bg1v"
ETH/USDC Pair

↓

0xABC123...
```

Then:

```solidity id="t9m8j3"
address(this)
```

becomes:

```text id="o7l1df"
0xABC123...
```

Every Pair has a different address.

Therefore every Pair has a different `DOMAIN_SEPARATOR`.

---

# Why Include the Contract Address?

Imagine two completely different Pair contracts.

```text id="7v7crr"
ETH/USDC Pair

↓

0xAAA...

-------------------

WBTC/ETH Pair

↓

0xBBB...
```

Even if every other field were identical,

their contract addresses differ.

Therefore their domains differ.

Therefore their signatures differ.

A signature created for one Pair cannot be reused on another Pair.

---

# Putting Everything Together

At this point we finally understood what `abi.encode(...)` was building.

Conceptually:

```text id="jwul0q"
Domain(

    Blueprint,

    Name,

    Version,

    Chain ID,

    Contract Address

)
```

↓

Encode into bytes.

↓

Hash.

↓

Store as `DOMAIN_SEPARATOR`.
