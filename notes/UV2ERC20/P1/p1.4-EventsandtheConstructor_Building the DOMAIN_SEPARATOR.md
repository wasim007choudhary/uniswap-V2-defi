# UniswapV2ERC20.md

# Part 1.4 — Events and the Constructor (Building the DOMAIN_SEPARATOR)

---

# Events Before the Constructor

Before reaching the constructor, we encounter two standard ERC20 events.

```solidity
event Approval(address indexed owner, address indexed spender, uint value);
event Transfer(address indexed from, address indexed to, uint value);
```

These are standard ERC20 events and exist so that wallets, explorers, and applications can track what happens to tokens.

---

# `Transfer`

```solidity
event Transfer(
    address indexed from,
    address indexed to,
    uint value
);
```

This event is emitted whenever LP tokens move.

Examples:

```text
Alice transfers 20 LP tokens to Bob

↓

Transfer(
    Alice,
    Bob,
    20
)
```

When minting LP tokens:

```text
address(0)

↓

Alice

↓

100 LP
```

the emitted event becomes:

```text
Transfer(
    address(0),
    Alice,
    100
)
```

Notice something interesting.

Minting is represented as tokens coming **from the zero address**.

---

Likewise, burning LP tokens emits:

```text
Transfer(
    Alice,
    address(0),
    100
)
```

which represents tokens leaving circulation.

---

# `Approval`

```solidity
event Approval(
    address indexed owner,
    address indexed spender,
    uint value
);
```

Whenever an allowance changes, this event is emitted.

Example:

```text
Alice approves Router
to spend 250 LP tokens

↓

Approval(
    Alice,
    Router,
    250
)
```

Wallets and block explorers read these events to display approvals.

---

# Constructor

Now we finally reach:

```solidity
constructor() public {
```

This constructor executes exactly **once**.

It runs only when this Pair contract is deployed.

After deployment, it is never executed again.

---

# Child Analogy

Imagine opening a new school.

Before students arrive, the school principal writes:

* school name,
* school address,
* registration number.

After opening day, these are not rewritten every morning.

They were established once.

The constructor behaves the same way.

It performs one-time initialization.

---

# What Does This Constructor Do?

Unlike many constructors, this one does **not** initialize balances.

It does **not** mint LP tokens.

It does **not** configure reserves.

Instead, it performs only one major task:

```text
Build the DOMAIN_SEPARATOR
```

Everything inside this constructor exists for that purpose.

---

# First Line

```solidity
uint chainId;
```

This simply declares a variable.

Initially:

```text
chainId

↓

empty
```

No value has been assigned yet.

---

# Inline Assembly

Next we saw:

```solidity
assembly {
    chainId := chainid
}
```

This immediately raised an important question.

> Why use assembly instead of normal Solidity?

The answer lies in the Solidity version.

Remember:

```text
pragma solidity =0.5.16;
```

Modern Solidity lets us write:

```solidity
block.chainid
```

However, Solidity 0.5.16 did **not** provide that feature.

Therefore Uniswap uses inline assembly to directly ask the EVM:

> "Which blockchain am I currently running on?"

The EVM returns the current chain ID.

---

# Example

If deployed on Ethereum Mainnet:

```text
chainId

↓

1
```

If deployed on Sepolia:

```text
chainId

↓

11155111
```

Different networks return different values.

---

# Why Does Permit Need the Chain ID?

Suppose Alice signs a Permit on Ethereum Mainnet.

Without including the chain ID, someone might try to replay that same signature on another blockchain where the contract also exists.

Including the chain ID prevents this.

Conceptually:

```text
Permit

+

Ethereum Mainnet

↓

Valid
```

but

```text
Same Permit

+

Another Chain

↓

Invalid
```

This is one layer of replay protection.

---

# Building the DOMAIN_SEPARATOR

The next line begins the construction:

```solidity
DOMAIN_SEPARATOR = keccak256(
```

At first glance this looks intimidating.

But conceptually it is very simple.

We are taking several pieces of information that uniquely identify this application and hashing them into one 32-byte value.

Think of it as creating a permanent fingerprint.

```text
Application Identity

↓

Hash

↓

DOMAIN_SEPARATOR
```

---

# The Identity Contains

The constructor includes:

```text
Application Name

↓

"Uniswap V2"

-------------------

Version

↓

"1"

-------------------

Chain ID

↓

Current blockchain

-------------------

Contract Address

↓

This deployed Pair contract
```

All of these together uniquely identify this deployment.

---

# Why Hash Everything?

Instead of repeatedly storing and comparing:

```text
Name

Version

Chain ID

Contract Address
```

we compress them into one fixed-size value:

```text
32-byte DOMAIN_SEPARATOR
```

This is faster, deterministic, and exactly what EIP-712 requires.

---

# Mental Picture

By the end of the constructor, our understanding became:

```text
Pair Deploys

↓

Constructor Runs Once

↓

Read Current Chain ID

↓

Collect

Name
Version
Chain ID
Contract Address

↓

Hash Them

↓

Store

DOMAIN_SEPARATOR
```

Notice that the constructor is **not** creating a signature.

It is **not** verifying anything.

It is simply preparing a permanent identity that the `permit()` function will later use when building and verifying EIP-712 signatures.
