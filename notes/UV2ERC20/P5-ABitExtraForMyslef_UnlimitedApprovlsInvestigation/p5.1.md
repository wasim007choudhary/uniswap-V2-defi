# `transferFrom()`

```solidity
function transferFrom(address from, address to, uint256 value) external returns (bool) {
    if (allowance[from][msg.sender] != type(uint256).max) {
        allowance[from][msg.sender] =
            allowance[from][msg.sender] - value;
    }

    _transfer(from, to, value);

    return true;
}
```

---

# Purpose

`transferFrom()` allows an approved address (called the **spender**) to transfer tokens **owned by someone else**.

Unlike `transfer()`, where you move **your own tokens**, `transferFrom()` allows you to move **another person's tokens**, but **only if they previously gave you permission**.

That permission is called the **allowance**.

---

# Real Example

Suppose:

```text
Alice owns 100 LP tokens.
```

The Uniswap Router wants to remove Alice's liquidity.

Can the Router simply take Alice's LP tokens?

**No.**

The Router doesn't own those tokens.

Instead, Alice must first approve the Router.

```text
Alice
      │
approve(Router, 100)
      │
      ▼
Allowance Created
```

Now the Router is allowed to execute:

```solidity
transferFrom(
    Alice,
    Pair,
    100
);
```

# Reading The Function

```solidity
if (allowance[from][msg.sender] != type(uint256).max)
```

This was the line that created the biggest confusion.

At first glance it looks like:

> "If allowance is MAX, skip decreasing it."

Immediately this raises a question.

---

# My First Question

I asked:

> **"If the allowance is MAX and it never decreases... doesn't that mean the spender gets unlimited coins?"**

Initially this felt like a security bug.

If the allowance never goes down, couldn't someone keep transferring forever?

The answer turned out to be:

**No.**

Because I was confusing two completely different variables.

---

# The Biggest Confusion

At first I thought these two values were basically the same thing.

```text
balanceOf
```

and

```text
allowance
```

They are **not**.

---

## `balanceOf`

This tells us:

> **How many tokens someone actually owns.**

Example:

```text
Alice

↓

100 LP
```

---

## `allowance`

This tells us:

> **How many of Alice's tokens another address is allowed to spend.**

Example:

```text
Router

↓

100 LP
```

Notice something important.

These values are completely independent.

One is ownership.

The other is permission.

---

# Child Analogy

Imagine Alice owns:

```text
10 chocolates
```

These chocolates represent:

```text
balanceOf
```

Now Alice tells Bob:

> "You may eat 3 chocolates."

That represents:

```text
allowance
```

Bob eats one.

Now:

```text
Alice still owns:

9 chocolates

Bob's remaining permission:

2 chocolates
```

Later Bob eats two more.

Now:

```text
Alice owns:

7 chocolates

Bob's permission:

0
```

Can Bob take another chocolate?

No.

Not because Alice ran out of chocolates.

She still has:

```text
7 chocolates
```

Bob simply has no permission left.

That was the first major realization.

---

# Then We Asked...

Suppose Alice instead says:

> "Bob, from today onward, whenever I have chocolates, you never have to ask me again."

How do we represent that in Solidity?

The ERC20 contract uses one special value.

```solidity
type(uint256).max
```

This is simply the largest value a uint256 can store.

Conceptually:

```text
2²⁵⁶ − 1
```

However,

ERC20 does **not** treat it as an ordinary number.

Instead,

it treats it as a special flag meaning:

```text
Unlimited Approval
```

Notice carefully.

It does **not** mean:

```text
Unlimited Tokens
```

It means:

```text
Unlimited Permission
```

That distinction became the key to understanding this function.

---

# Scenario 1 — Normal Approval

Alice executes:

```solidity
approve(
    Router,
    100
);
```

Current state:

```text
Balance = 500

Allowance = 100
```

Router transfers:

```text
40
```

Result:

```text
Balance = 460

Allowance = 60
```

Router transfers again:

```text
60
```

Result:

```text
Balance = 400

Allowance = 0
```

The next transfer immediately fails because the Router has no remaining permission.

---

# Scenario 2 — Unlimited Approval

Alice executes:

```solidity
approve(
    Router,
    type(uint256).max
);
```

Current state:

```text
Balance = 500

Allowance = MAX
```

Router transfers:

```text
40
```

Result:

```text
Balance = 460

Allowance = MAX
```

Router transfers again:

```text
200
```

Result:

```text
Balance = 260

Allowance = MAX
```

Notice something.

The allowance never changes.

Because Alice already told the Router:

> "You never need to ask me again."

---

# The Question That Scared Me

At this point I immediately asked:

> **"Then can't the owner be drained?"**

The answer is:

**Yes.**

If you approve a malicious contract with:

```solidity
approve(
    contract,
    type(uint256).max
);
```

that contract has unlimited permission to spend your tokens.

This is why wallets display warnings before granting unlimited approvals.

---

# Then Another Question

I asked:

> **"Then why did Uniswap even design it like this?"**

The answer turned out to be usability.

Without unlimited approval:

```text
Approve

↓

Swap

↓

Approve

↓

Swap

↓

Approve

↓

Swap
```

Every interaction requires another approval transaction.

That means:

* more wallet popups,
* more gas,
* more waiting.

With unlimited approval:

```text
Approve Once

↓

Swap

↓

Swap

↓

Swap
```

The user only approves once.

This dramatically improves the user experience.

---

# The Biggest Security Question

Then I wondered:

> **"Who controls this approval? A human?"**

For the official Uniswap V2 Router:

**No.**

The approval is granted to the Router contract itself.

Not to a developer.

Not to an administrator.

Not to a company employee.

---

# Can The Developers Go Rogue?

Initially I thought:

> "What if tomorrow a Uniswap developer becomes malicious?"

Fortunately,

that is not how smart contracts work.

Once the official Router contract is deployed,

its code becomes immutable.

That means it cannot suddenly change tomorrow.

The Router can only execute the code that already exists inside it.

It cannot magically add a new function saying:

```solidity
transferFrom(
    Alice,
    Hacker,
    AliceBalance
);
```

because the deployed code cannot be edited.

That was another huge realization.

With the official Uniswap Router,

I am trusting the immutable code,

not a human being.

---

# But Can I Still Be Drained?

Yes.

If you approve the **wrong contract**.

For example:

```text
Official Router

↓

0xABC...
```

is different from:

```text
Fake Router

↓

0xXYZ...
```

If you accidentally approve:

```text
0xXYZ...
```

then that contract receives your permission,

not the official Router.

This is why users should always verify the contract they are approving.

---

# Step 2 — Decreasing The Allowance

If the allowance is **not** unlimited:

```solidity
allowance[from][msg.sender] =
    allowance[from][msg.sender] - value;
```

Example:

```text
Allowance = 100

Transfer = 30

↓

Allowance = 70
```

If:

```text
Allowance = 20

Transfer = 50
```

Solidity 0.8+ detects an underflow.

The transaction immediately reverts.

No tokens move.

---

# Step 3 — Transfer The Tokens

Finally:

```solidity
_transfer(
    from,
    to,
    value
);
```

This updates:

* sender balance,
* receiver balance,
* emits the Transfer event.

Notice something.

The transfer only happens **after** the allowance logic succeeds.

---

# Final Mental Model

By the end of this discussion,

my understanding completely changed.

Initially I thought:

```text
Unlimited Allowance

=

Unlimited Tokens
```

That is wrong.

The correct understanding is:

```text
Balance

↓

How many tokens Alice owns.

----------------------------

Allowance

↓

Whether another address has permission
to spend Alice's tokens.
```

These are completely different concepts.

`type(uint256).max` simply tells the contract:

> **"Do not decrease this permission after every transfer."**

It does **not** create new tokens.

The spender can only transfer tokens that the owner actually possesses.

Once the owner's balance reaches zero,

all future transfers fail,

even if the allowance remains `type(uint256).max`.
