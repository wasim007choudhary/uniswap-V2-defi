# Part 4.3 — Building the Digest (Big Picture Before Diving Into Each Line)

---

# The Biggest Line Inside `permit()`

After the deadline check succeeds, we finally reach what initially looked like the most complicated line in the entire contract.

```solidity id="d9q8bm"
bytes32 digest = keccak256(
    abi.encodePacked(
        "\x19\x01",
        DOMAIN_SEPARATOR,
        keccak256(
            abi.encode(
                PERMIT_TYPEHASH,
                owner,
                spender,
                value,
                nonces[owner]++,
                deadline
            )
        )
    )
);
```

When we first saw it, it looked overwhelming.

There were:

* two `keccak256()` calls,
* `abi.encodePacked()`,
* `abi.encode()`,
* `DOMAIN_SEPARATOR`,
* `PERMIT_TYPEHASH`,
* `nonces[owner]++`.

Instead of reading from the inside out, we deliberately stopped and asked:

> **What is this entire block trying to produce?**

---

# The Answer

Everything above exists for one purpose.

It produces:

```text id="1bjlwd"
digest
```

This `digest` is the **exact message that Alice signs off-chain**.

Later,

the contract rebuilds this same digest and verifies that the signature really belongs to Alice.

So conceptually:

```text id="3w18t2"
Permit Data

↓

Build Digest

↓

Wallet Signs Digest

↓

Signature
```

---

# Think of It Like a Fingerprint

Imagine writing a contract on paper.

The paper contains:

* owner,
* spender,
* amount,
* deadline,
* nonce.

Instead of signing every letter individually,

you first create a unique fingerprint of the document.

```text id="odk3kx"
Document

↓

Fingerprint

↓

Sign Fingerprint
```

That fingerprint is exactly what the `digest` represents.

# Why Two Hashes?

One of the first questions we asked was:

> **Why are there two `keccak256()` calls?**

Initially it looked redundant.

But after slowing down, we realized they serve completely different purposes.

The **inner hash** creates:

```text id="hjlwm1"
Permit Hash
```

It answers:

> **What exactly is Alice approving?**

The **outer hash** creates:

```text id="fjlwm2"
digest
```

It answers:

> **What exactly will Alice sign?**

Those are different questions.

Therefore they produce different hashes.

# Why Didn't We Explain Everything Immediately?

Although we now had the roadmap,

we intentionally **did not** jump into every nested function at once.

Instead we agreed to understand it exactly like we had studied the constructor:

**one argument at a time.**

The order we chose was:

1. `"\x19\x01"`
2. `DOMAIN_SEPARATOR`
3. `keccak256(abi.encode(...))`
4. `PERMIT_TYPEHASH`
5. `owner`
6. `spender`
7. `value`
8. `nonces[owner]++`
9. `deadline`
10. Outer `keccak256`
11. `digest`

That slow approach prevented us from getting lost inside the nesting.

---

# Important Realization

At this stage, **nothing has been signed yet.**

We are **only constructing** the message that *will* be signed.

The wallet has not yet used Alice's private key.

No signature exists yet.

We are simply building the exact bytes that both:

* the wallet (off-chain), and
* the Pair contract (on-chain)

must independently compute.

If they both compute the same digest,

then the signature can later be verified successfully.

---

# End of Part 4.3

By the end of this section, we stopped looking at the nested expression as one giant line.

Instead, we saw it as three independent stages:

```text id="jlwm10"
Build Permit Hash

↓

Combine with Domain

↓

Build Final Digest
```

Only after understanding this roadmap did we begin dissecting each individual argument.

The very first one was the mysterious prefix:

```solidity id="jlwm11"
"\x19\x01"
```

At first it looked like random bytes, but we eventually discovered that it is a fixed two-byte prefix required by the EIP-712 standard and serves as the header for typed-data signatures.
