# Part 4.10 — `ecrecover()`: How the Contract Proves Alice Signed

---

# The Next Lines

After building the digest, the contract executes:

```solidity id="kv2g4m"
address recoveredAddress =
    ecrecover(
        digest,
        v,
        r,
        s
    );
```

Immediately followed by:

```solidity id="mr4h6x"
require(
    recoveredAddress != address(0) &&
    recoveredAddress == owner,
    "UniswapV2: INVALID_SIGNATURE"
);
```

This is the heart of the entire Permit system.

Everything we learned earlier was preparing us for these two lines.

---

# The Big Question

At this point we asked:

> **How can the contract possibly know Alice signed something?**

Remember:

The contract never sees:

* Alice's private key,
* Alice's wallet,
* Alice clicking "Sign".

So how can it know?

The answer is:

It doesn't know directly.

Instead,

it mathematically verifies the signature.

---

# Inputs to `ecrecover()`

The function receives:

```text id="w6j2y1"
digest

↓

The message

-----------------------

v

↓

Signature Part

-----------------------

r

↓

Signature Part

-----------------------

s

↓

Signature Part
```

Together,

these four values contain enough information to recover the Ethereum address that created the signature.

---

# What Does `ecrecover()` Return?

Many people initially think:

```text id="tw7s9b"
ecrecover()

↓

true / false
```

That is **not** what it returns.

Instead,

it returns:

```text id="w1sl2h"
Ethereum Address
```

Conceptually:

```text id="7z9o4m"
digest

+

v

+

r

+

s

↓

ecrecover()

↓

Recovered Wallet Address
```

---

# Example

Suppose Alice signed the digest.

The contract executes:

```solidity id="7j3srm"
address recoveredAddress =
    ecrecover(...);
```

Result:

```text id="h4z6po"
0xAlice...
```

The recovered address is Alice's wallet.

---

# Then Comes the Comparison

Immediately afterwards:

```solidity id="v8xy8m"
recoveredAddress
==
owner
```

Suppose:

```text id="p2h3vq"
owner

↓

Alice
```

and

```text id="4l2ow5"
recoveredAddress

↓

Alice
```

Now:

```text id="4y5mgd"
Alice

==

Alice
```

The comparison succeeds.

The contract now knows:

> **The signature was created by the owner.**

---

# What If Someone Tries to Cheat?

Imagine Mallory calls:

```text id="0r7jv2"
permit(

owner = Alice,

spender = Mallory,

...
)
```

but Mallory signs the digest herself.

Now:

```text id="v6j6c4"
ecrecover(...)

↓

Mallory
```

The comparison becomes:

```text id="r3u2bc"
Mallory

==

Alice
```

False.

The transaction immediately reverts.

Mallory cannot pretend to be Alice.

---

# What If Someone Changes the Amount?

Suppose Alice really signed:

```text id="4n3ik2"
100 LP
```

But someone modifies the transaction to:

```text id="e0k5gh"
1000 LP
```

before submitting it.

Will the signature still work?

No.

Why?

Because the contract rebuilds the digest using:

```text id="5zv5r4"
1000 LP
```

The wallet originally signed:

```text id="d3x7ni"
100 LP
```

Those produce completely different digests.

`ecrecover()` now recovers a different address (or fails entirely).

The signature becomes invalid.

---

# The Important Realization

The signature is tied to **every field**.

Changing even one field changes the digest.

Changing the digest invalidates the signature.

Examples:

Change:

```text id="oef3nn"
Owner
```

↓

Different digest.

---

Change:

```text id="fy6j8u"
Spender
```

↓

Different digest.

---

Change:

```text id="9z5t8s"
Value
```

↓

Different digest.

---

Change:

```text id="8v8cb4"
Nonce
```

↓

Different digest.

---

Change:

```text id="jh0f4p"
Deadline
```

↓

Different digest.

Everything is cryptographically linked together.

# Child Analogy

Imagine Alice signs a contract saying:

```text id="x6t9ur"
Sell bicycle

₹5000
```

Someone secretly changes it to:

```text id="0q1pl7"
Sell bicycle

₹50
```

Would Alice's signature still be valid?

Of course not.

The document changed.

The signature belongs only to the original document.

Ethereum signatures behave exactly the same way.

---

# Why Check `address(0)`?

The require statement first checks:

```solidity id="9qg4k7"
recoveredAddress
!=
address(0)
```

Why?

Because `ecrecover()` returns:

```text id="z7r5mv"
address(0)
```

if recovery fails.

That usually means:

* invalid signature,
* malformed signature,
* invalid `v`,
* invalid `r`,
* invalid `s`.

Instead of accepting such a result,

the contract immediately rejects it.

---

# The Complete Verification

By the end of this section,

our mental picture became:


# Very Important Observation

Notice something subtle.

The contract never asks:

```text id="3n2sj5"
Who sent this transaction?
```

It never compares:

```text id="7f5sq3"
msg.sender
```

with:

```text id="4r6dc1"
owner
```

The caller could be:

* the Router,
* a relayer,
* another wallet,
* anyone.

The only thing that matters is:

> **Who created the signature?**

That was one of the biggest conceptual shifts from traditional ERC20 approval.

# End of Part 4.10

At this point,

the contract has successfully proven that:

* the signature is valid,
* it belongs to the specified owner,
* none of the signed data has been modified.

Only one final step remains.

After all of this verification succeeds,

the contract finally performs the action Alice authorized:

```solidity id="8h9v5t"
_approve(
    owner,
    spender,
    value
);
```

This brings us full circle to the helper function we studied much earlier, showing how the entire Permit mechanism ultimately ends by updating the standard ERC20 allowance.
