# Part 4.6 — The Inner `abi.encode(...)` (Building the Permit Data)

---

# The Third Piece

At this point, we already understood the first two components of the final EIP-712 message.

```text
0x1901

+

DOMAIN_SEPARATOR

+

?
```

The remaining piece was:

```solidity
keccak256(
    abi.encode(
        PERMIT_TYPEHASH,
        owner,
        spender,
        value,
        nonces[owner]++,
        deadline
    )
)
```

This entire block is responsible for describing **exactly what Alice is approving**.

---

# Big Picture

Before looking at each field individually, we again stopped and asked:

> **What is this entire `abi.encode(...)` trying to build?**

The answer is:

It builds the **Permit object**.

Conceptually:

```text
Permit(

owner,

spender,

value,

nonce,

deadline

)
```

Notice that this is no longer talking about:

* application identity,
* chain,
* contract address.

Those were handled by the `DOMAIN_SEPARATOR`.

Now we are describing the actual permission Alice wants to give.

---

# Step 1

The Permit data is first encoded.

```text
Permit Fields

↓

abi.encode(...)

↓

Bytes
```

---

# Step 2

Those bytes are hashed.

```text
Bytes

↓

keccak256(...)

↓

permitHash
```

Notice something important.

The contract is **not** hashing every field individually.

Instead,

the complete Permit structure is encoded first,

then hashed once.

---

# Why Another Hash?

Earlier we already asked:

> **Why are there two hashes?**

Now we could answer it much more clearly.

The inner hash answers:

> **"What exactly is Alice approving?"**

The outer hash answers:

> **"What exact message will Alice sign?"**

They solve different problems.

---

# Looking Inside `abi.encode(...)`

The encoded Permit contains six values.

```solidity
abi.encode(

    PERMIT_TYPEHASH,

    owner,

    spender,

    value,

    nonces[owner]++,

    deadline

)
```

We intentionally decided to study them one at a time.

---

# The First Value

```solidity
PERMIT_TYPEHASH
```

Immediately we recognised it.

Earlier in the contract we had already seen:

```solidity
bytes32 public constant PERMIT_TYPEHASH =
    keccak256(
        "Permit(address owner,address spender,uint256 value,uint256 nonce,uint256 deadline)"
    );
```

So here,

the contract is **reusing** that constant.

It is **not** computing it again.

---

# A Question We Asked Earlier

Earlier, while studying the constructor, we asked something very important.

> **"Here values are not hashed, only the placeholders?"**

The answer became much clearer now.

When we created:

```text
PERMIT_TYPEHASH
```

we were hashing only the **blueprint**.

Conceptually:

```text
Permit(

owner,

spender,

value,

nonce,

deadline

)
```

Those are not real values.

They are simply field names describing the structure.

---

# Now Real Values Are Inserted

This is the moment where actual values appear.

Example:

Instead of:

```text
owner
```

we now have:

```text
Alice
```

Instead of:

```text
spender
```

we now have:

```text
Router
```

Instead of:

```text
value
```

we now have:

```text
100 LP
```

Instead of:

```text
nonce
```

we now have:

```text
7
```

Instead of:

```text
deadline
```

we now have:

```text
Tomorrow
```

So the blueprint we created earlier is now being filled with real information.

---

# Child Analogy

Earlier we compared `PERMIT_TYPEHASH` to an empty school admission form.

Imagine the blank form contains:

```text
Name:

Age:

Class:
```

That blank form was the blueprint.

Now a student actually fills it in.

```text
Name:

Alice

--------------------

Age:

18

--------------------

Class:

12
```

The form is no longer empty.

It now contains real data.

That is exactly what is happening here.

---

# Why Include `PERMIT_TYPEHASH` Again?

Another excellent question naturally arises.

> **If we already know this is a Permit, why include `PERMIT_TYPEHASH` inside the encoded data?**

Because EIP-712 requires every structured message to begin with its **type hash**.

Conceptually, it says:

```text
This data follows

the Permit structure.
```

Only after identifying the structure do we insert the actual field values.

---

# Mental Picture

By this point, our understanding became:

```text
Blueprint

↓

PERMIT_TYPEHASH

+

Actual Values

↓

owner

spender

value

nonce

deadline

↓

```

The blueprint and the real values are combined into one encoded structure before hashing.

# End of Part 4.6

At this stage we understood **why `PERMIT_TYPEHASH` appears inside the encoded Permit** and how it connects to the blueprint we studied earlier.

The remaining fields are no longer placeholders.

They are the **actual data** Alice is approving:

```solidity
owner,
spender,
value,
nonces[owner]++,
deadline
```
