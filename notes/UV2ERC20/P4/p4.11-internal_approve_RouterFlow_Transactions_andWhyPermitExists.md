# UniswapV2ERC20.md

# Part 4.11 — `_approve()`, Router Flow, Transactions, and Why `permit()` Exists

---

# The Final Line

After the signature has been successfully verified, the contract executes only one line:

```solidity id="g4tk9n"
_approve(
    owner,
    spender,
    value
);
```

At first glance,

this looks almost disappointing.

After all of the cryptography,

all of the hashing,

all of the signatures,

the function simply calls:

```text id="0gx5xf"
_approve()
```

The natural question became:

> **Is that really all?**

Yes.

Everything before this point existed only to answer one question:

> **"Did Alice really authorize this approval?"**

Once the answer is "yes",

the contract performs the exact same allowance update that a normal ERC20 approval would perform.

---

# One of Our Biggest Questions

We asked:

> **Does `permit()` call `approve()`?**

The answer is:

**No.**

It never executes:

```solidity id="v1m0sv"
approve(...)
```

Instead,

it directly executes:

```solidity id="8swccj"
_approve(...)
```

---

# Why Not Call `approve()`?

Remember the implementation of `approve()`.

```solidity id="q0up1y"
function approve(
    address spender,
    uint value
)
{
    _approve(
        msg.sender,
        spender,
        value
    );
}
```

Notice something very important.

The owner is always:

```text id="fiz0qc"
msg.sender
```

That is a fundamental rule of ERC20.

---

# But Inside `permit()`

Suppose:

```text id="1z0wrz"
Owner

↓

Alice

-------------------------

Caller

↓

Router
```

Inside `permit()`,

who is:

```text id="9r7zpj"
msg.sender
```

?

It is:

```text id="c9r2vi"
Router
```

not Alice.

If the contract simply called:

```solidity id="s9gg1v"
approve(...)
```

it would execute:

```solidity id="8rm4g9"
_approve(
    Router,
    Router,
    value
);
```

which is completely wrong.

The allowance should become:

```text id="4c6t2f"
Alice

↓

Router
```

not

```text id="1gt3yy"
Router

↓

Router
```

---

# This Was the Whole Point

Now we finally understood why `permit()` exists.

It replaces:

```text id="w5gk4n"
Owner

=

msg.sender
```

with:

```text id="s9v8jd"
Owner

=

Recovered Signature
```

Once the signature proves:

```text id="e7lyx4"
Recovered Address

==

Alice
```

the contract can safely execute:

```solidity id="x1m61z"
_approve(
    Alice,
    Router,
    value
);
```

even though:

```text id="6z95s4"
msg.sender

=

Router
```

---

# Another Question We Asked

At one point we wondered:

> **Couldn't Uniswap simply override `approve()` instead?**

Unfortunately,

no.

ERC20 defines:

```solidity id="vptm9k"
approve(
    address spender,
    uint value
)
```

Permit requires:

```text id="p2lq34"
owner

deadline

v

r

s
```

Those extra parameters mean it is no longer the same function.

It would not be an override.

It would be a completely different function.

That is why EIP-2612 introduced a brand-new function:

```text id="bx8u6g"
permit(...)
```

instead of modifying ERC20.

---

# The Real Transaction Flow

One of the biggest misconceptions we corrected was about transactions.

Initially,

it seemed like:

```text id="4v7e4n"
Permit

↓

Transaction

↓

Transfer

↓

Another Transaction
```

But that is **not** what happens.

The real flow is:

```text id="x0kkm4"
Alice

↓

Signs

(No Gas)

↓

Frontend Receives Signature

↓

Alice Sends

ONE Transaction

↓

Router

↓

Pair.permit()

↓

Pair.transferFrom()

↓

Pair.burn()

↓

Alice Receives

token0

+

token1
```

Notice carefully.

Only one blockchain transaction exists.

---

# Who Pays the Gas?

Another important discussion.

At first we wondered:

> **The Router is calling the Pair. Doesn't the Router pay? Won't it go broke?**

No.

The Router never pays.

The Pair never pays.

The gas is paid by whoever started the blockchain transaction.

In the normal Uniswap flow,

that is:

```text id="y0e2jx"
Alice
```

---

# Why Doesn't the Router Pay?

Think of one blockchain transaction.

```text id="7k4djr"
Alice

↓

Router

↓

Pair
```

Inside the Pair:

```text id="md0a7g"
msg.sender

=

Router
```

But the transaction itself was still created by:

```text id="d7wv85"
Alice
```

Alice pays the gas.

The Router is simply another contract executing inside Alice's transaction.

---

# One Transaction vs Many

This led to another important realization.

Suppose we have:

```solidity id="crh8mk"
function x() {

    approve();

    transfer();

    transferFrom();

}
```

If Alice calls:

```text id="cbv0mw"
x()
```

that is still:

```text id="v4lqnm"
ONE

Blockchain Transaction
```

Internally,

many function calls occur,

but they all execute inside the same transaction.

---

# Separate Calls

Suppose instead Alice manually executes:

```text id="j0j2xg"
approve()

↓

transfer()

↓

transferFrom()
```

one after another.

Now there are:

```text id="p8v9rd"
Three

Separate Transactions
```

Three wallet confirmations.

Three transaction hashes.

Three base transaction costs.

---

# Then Why Was Permit Invented?

This became another major discussion.

Imagine Uniswap **didn't** have:

```text id="klw9gn"
removeLiquidityWithPermit()
```

Instead,

the Router only supported:

```text id="zb7s53"
permit()
```

Would Permit save gas?

Not really.

You would perform:

```text id="of44ml"
Transaction 1

permit()

----------------------------

Transaction 2

removeLiquidity()
```

Still two transactions.

In fact,

Permit would even perform extra signature verification,

making it slightly more expensive than a plain on-chain `approve()` transaction.

---

# The Real Benefit

Permit becomes valuable only when another function uses it.

For example:

```text id="tx6hsl"
removeLiquidityWithPermit()
```

Internally:

```text id="5gk2ha"
permit()

↓

transferFrom()

↓

burn()
```

Everything executes inside:

```text id="g4fd9f"
ONE

Blockchain Transaction
```

Now the separate approval transaction disappears.

That is where the user experience improves.

---

# End of `permit()`

At this point, we had completed our line-by-line study of the entire `permit()` function.

More importantly,

we no longer viewed it as "a function that skips `approve()`."

Instead, we understood its true purpose:

> **`permit()` is a signature-based authorization mechanism that allows the Pair contract to safely update an ERC20 allowance without requiring the owner to send a separate on-chain approval transaction.**

The one-transaction user experience is the result of that mechanism being integrated into higher-level Router functions such as `removeLiquidityWithPermit()`.
