# UniswapV2ERC20.md

# Part 4.1 — Entering `permit()`: Why Does This Function Exist?

---

# Reaching `permit()`

After finishing the standard ERC20 functions, we finally arrived at:

```solidity
function permit(
    address owner,
    address spender,
    uint value,
    uint deadline,
    uint8 v,
    bytes32 r,
    bytes32 s
) external {
```

Immediately we noticed something very unusual.

Unlike normal ERC20 functions, `permit()` accepts:

```text
owner

spender

value

deadline

v

r

s
```

Three of these variables were completely new:

```text
v

r

s
```

They had nothing to do with balances or allowances.

Instead, they belonged to Ethereum's digital signature system.

---

# First Question

Before reading a single line inside `permit()`, we asked the most important question:

> **Why does this function even exist?**

ERC20 already has:

```text
approve()
```

So why invent another approval mechanism?

---

# Traditional ERC20 Approval

Suppose Alice owns LP tokens.

She wants the Router to spend:

```text
100 LP
```

The normal ERC20 flow is:

```text
Transaction 1

Alice

↓

approve(
    Router,
    100
)

↓

Allowance Updated

--------------------------

Transaction 2

Router

↓

transferFrom(...)
```

Everything works.

But there is one inconvenience.

Alice must send **two separate blockchain transactions**.

---

# The Problem

Every blockchain transaction requires:

* wallet confirmation,
* gas,
* waiting for inclusion in a block.

So removing liquidity usually looked like:

```text
Step 1

Approve

↓

Wait

↓

Step 2

Remove Liquidity
```

Users had to interact twice.

---

# Permit's Goal

Permit solves exactly this problem.

Instead of:

```text
Approve Transaction

↓

Remove Liquidity Transaction
```

we eventually get:

```text
One Signature

↓

One Blockchain Transaction
```

Notice something important.

Permit does **not** eliminate approval.

Approval still happens.

The difference is **how** approval is authorized.

---

# The Biggest Misconception We Had

Initially we wondered:

> Does `permit()` internally call `approve()`?

The answer turned out to be:

**No.**

It never executes:

```solidity
approve(...)
```

Instead, after verifying the signature, it calls:

```solidity
_approve(
    owner,
    spender,
    value
);
```

Remember our earlier helper function discussion.

Both:

```text
approve()

↓

_approve()
```

and

```text
permit()

↓

_approve()
```

end up using exactly the same helper.

Only the authorization step is different.

---

# How Does Normal `approve()` Know the Owner?

Earlier we studied:

```solidity
approve(...)
```

which internally executes:

```solidity
_approve(
    msg.sender,
    spender,
    value
);
```

The owner automatically becomes:

```text
msg.sender
```

Simple.

---

# Why Doesn't That Work Here?

Now imagine the Router trying to remove Alice's liquidity.

The Router eventually calls:

```text
Pair Contract
```

Inside the Pair,

who is:

```text
msg.sender
```

?

Not Alice.

It is:

```text
Router
```

If the Pair simply executed:

```solidity
approve(...)
```

it would approve:

```text
Router

↓

Router
```

instead of:

```text
Alice

↓

Router
```

That would be completely wrong.

---

# Why Not Change `approve()`?

We also asked:

> Could Uniswap simply override `approve()`?

Unfortunately,

no.

ERC20 defines:

```solidity
approve(
    address spender,
    uint value
)
```

Those are the only parameters.

Permit requires much more information:

```text
owner

deadline

v

r

s
```

Changing the function signature would no longer be overriding ERC20.

It would become a completely different function.

---

# So How Does Permit Know the Owner?

This became the biggest architectural idea.

Instead of trusting:

```text
msg.sender
```

Permit trusts:

```text
Digital Signature
```

Conceptually:

```text
approve()

↓

Owner

=

msg.sender

----------------------------

permit()

↓

Owner

=

Recovered Signature
```

This single design change is what makes Permit possible.

---

# Child Analogy

Imagine Alice owns a locker.

### Traditional Approval

Alice walks to the locker herself.

The security guard sees:

```text
Alice is standing here.

↓

Open locker.
```

The guard trusts Alice because she is physically present.

That is exactly like:

```text
msg.sender
```

---

### Permit

Now Alice is not physically present.

Instead,

she signs an authorization letter.

Someone else delivers that letter.

The guard does **not** care who delivered it.

The guard only asks:

```text
Did Alice really sign this?
```

If yes,

the locker is opened.

That is exactly how Permit works.

The signature replaces physical presence.

---

# Another Important Realization

Permit does **not** magically make approvals free.

The approval still happens.

Storage still changes.

Allowance still updates.

The only thing that changes is:

```text
How Alice proves

"I approve this."
```

Instead of proving it by sending a transaction herself,

she proves it by signing a message.

---

At this point we had not yet discussed:

```text
Traditional ERC20

↓

Approval comes from

↓

Digital Signature
```

Everything else inside `permit()` exists only to securely verify that signature before updating the allowance.

---

# End of Part 4.1

By this point we understood **why `permit()` exists**.

Only then did we begin reading its implementation line by line.

The very first line inside the function was:

```solidity
require(
    deadline >= block.timestamp,
    "UniswapV2: EXPIRED"
);
```

This seemingly simple check introduced our next discussion:

* why signatures need expiration,
* why replay protection alone is not enough,
* why both `deadline` **and** `nonce` are required,
* and how they solve two different security problems.
