# Part 4.5 — Why Is `DOMAIN_SEPARATOR` Included Again?

---

# The Second Argument

After understanding the EIP-712 prefix,

we moved to the second argument inside:

```solidity id="jqw9pr"
abi.encodePacked(
    "\x19\x01",
    DOMAIN_SEPARATOR,
    permitHash
)
```

The second argument is:

```solidity id="i3ul0j"
DOMAIN_SEPARATOR
```

At first, this raised a very natural question.

> **Didn't we already build the `DOMAIN_SEPARATOR` inside the constructor?**

Yes.

The constructor already calculated it once and stored it.

Now we were simply reading the stored value.

---

# Looking Back at the Constructor

Earlier we studied:

```solidity id="rjv7gq"
DOMAIN_SEPARATOR = keccak256(
    abi.encode(
        ...domain fields...
    )
);
```

That happened only once when the Pair contract was deployed.

After deployment:

```text id="n6vqlr"
DOMAIN_SEPARATOR

↓

Stored Permanently
```

Nothing inside `permit()` recalculates it.

The function simply reads the existing value.

---

# Then Why Include It Here?

This became one of the biggest conceptual questions.

We asked:

> **If the Permit already contains the owner, spender, amount, nonce, and deadline, why isn't that enough?**

The answer is:

Because those fields describe **what Alice is approving**.

They do **not** describe **where** she is approving it.

The `DOMAIN_SEPARATOR` provides that missing context.

---

# Child Analogy

Imagine Alice signs this note:

```text id="s0gl1x"
Allow John
to collect my package.
```

The note says **what** John is allowed to do.

But imagine the note never mentions:

* which courier,
* which warehouse,
* which company.

Could someone try to use that same note somewhere else?

Possibly.

Now imagine the note instead says:

```text id="hjh4qf"
Allow John
to collect my package

from Warehouse A.
```

Suddenly the permission becomes tied to one specific place.

The `DOMAIN_SEPARATOR` plays the same role.

---

# Passport Analogy

Another way we understood it was through a passport.

Imagine two people both named:

```text id="ez17wa"
Alice
```

The name alone is not enough to identify someone.

A passport also includes information such as:

* country,
* passport number,
* date of birth.

Together those identify one specific person.

Similarly,

the Permit data alone is not enough.

The `DOMAIN_SEPARATOR` uniquely identifies one specific signing domain.

---

# What Does the Domain Represent?

Earlier we learned that the constructor built the domain using:

```text id="ef7nxb"
Application Name

↓

"Uniswap V2"

---------------------

Version

↓

"1"

---------------------

Chain ID

↓

Current Network

---------------------

Contract Address

↓

This Pair Contract
```

Together these form the identity of the application that Alice intended to interact with.

---

# Example

Suppose two Pair contracts exist.

```text id="ndskm9"
ETH / USDC Pair

↓

0xAAA...

-----------------------

WBTC / ETH Pair

↓

0xBBB...
```

Assume everything else is identical:

* same owner,
* same spender,
* same amount,
* same deadline,
* same nonce.

Would the final digest be identical?

No.

Because:

```text id="gt1m8l"
DOMAIN_SEPARATOR

↓

Different
```

The contract addresses differ.

Therefore the domains differ.

Therefore the final digest differs.

---

# Different Chains

Now imagine the exact same Pair contract is deployed on two different blockchains.

```text id="ymd8xv"
Ethereum Mainnet

↓

Chain ID = 1

-----------------------

Sepolia

↓

Chain ID = 11155111
```

Again,

every Permit field could be identical.

But the domain changes because:

```text id="9o2w1p"
Chain ID

↓

Different
```

Therefore the final digest also changes.

This prevents signatures from being reused across chains.

---

# Putting It Together

By this point our mental picture became:

```text id="l1i6dj"
Permit Fields

↓

What Alice approves

----------------------------

DOMAIN_SEPARATOR

↓

Where Alice approves it
```

Both pieces are required.

Neither replaces the other.

---

# Why Store the Domain Instead of Rebuilding It?

Another small observation.

The contract could theoretically rebuild the domain every time `permit()` runs.

However,

that would mean repeatedly hashing:

* name,
* version,
* chain ID,
* contract address,

for every Permit.

Instead,

the constructor computes it once.

Later,

`permit()` simply reads the stored value.

This is simpler and avoids repeating the same work every time.

---

# Connection to the Final Digest

At this point,

the message being assembled looked like:

```text id="h1sud2"
0x1901

+

DOMAIN_SEPARATOR

+

permitHash
```

Notice that we still hadn't produced the final digest.

We were only collecting the three ingredients that would eventually be hashed together.

---

# Important Realization

The `DOMAIN_SEPARATOR` is **not** part of Alice's approval itself.

Alice is **not** approving:

```text id="mbylgq"
DOMAIN_SEPARATOR
```

Instead,

the domain tells the signature:

> **"This approval belongs to this exact application, on this exact chain, for this exact contract."**

That distinction is what prevents signatures from being valid in unintended places.

---

# End of Part 4.5

By the end of this section, the first two components of the EIP-712 message were completely understood:

```text id="gm7vnn"
0x1901

↓

EIP-712 Header

+

DOMAIN_SEPARATOR

↓

Application Identity
```

The only remaining piece was the most important one:

```solidity id="jylb9v"
keccak256(
    abi.encode(
        PERMIT_TYPEHASH,
        owner,
        spender,
        value,
        nonces[owner]++,
        deadline
    )
)
```

This inner hash represents the **actual Permit data** that Alice is authorizing, and in the next section we break it down field by field, beginning with `PERMIT_TYPEHASH`.
