# UniswapV2ERC20.md

# Part 4.2 — `deadline`: Why Does a Signature Need an Expiration Time?

---

# The First Line Inside `permit()`

The first executable statement inside the function is:

```solidity
require(
    deadline >= block.timestamp,
    "UniswapV2: EXPIRED"
);
```

Compared to everything we had just discussed, this line looked surprisingly simple.

However, it raised an important question.

> **Why does Permit even need a deadline?**

Didn't we already have:

```text
nonces
```

to stop replay attacks?

At first, it almost seemed like the deadline was unnecessary.

---

# What Is `deadline`?

The parameter:

```solidity
uint deadline
```

represents one thing:

> **The last timestamp at which this signature is allowed to be used.**

Example:

```text
Current Time

↓

1,700,000,000

---------------------

Deadline

↓

1,700,003,600
```

If the transaction arrives before that time,

the Permit may continue.

If it arrives after,

the transaction immediately fails.

---

# Understanding the `require`

The check is simply:

```text
deadline

>=

Current Block Time
```

If true:

```text
Continue Execution
```

If false:

```text
Revert

↓

"UniswapV2: EXPIRED"
```

Nothing else happens.

The allowance is never updated.

---

# Example

Suppose Alice signs today.

She chooses:

```text
Deadline

↓

Tomorrow
```

If someone submits the Permit today:

```text
Today

<

Tomorrow

↓

Valid
```

Everything proceeds normally.

---

Now imagine someone waits several weeks.

Current time:

```text
Two Weeks Later
```

Deadline:

```text
Yesterday
```

Now:

```text
deadline

<

block.timestamp
```

The transaction immediately reverts.

The signature is useless.

---

# Child Analogy

Imagine your parents write you a permission slip.

```text
Permission

Valid Until

Friday
```

If you try using it on Thursday,

the teacher accepts it.

If you try using the same note next month,

the teacher replies:

> "This permission has expired."

The note might still contain your parents' real signatures,

but it is no longer valid.

That is exactly how `deadline` works.

---

# Then We Asked...

At this point we naturally wondered:

> **If we already have nonces, why do we also need deadlines?**

This became one of the most important conceptual discussions.

Because although both improve security,

they solve **completely different problems**.

---

# What Does the Nonce Prevent?

Suppose Alice signs:

```text
Permit

Nonce = 7
```

Someone submits it.

It succeeds.

Now the nonce becomes:

```text
8
```

If someone tries to submit the same signature again,

the contract now expects:

```text
Nonce = 8
```

but the signature still contains:

```text
Nonce = 7
```

Therefore the digest changes.

The signature becomes invalid.

So the nonce prevents:

```text
Reusing

the

same

signature

after

it

has

already

been

used.
```

---

# But What If Nobody Uses It?

Now imagine a different situation.

Alice signs today.

Nobody submits it.

One year passes.

The nonce is still:

```text
7
```

because the signature was never used.

Now someone discovers that old signed message.

Without a deadline,

they could submit it one year later.

The nonce would still match.

The Permit would still succeed.

---

# This Is What the Deadline Prevents

Even if the signature has **never been used before**,

it eventually becomes invalid.

Example:

```text
Alice signs

↓

Deadline = Tomorrow

↓

Nobody submits it

↓

One Month Later

↓

Expired

↓

Cannot Be Used
```

The nonce alone cannot provide this protection.

---

# Mental Picture

At this point we realized:

```text
Nonce

↓

Stops

Multiple Uses

----------------------------

Deadline

↓

Stops

Very Old Uses
```

They solve different problems.

One does **not** replace the other.

Both are necessary.

---

# Why Doesn't Alice Just Choose a Huge Deadline?

She can.

Nothing prevents choosing:

```text
10 Years
```

or even:

```text
Very Large Timestamp
```

Some applications intentionally use long deadlines.

Others use:

```text
10 Minutes

30 Minutes

1 Hour
```

to reduce risk.

The choice belongs to whoever creates the Permit.

---

# Real Example

Imagine Alice signs:

```text
Owner

↓

Alice

--------------------

Spender

↓

Router

--------------------

Amount

↓

100 LP

--------------------

Deadline

↓

July 10
```

If the Router submits it on:

```text
July 8
```

everything works.

If the Router submits it on:

```text
July 20
```

execution never gets past the first line.

The contract immediately reverts with:

```text
UniswapV2: EXPIRED
```

---

# One More Important Observation

Notice what happens **before** any cryptography.

The contract checks expiration first.

Only if the signature is still valid in time does it continue to build:

* the Permit hash,
* the digest,
* and eventually call `ecrecover()`.

This avoids wasting gas performing expensive cryptographic work for a signature that has already expired.

---

# Summary of Our Understanding

By the end of this discussion, we understood:

```text
deadline

↓

Defines

"How long may this signature be used?"

----------------------------

nonce

↓

Defines

"Has this signature already been used?"
```

Together they provide stronger protection than either one alone.

---

# End of Part 4.2

Once the expiration check passed, we finally reached the largest expression in the entire function:

```solidity
bytes32 digest = keccak256(
    abi.encodePacked(
        "\x19\x01",
        DOMAIN_SEPARATOR,
        keccak256(
            abi.encode(
                PERMIT_TYPEHASH,
                owner,
                spender,
                value,
                nonces[owner]++,
                deadline
            )
        )
    )
);
```

This was the point where all of our earlier discussions about:

* `DOMAIN_SEPARATOR`,
* `PERMIT_TYPEHASH`,
* `nonces`,
* `abi.encode`,
* and `abi.encodePacked`

finally came together into one complete EIP-712 message.
