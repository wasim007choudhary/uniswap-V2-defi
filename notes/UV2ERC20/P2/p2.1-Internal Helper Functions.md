# UniswapV2ERC20.md

# Part 2.1 — Internal Helper Functions (`_mint`, `_burn`, `_approve`, `_transfer`)

---

# Leaving the Constructor

After understanding how the `DOMAIN_SEPARATOR` is built, we reached the first functions inside `UniswapV2ERC20`.

```solidity
function _mint(address to, uint value) internal {
    totalSupply = totalSupply.add(value);
    balanceOf[to] = balanceOf[to].add(value);
    emit Transfer(address(0), to, value);
}

function _burn(address from, uint value) internal {
    balanceOf[from] = balanceOf[from].sub(value);
    totalSupply = totalSupply.sub(value);
    emit Transfer(from, address(0), value);
}

function _approve(address owner, address spender, uint value) private {
    allowance[owner][spender] = value;
    emit Approval(owner, spender, value);
}

function _transfer(address from, address to, uint value) private {
    balanceOf[from] = balanceOf[from].sub(value);
    balanceOf[to] = balanceOf[to].add(value);
    emit Transfer(from, to, value);
}
```

At first glance, something immediately looked strange.

We already knew ERC20 has:

* `approve()`
* `transfer()`

so our first question became:

> **Why create `_approve()` and `_transfer()` as well?**

Similarly,

ERC20 has minting and burning logic, so why create:

```text
_mint()

_burn()
```

instead of putting all the logic directly into the public functions?

---

# First Observation

Notice something.

These functions are **not** the ERC20 functions users call.

Users never call:

```text
_mint()

_burn()

_approve()

_transfer()
```

directly.

Instead, these are helper functions used internally by the contract.

---

# Why Not Put Everything Directly Inside `approve()`?

Imagine there was no helper function.

The contract might look like this:

```solidity
function approve(...) external {

    allowance[msg.sender][spender] = value;

    emit Approval(...);

}
```

Now suppose another function also needs to update an allowance.

For example:

```text
permit()
```

Without `_approve()`, we'd have to copy the same code again.

```text
approve()

↓

copy logic

-------------------

permit()

↓

copy logic
```

Now the contract contains duplicated code.

---

# Using a Helper Function

Instead, Uniswap writes the logic once.

```text
_approve()

↓

Contains the real logic
```

Then:

```text
approve()

↓

calls

↓

_approve()

--------------------

permit()

↓

calls

↓

_approve()
```

Now there is only one implementation.

If the logic ever changes,

it changes in exactly one place.

---

# Child Analogy

Imagine your mother teaches you how to tie your shoes.

Instead of learning it separately for:

* school shoes,
* sports shoes,
* party shoes,

you learn **one method**.

Every pair of shoes simply reuses the same method.

That is exactly what helper functions do.

The knowledge exists once.

Everyone reuses it.

---

# Why Are They Internal/Private?

Another question naturally followed.

Why are these functions marked:

```text
internal
```

or

```text
private
```

instead of

```text
public
```

or

```text
external
```

Because users should never be able to call them directly.

Imagine if anyone could execute:

```solidity
_mint(
    Alice,
    1_000_000
);
```

That would instantly create one million LP tokens.

Obviously that would destroy the protocol.

Only trusted contract logic should be allowed to mint.

Therefore:

```text
_mint()

↓

internal
```

Only the contract itself (or contracts inheriting it) may call it.

---

Similarly,

users should never call:

```solidity
_burn(...)
```

directly.

The Pair decides when LP tokens should be burned.

For example:

```text
Remove Liquidity

↓

Pair.burn()

↓

_burn()
```

The user does not directly execute `_burn()`.

---

# Why Is `_approve()` Private?

Notice:

```solidity
function _approve(...) private
```

not

```text
internal
```

During our discussion we asked why.

The answer is:

Only this contract needs `_approve()`.

There is no reason for child contracts to override or reuse it.

Everything that needs approval logic already exists inside this contract.

Making it `private` slightly reduces the surface area and expresses the intent:

> **This helper belongs only to `UniswapV2ERC20`.**

---

# `_mint()`

Now we looked at the first helper.

```solidity
function _mint(address to, uint value) internal {

    totalSupply = totalSupply.add(value);

    balanceOf[to] = balanceOf[to].add(value);

    emit Transfer(address(0), to, value);

}
```

Conceptually, minting means:

```text
Create brand-new LP tokens.
```

Nothing is transferred from another user.

The tokens simply come into existence.

---

# Why `address(0)`?

ERC20 defines minting as:

```text
address(0)

↓

Alice
```

There is no real sender.

The zero address acts as a symbolic "source" indicating that the tokens were newly created.

---

# Order of Updates

One question we specifically discussed was:

> **Why update `totalSupply` first?**

```solidity
totalSupply += value;

balanceOf[to] += value;
```

Why not reverse the order?

```solidity
balanceOf[to] += value;

totalSupply += value;
```

---

# Does the Order Matter?

For these functions,

**no**.

Suppose:

```text
Supply = 1000

Alice = 100
```

Mint:

```text
50
```

Option A

```text
Supply

1000

↓

1050

↓

Alice

100

↓

150
```

Option B

```text
Alice

100

↓

150

↓

Supply

1000

↓

1050
```

The final blockchain state is identical.

---

# Why Is It Safe?

Because between those two storage writes:

* there is no external call,
* no callback,
* no reentrancy,
* no contract interaction.

Nothing can interrupt execution and observe the intermediate state.

Therefore the ordering is simply a coding choice.

The same idea applies to `_burn()`.

---

# `_burn()`

```solidity
function _burn(address from, uint value) internal {

    balanceOf[from] =
        balanceOf[from].sub(value);

    totalSupply =
        totalSupply.sub(value);

    emit Transfer(
        from,
        address(0),
        value
    );

}
```

Burning is simply the opposite of minting.

Instead of creating LP tokens,

the contract permanently destroys them.

Conceptually:

```text
Alice

↓

100 LP

↓

Burn 20

↓

Alice

80 LP
```

At the same time:

```text
Total Supply

1000

↓

980
```

The burn event is represented by:

```text
Alice

↓

address(0)
```

again using the zero address as the symbolic destination.

---

# Important Connection

Although `_mint()` and `_burn()` live inside `UniswapV2ERC20`,

they are **not** called by ordinary users.

Later, when we return to `UniswapV2Pair`, we'll see functions like:

```text
mint()

↓

calls

↓

_mint()
```

and

```text
burn()

↓

calls

↓

_burn()
```

The Pair controls **when** LP tokens are created or destroyed.

`UniswapV2ERC20` only provides the reusable ERC20 mechanics.

---

# End of Part 2.1

By this point we understood:

* why helper functions exist,
* why they are not public,
* why `_mint()` and `_burn()` are reusable building blocks,
* why updating `totalSupply` before or after `balanceOf` does not change the final result.

The next section will cover the remaining two helpers:

```solidity
_approve()

_transfer()
```

These are the functions that both the standard ERC20 API and the later `permit()` implementation rely upon.
