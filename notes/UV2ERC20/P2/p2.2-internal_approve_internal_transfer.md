# UniswapV2ERC20.md

# Part 2.2 — `_approve()` and `_transfer()`

---

# Two Remaining Helper Functions

After understanding `_mint()` and `_burn()`, we reached the final two helper functions.

```solidity id="3vbgo6"
function _approve(
    address owner,
    address spender,
    uint value
) private {
    allowance[owner][spender] = value;
    emit Approval(owner, spender, value);
}

function _transfer(
    address from,
    address to,
    uint value
) private {
    balanceOf[from] = balanceOf[from].sub(value);
    balanceOf[to] = balanceOf[to].add(value);
    emit Transfer(from, to, value);
}
```

At first, these looked almost too simple.

However, they became extremely important later when we studied `permit()`.

---

# `_approve()`

Let's start with:

```solidity id="skm4zn"
function _approve(
    address owner,
    address spender,
    uint value
)
```

Notice something interesting.

Unlike the public ERC20 function,

this helper does **not** know anything about:

```text id="g0uq0z"
msg.sender
```

Instead,

it simply accepts three parameters:

```text id="v8v0dn"
owner

spender

value
```

and performs one job.

---

# What Does It Actually Do?

The body is extremely small.

```solidity id="w98yyq"
allowance[owner][spender] = value;
```

Conceptually:

```text id="0gw6qf"
Owner

↓

Spender

↓

Allowed Amount
```

Example:

```text id="zvlm63"
Owner

↓

Alice

----------------

Spender

↓

Router

----------------

Value

↓

100 LP
```

Internally this becomes:

```text id="s6b0w6"
allowance[Alice][Router]

=

100
```

Nothing more.

Nothing less.

---

# Approval Event

Immediately afterwards:

```solidity id="tsqg7g"
emit Approval(
    owner,
    spender,
    value
);
```

An event is emitted.

This allows:

* wallets,
* explorers,
* frontends,

to observe that the allowance has changed.

---

# Why Doesn't `_approve()` Verify Anything?

One of the important realizations was this:

Notice that `_approve()` never checks:

```text id="y8rkzr"
Who is calling?

Who owns the tokens?

Is the signature valid?
```

It performs **zero authorization checks**.

Initially this may seem dangerous.

---

# Why Is That Safe?

Because `_approve()` is **not** responsible for deciding whether approval is allowed.

Instead,

whoever calls `_approve()` must already have performed the necessary checks.

Think of `_approve()` as a worker.

It does exactly one job:

```text id="xzrsm1"
Update allowance.
```

It does **not** decide whether it *should* update the allowance.

---

# Child Analogy

Imagine a bank clerk.

The clerk knows how to update an account balance.

But the clerk doesn't decide whether a withdrawal is legitimate.

Someone else first verifies:

* identity,
* authorization,
* documents.

Only after approval does the clerk update the records.

`_approve()` behaves exactly like that clerk.

---

# Where Is Authorization Done?

Later we discovered:

For the normal ERC20 flow:

```text id="1h6d4q"
approve()

↓

Uses

msg.sender

↓

Calls

_approve()
```

For Permit:

```text id="ezfq42"
permit()

↓

Verifies signature

↓

Calls

_approve()
```

Completely different authorization.

Same helper function.

This was one of the elegant design choices in Uniswap.

---

# `_transfer()`

Next we reached:

```solidity id="fj5h9n"
function _transfer(
    address from,
    address to,
    uint value
)
```

Again,

notice something.

There is no:

```text id="gfeplv"
msg.sender
```

The helper simply receives:

```text id="yzvk0m"
from

to

value
```

---

# What Does It Do?

First:

```solidity id="h7ur1l"
balanceOf[from] =
    balanceOf[from].sub(value);
```

The sender's LP balance decreases.

Then:

```solidity id="mrl7hu"
balanceOf[to] =
    balanceOf[to].add(value);
```

The receiver's LP balance increases.

Finally:

```solidity id="m3hwdw"
emit Transfer(
    from,
    to,
    value
);
```

The ERC20 Transfer event is emitted.

---

# Example

Suppose:

```text id="97k78s"
Alice

↓

200 LP

----------------

Bob

↓

50 LP
```

Transfer:

```text id="cx8z4u"
20 LP
```

After `_transfer()`:

```text id="9h2k6i"
Alice

↓

180 LP

----------------

Bob

↓

70 LP
```

The event becomes:

```text id="9clb6w"
Transfer(

Alice,

Bob,

20

)
```

---

# Why Doesn't `_transfer()` Check Ownership?

Again,

another important observation.

There is **no** verification like:

```text id="8xj7l5"
Is from == msg.sender?
```

or

```text id="u7gw6o"
Does the caller have permission?
```

Just like `_approve()`,

this helper performs only the mechanics of updating balances.

Authorization happens elsewhere.

---

# The Pattern

At this point we recognized a recurring design pattern.

```text id="y5rj5j"
Public Function

↓

Validate

↓

Call Helper

↓

Helper updates storage
```

This separation keeps responsibilities clear.

---

# Why Is This Better?

Imagine if every public function contained all of the storage logic itself.

Example:

```text id="vc7vkg"
approve()

↓

update allowance

-------------------

permit()

↓

update allowance

-------------------

some future function

↓

update allowance
```

The same logic would appear over and over again.

Instead,

everything funnels through one helper.

```text id="6ccmpq"
approve()

↓

        _approve()

permit()

↓

```

Only one implementation exists.

---

# Connection to Permit

This section became especially important later.

Initially we wondered:

> If `permit()` updates the allowance, does it simply call `approve()`?

The answer turned out to be:

**No.**

Instead:

```text id="4gmx6o"
permit()

↓

Signature Verified

↓

_approve()
```

while

```text id="vvjwzs"
approve()

↓

Uses msg.sender

↓

_approve()
```

Both end up updating:

```text id="sqzjrc"
allowance[owner][spender]
```

but they reach that point through completely different authorization mechanisms.

---

# Important Realization

This was one of the biggest architectural ideas we learned.

The helper functions:

```text id="e1yxgb"
_mint()

_burn()

_approve()

_transfer()
```

are intentionally "dumb."

They do not make permission decisions.

They simply perform the requested state changes.

The public functions decide **whether** those changes are allowed.

This separation makes the contract:

* easier to understand,
* easier to maintain,
* and avoids duplicating storage-update logic.
