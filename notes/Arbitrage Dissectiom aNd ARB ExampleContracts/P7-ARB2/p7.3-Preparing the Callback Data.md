# 15. Preparing the Callback Data

After calculating the repayment amount, we reach one of the most important lines in the entire contract.

```solidity
bytes memory data =
    abi.encode(
        msg.sender,
        params,
        amountToRepay
    );
```

Many beginners completely ignore this line.

In reality...

this line is what makes Flash Swaps possible.

Without it,

our callback would have no idea

- who started the arbitrage,
- what pair we borrowed from,
- where we should trade,
- how much we borrowed,
- or how much we must repay.

---

# Why Do We Need To Encode Data?

Remember something.

`FlashSwap()` and `uniswapV2Call()` are two completely different function calls.

```
User

↓

FlashSwap()

↓

Pair.swap()

↓

Pair Contract

↓

uniswapV2Call()
```

By the time

```
uniswapV2Call()
```

runs,

the local variables inside

```
FlashSwap()
```

no longer exist.

Imagine this simple example.

```solidity
function foo() external {
    uint256 x = 100;
}
```

When

```
foo()
```

returns,

does Solidity remember

```
x
```

?

No.

It disappears.

Exactly the same thing happens here.

Without encoding,

our callback would know absolutely nothing.

---

# Think of It Like A Delivery Box

Imagine you're moving house.

Instead of carrying

- clothes
- books
- laptop
- charger

one by one,

you put everything into a single box.

```
Box

│
├── Clothes
├── Laptop
├── Books
└── Charger
```

Then someone else receives the box.

Opens it.

Everything is inside.

That is exactly what

```
abi.encode()
```

does.

It packages data into one byte array.

---

# What Are We Packing?

We encode

```solidity
msg.sender
```

This tells us

Who should receive the final profit?

---

We encode

```solidity
params
```

This includes

```
tokenIn

tokenOut

pair0

pair1

amountToBorrow

minProfit
```

Instead of encoding six values separately,

we simply encode the struct.

---

Finally

```solidity
amountToRepay
```

We already calculated this.

There is no reason to calculate it again.

So we simply send it to ourselves.

---

# Visual Representation

```
abi.encode()

│
├── originalCaller
├── ARB2param
└── repaymentAmount
```

↓

Produces

```
bytes
```

↓

Passed into

```
Pair.swap()
```

# Why bytes?

Look at the Pair interface.

```solidity
swap(
    uint amount0Out,
    uint amount1Out,
    address to,
    bytes calldata data
)
```

Notice something.

The Pair doesn't know

or care

what the bytes represent.

It simply stores them.

Then later,

it gives the exact same bytes back during the callback.

Think of it like a sealed envelope.

```
ARB Contract

↓

Writes Letter

↓

Gives Envelope To Pair

↓

Pair Doesn't Read It

↓

Pair Gives Envelope Back

↓

ARB Contract Opens Envelope
```

# 16. Starting The Flash Swap

Now we finally call

```solidity
pair.swap(...)
```

```solidity
IUV2Pair(params.pair0).swap(
    amount0Out,
    amount1Out,
    address(this),
    data
);
```

This is the line that starts everything.

Before this line,

nothing happened.

After this line,

the Pair immediately begins executing.

---

# amount0Out vs amount1Out

A Pair always stores

```
token0

token1
```

Suppose

```
token0 = DAI

token1 = WETH
```

Now suppose we want to borrow

```
WETH
```

Should we write

```
amount0Out = 1 ether
```

?

No.

Because

```
amount0Out

means

borrow token0.
```

We actually want

```
amount1Out = 1 ether
```

because

```
token1

=

WETH.
```

That is why we calculated

```solidity
bool isToken0;
```

earlier.

---

If

```
Borrow token0
```

then

```solidity
amount0Out = borrowAmount;

amount1Out = 0;
```

---

If

```
Borrow token1
```

then

```solidity
amount0Out = 0;

amount1Out = borrowAmount;
```

Exactly one of them is non-zero.

Never both.

---

# Why Is `to` Equal To `address(this)`?

```solidity
to = address(this)
```

The Pair is about to send us tokens.

Where should it send them?

To our arbitrage contract.

```
Pair0

↓

Transfers WETH

↓

ARB Contract
```

Not to the user.

Not to Pair1.

Our contract receives the borrowed tokens first.

Only then can it decide what to do.

---

# Why Do We Pass `data`?

This is the magic switch.

If

```solidity
data.length == 0
```

the Pair performs

```
Normal Swap
```

No callback.

No Flash Swap.

---

If

```solidity
data.length > 0
```

the Pair performs

```
Flash Swap
```

because it knows

> "Someone wants a callback."

This tiny parameter completely changes the behaviour of the Pair.

---

# What Happens Inside Pair.swap()?

The sequence is roughly

```
swap()

↓

Transfer borrowed tokens

↓

Check

Is data empty?

↓

No

↓

Call

uniswapV2Call()
```

Notice something.

The callback happens

BEFORE

the Pair checks repayment.

That is intentional.

It gives us time to perform arbitrage.

---

# Timeline

```
FlashSwap()

↓

Pair.swap()

↓

Pair transfers WETH

↓

ARB receives WETH

↓

Pair calls

uniswapV2Call()

↓

Our Arbitrage Runs

↓

We Repay Pair

↓

Pair Verifies Invariant

↓

Transaction Ends
```

Notice something important.

The Pair has already sent us WETH.

We have not yet paid anything.

This is why it is called a

Flash

Swap.

---

# Why Doesn't Everyone Steal The Tokens?

Someone usually asks:

> "The Pair already sent us WETH.
>
> Why don't we just keep it?"

Because Ethereum transactions are atomic.

Either

everything succeeds,

or

everything reverts.

Suppose we never repay.

Then Pair.swap()

eventually reaches its invariant check.

It notices

```
Pool Lost Value
```

Immediately

```
REVERT
```

The revert undoes

everything.

Including

the earlier WETH transfer.

After the transaction finishes,

it is exactly as if the transfer never happened.

This is one of Ethereum's most powerful guarantees.

---

# Mental Model

Think of Flash Swaps like borrowing a library book.

```
Library

↓

Take Book

↓

Read Book

↓

Return Book

↓

Leave
```

The library lets you leave with the book

only because

it knows

you cannot leave the building

unless the book is returned.

Ethereum transactions work similarly.

The EVM guarantees that either

- the entire sequence succeeds,

or

- none of it happened.

That guarantee is what makes Flash Swaps possible.
