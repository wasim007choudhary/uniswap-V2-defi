# 13. Understanding `ARB2param`

Before looking at `FlashSwap()`, we need to understand the data structure that drives the entire contract.

```solidity

    address tokenIn;
    address tokenOut;
    address pair0;
    address pair1;
    uint256 amountToBorrow;
    uint256 minProfit;
}
```

Think of this struct as the **shopping list** for the arbitrage.

Instead of passing six separate arguments, we package everything into one object.

## tokenIn

```solidity

```

This is the token we borrow from Pair0.

In our examples,


Notice the wording.

We call it **tokenIn** because this token will later become the **input token** for Pair1.

We borrow it first.

Then immediately send it into Pair1.

## minProfit

```solidity
uint256 minProfit;
```

Suppose we expect

```
100 DAI
```

profit.

But another trader performs a swap before our transaction executes.

Now the profit becomes

```
2 DAI
```

Technically,

the arbitrage still succeeds.

But it is not worth doing.

Gas alone could exceed the profit.

So we define

```
Minimum Acceptable Profit
```

If profit becomes smaller,

the transaction reverts.

# 14. FlashSwap()

Now let's study the function that starts everything.


Notice something.

Nothing has been borrowed yet.

Nothing has been swapped.

Nothing has been repaid.

This function only prepares the Flash Swap.

Think of it as filling out a loan application.


The real work does **NOT** happen here.

# Step 1

```solidity

address _token1 = IUV2Pair(params.pair0).token1();
```

# Why do we read token0/token1?

Because later we need to decide

```
Should amount0Out be used?

or

Should amount1Out be used?
```

Without knowing which token occupies which position,

we cannot call

```

```

correctly.

---

# Step 2

```solidity
if (
    params.tokenIn != _token0 &&
    params.tokenIn != _token1

    revert ...
}
```

Imagine Pair0 contains

```

ETH
```


Impossible.

The Pair doesn't own LINK.

So we reject immediately.

This is simply a sanity check.

# Step 5

```solidity
uint256 amountToRepay;
```

Now comes the most important idea.

We calculate repayment

BEFORE

borrowing.

Many beginners think

```
Borrow

â†“

Trade

â†“

Calculate Repayment
```

That is not what we do.

Instead,

```
Read reserves

â†“

Calculate repayment

â†“

Borrow

â†“

Trade

â†“

Repay
```

Why?

Because reserves change after swaps.

If we waited,

the calculation would be wrong.

So we compute repayment while Pair0 is still untouched.

# Step 6

```solidity
amountToRepay =
getAmountIn(...)
```

Notice something subtle.

We are **not** calculating

```
How much DAI
will Pair1 give us?
```

Instead,

we calculate

```
How much DAI
must Pair0 receive
to accept this Flash Swap?
```

These are completely different questions.

One uses

```
Pair0 reserves
```

The other uses

```
Pair1 reserves.
```

Many people accidentally mix them together.

Never do that.

One computes repayment.

The other computes expected proceeds.

```
Pair0

â†“

Repayment

â†“

getAmountIn()
```

```
Pair1

â†“

Expected Output

â†“

getAmountOut()
```

Keep these mentally separated.

It makes the entire contract much easier to understand.
