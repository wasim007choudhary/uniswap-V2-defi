# Solidity Implementation

After deriving the mathematics, the next question is:

> **"How is this actually implemented in Solidity?"**

The answer is surprisingly straightforward.

The smart contract first reads the reserves from both liquidity pools.

```solidity
(uint112 reserveAIn, uint112 reserveAOut,) = pairA.getReserves();

(uint112 reserveBIn, uint112 reserveBOut,) = pairB.getReserves();
```

Using these reserve values, it computes the coefficients of the quadratic equation derived earlier.

Recall that our derivation gave us

\[
a = k^2
\]

\[
b = 2kY_AX_B
\]

\[
c = (Y_AX_B)^2 - f^2X_AY_AX_BY_B
\]

\[
k = fX_B + f^2X_A
\]

where

| Mathematics | Solidity Variable |
|-------------|-------------------|
| \(X_A\) | `reserveAOut` |
| \(Y_A\) | `reserveAIn` |
| \(X_B\) | `reserveBIn` |
| \(Y_B\) | `reserveBOut` |

The implementation then becomes:

```solidity
function getOptimalAmountIn(
    uint256 reserveAIn,
    uint256 reserveAOut,
    uint256 reserveBIn,
    uint256 reserveBOut
) public pure returns (uint256 amountIn) {

    uint256 FEE_NUMERATOR = 997;
    uint256 FEE_DENOMINATOR = 1000;

    // k = fXB + f²XA
    uint256 k =
        FEE_NUMERATOR * reserveBIn +
        FEE_NUMERATOR * FEE_NUMERATOR * reserveAOut;

    // a = k²
    uint256 a = k * k;

    // b = 2kYAXB
    uint256 b =
        2 *
        k *
        reserveAIn *
        reserveBIn;

    // c = (YAXB)² - f²XAYAXBYB
    uint256 c =
        (reserveAIn * reserveBIn) *
        (reserveAIn * reserveBIn)
        -
        (FEE_NUMERATOR * FEE_NUMERATOR) *
        reserveAOut *
        reserveAIn *
        reserveBIn *
        reserveBOut;

    uint256 discriminant =
        Math.sqrt(
            b * b -
            4 * a * c
        );

    amountIn =
        (discriminant - b) /
        (2 * a);
}
```

# Why Compute `k` First?

Notice that the expression

\[
fX_B + f^2X_A
\]

appears multiple times in the equations.

Instead of computing it repeatedly, we calculate it once.

```solidity
uint256 k = ...;
```

and reuse it.

```solidity
uint256 a = k * k;

uint256 b = 2 * k * reserveAIn * reserveBIn;
```

This makes the code:

- Easier to read
- Easier to maintain
- Slightly more gas efficient

# Note About the Fee Representation

Throughout this derivation, we defined

\[
f=\frac{997}{1000}
\]

instead of writing

\[
f=0.997
\]

This was done because Solidity does **not** support floating-point numbers.

By representing the fee as

```text
997 / 1000
```

we can perform all calculations using integer arithmetic while preserving the same mathematical result.

For example,

instead of writing

\[
fX_B
\]

we write

\[
\frac{997}{1000}X_B
\]

Likewise,

instead of

\[
f^2X_A
\]

we write

\[
\left(\frac{997}{1000}\right)^2X_A
\]

This is exactly why Uniswap V2 implementations use the constants

```solidity
uint256 FEE_NUMERATOR = 997;
uint256 FEE_DENOMINATOR = 1000;
```

rather than the decimal value `0.997`.

> **Important:** Since our derivation was performed using \(f=\frac{997}{1000}\), every occurrence of `f` in the equations represents the scaled fee factor `997/1000`. Any Solidity implementation must preserve this scaling consistently throughout the formula to produce the correct optimal input amount.
