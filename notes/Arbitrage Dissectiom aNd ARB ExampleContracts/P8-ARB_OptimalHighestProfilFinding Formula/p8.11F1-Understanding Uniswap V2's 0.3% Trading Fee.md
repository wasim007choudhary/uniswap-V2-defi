# Step F1 – Understanding Uniswap V2's 0.3% Trading Fee

> **Goal of this step**
>
> Before deriving the fee-adjusted arbitrage formula, we first need to understand **where the numbers `997` and `1000` come from**.
>
> These numbers aren't arbitrary.
>
> They are simply Uniswap V2's way of representing a **0.3% trading fee** using integer arithmetic.
>
> Once we understand this, the rest of the derivation follows almost exactly like the zero-fee version.

---

# The Zero-Fee World

Throughout our previous derivation, we assumed there were **no trading fees**.

If a trader sends

```
100 DAI
```

into the pool,

the pool behaves exactly as if it received

```
100 DAI
```

The entire input participates in the swap.

```
User

100 DAI
   │
   ▼
Pool

Receives

100 DAI
```

Because there is no fee,

our swap equation was

```
          xdy
dx = -------------
       y + dy
```

Every unit of input contributes to the price calculation.

---

# The Real Uniswap V2 World

Uniswap V2 charges a **0.30% trading fee** on every swap.

Suppose a trader sends

```
100 DAI
```

to the pool.

Does the pricing equation use all 100 DAI?

No.

First,

the protocol charges a **0.30% fee**.

```
100 × 0.003 = 0.30 DAI
```

Only the remaining

```
99.70 DAI
```

is considered the **effective input** used in the pricing formula.

Graphically,

```
100 DAI
   │
   ▼
──────────────
Fee = 0.30 DAI
──────────────
   │
   ▼
99.70 DAI

Used For Pricing
```

This distinction is extremely important.

---

# Does The Pool Receive Only 99.70 DAI?

No.

The pool balance actually increases by the full

```
100 DAI
```

The fee isn't removed from the pool.

Instead,

it remains inside the pool,

increasing its reserves.

That accumulated value belongs to the liquidity providers (LPs).

So there are two different perspectives.

### From the pool's perspective

```
Pool Balance

+100 DAI
```

### From the pricing formula's perspective

```
Effective Input

99.70 DAI
```

The pricing algorithm behaves **as if only 99.70 DAI were swapped**.

---

# Why Doesn't Uniswap Use 99.7% Directly?

Solidity does not support floating-point numbers.

Writing

```
99.7%
```

or

```
0.997
```

isn't practical inside smart contracts.

Instead,

everything is scaled by **1000**.

```
99.7%

↓

997 / 1000
```

Since

```
997
-----
1000
```

equals

```
0.997
```

both expressions represent exactly the same value.

The only difference is that one works with integer arithmetic.

---

# Effective Input

Suppose the trader provides

```
dy
```

The amount that actually participates in pricing is

```
                 997
dy_effective = ------ × dy
                1000
```

The remaining

```
3
----
1000
```

of the input remains in the pool as the liquidity provider fee.

# Which Part Of The Formula Changes?

Recall the zero-fee swap equation.

```
          xdy
dx = -------------
       y + dy
```

Since only the input amount changes,

we replace

```
dy
```

with

```
997
---- × dy
1000
```

The equation becomes

```
               997
          x × ----- × dy
              1000
dx = ----------------------------
                 997
          y + ----- × dy
                1000
```

This is already the fee-adjusted formula.

It simply hasn't been simplified yet.

---

# Remove The Fraction

Multiply both the numerator and denominator by

```
1000
```

The denominator becomes

```
1000y + 997dy
```

The numerator becomes

```
997xdy
```

The final result is

```
            997xdy
dx = ----------------------
       1000y + 997dy
```

This is the exact swap equation implemented by Uniswap V2.

---

# Compare Both Formulas

### Zero Fee

```
          xdy
dx = -------------
       y + dy
```

### Uniswap V2 (0.3% Fee)

```
            997xdy
dx = ----------------------
       1000y + 997dy
```

Notice something remarkable.

The overall structure is identical.

Only two numbers have appeared:

- `997`
- `1000`

Those numbers simply encode the 0.3% fee.

---

# Why Is Only The Input Multiplied By 997?

This is one of the most common questions.

Suppose you send

```
100 DAI
```

The protocol first charges the fee.

Only after that does it calculate how many output tokens you receive.

The process is

```
User Input
      │
      ▼
Charge Fee
      │
      ▼
Effective Input
      │
      ▼
Calculate Output
```

The output tokens are **not reduced by 0.3%**.

Instead,

the input participating in the swap is reduced before the pricing equation is evaluated.

That's why only the input amount is multiplied by

```
997
```
