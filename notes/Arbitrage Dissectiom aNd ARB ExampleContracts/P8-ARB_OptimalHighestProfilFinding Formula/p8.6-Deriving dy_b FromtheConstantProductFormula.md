# Step 6 – Deriving `dy_b` From the Constant Product Formula

> **Goal of this step**
>
> In **Step 5**, we derived the output of the **first swap**.
>
> ```
> dyₐ
> ↓
> AMM A
> ↓
> dxₐ
> ```
>
> Now we perform the exact same derivation for the **second swap**.
>
> Our objective is to answer:
>
> > **"Given `dxₐ` WETH from the first swap, how much `dy_b` DAI do we receive from AMM B?"**
>
> Nothing fundamentally changes.
>
> We're simply applying the Constant Product Formula to a different liquidity pool.

---

# Zoom Into AMM B

Forget AMM A for a moment.

We're only interested in the second swap.

Initially the pool contains

```
Reserve X = x_b

Reserve Y = y_b
```

As always,

the pool satisfies

```
x_b · y_b = k
```

Exactly the same invariant.

Nothing new.

---

# We Perform The Second Swap

This time,

our input token is **not** DAI.

Our input is

```
dxₐ
```

because that's the WETH we received from the first swap.

So we send

```
dxₐ
```

into AMM B.

Since these tokens enter the pool,

the X reserve increases.

```
New X Reserve

=

x_b + dxₐ
```

---

At the same time,

the pool sends us DAI.

Call that amount

```
dy_b
```

Since DAI leaves the pool,

the Y reserve decreases.

```
New Y Reserve

=

y_b - dy_b
```

Notice the signs.

```
Input Token

+

because it enters the pool.
```

```
Output Token

-

because it leaves the pool.
```

Exactly the same reserve updates we used for AMM A.

---

# Apply The Constant Product Formula

After the swap,

the invariant must still hold.

```
(New X Reserve)

×

(New Y Reserve)

=

k
```

Substitute the new reserves.

```
(x_b + dxₐ)

×

(y_b - dy_b)

=

k
```

Before the swap,

we already knew

```
x_by_b

=

k
```

Since both equal the same constant,

they must equal each other.

```
(x_b + dxₐ)(y_b - dy_b)

=

x_by_b
```

This is exactly the same equation we solved in Step 5,

only mirrored.

---

# Solve For `dy_b`

Our goal is

> **Isolate `dy_b`.**

---

## Step 1 – Expand The Left Side

Apply the distributive property.

```
x_by_b

-

x_bdy_b

+

dxₐy_b

-

dxₐdy_b

=

x_by_b
```

---

## Step 2 – Cancel Common Terms

Subtract

```
x_by_b
```

from both sides.

Result

```
-x_bdy_b

+

dxₐy_b

-

dxₐdy_b

=

0
```

---

## Step 3 – Move The `dy_b` Terms Together

Move

```
dxₐy_b
```

to the other side (or equivalently move the `dy_b` terms together).

Result

```
dxₐy_b

=

x_bdy_b

+

dxₐdy_b
```

Notice something.

Both terms on the right contain

```
dy_b
```

Perfect.

---

## Step 4 – Factor Out `dy_b`

```
dxₐy_b

=

dy_b(x_b + dxₐ)
```

Now the unknown appears only once.

---

## Step 5 – Divide Both Sides

Divide both sides by

```
x_b + dxₐ
```

Result

```
            y_bdxₐ
dy_b = ----------------
         x_b + dxₐ
```

We've successfully derived the output amount of the second swap.

---

# Compare It With The Standard Uniswap Formula

The familiar swap equation is

```
                 reserveOut × amountIn
amountOut = ------------------------------
             reserveIn + amountIn
```

Rename the variables.

```
reserveOut

↓

y_b
```

```
reserveIn

↓

x_b
```

```
amountIn

↓

dxₐ
```

```
amountOut

↓

dy_b
```

Immediately,

we obtain

```
            y_bdxₐ
dy_b = ----------------
         x_b + dxₐ
```

Once again,

this is simply the standard Uniswap swap formula written with different notation.

---

# But There's Still A Problem

Look carefully.

Our optimization variable is

```
dyₐ
```

Remember,

our profit function is

```
F(dyₐ)

=

dy_b

-

dyₐ
```

This function is supposed to depend on

```
dyₐ
```

However,

our equation for

```
dy_b
```

still contains

```
dxₐ
```

That means our function still depends on an intermediate variable.

That's a problem.

---

# Think Of A Recipe

Imagine baking bread.

```
Flour

↓

Dough

↓

Bread
```

Suppose someone asks

> **"How much bread do I get from 2 kg of flour?"**

You wouldn't stop at

```
Flour

↓

Dough
```

Eventually,

you want

```
Flour

↓

Bread
```

without mentioning the intermediate dough.

Exactly the same thing is happening here.

Currently,

our arbitrage looks like

```
dyₐ

↓

dxₐ

↓

dy_b
```

Eventually,

we want

```
dyₐ

↓

dy_b
```

---

# Eliminate The Intermediate Variable

Fortunately,

we already know

```
             xₐdyₐ
dxₐ = ----------------
         yₐ + dyₐ
```

So wherever

```
dxₐ
```

appears,

we simply replace it.

Start with

```
            y_bdxₐ
dy_b = ----------------
         x_b + dxₐ
```

Substitute the expression for

```
dxₐ
```

Result

```
                     xₐdyₐ
                ----------------
                  yₐ + dyₐ
dy_b = y_b ------------------------------
                      xₐdyₐ
          x_b + ----------------
                    yₐ + dyₐ
```

At first glance,

this looks ugly.

Don't panic.

Nothing new happened.

We simply replaced one expression with another.

---

# What Did We Gain?

Before substitution,

our dependency chain looked like

```
dyₐ

↓

dxₐ

↓

dy_b
```

After substitution,

the intermediate variable disappears.

Everything now depends only on

```
dyₐ
```

Exactly what we wanted.

---

# Why Is This So Important?

Remember,

our optimization problem is

```
F(dyₐ)

=

dy_b

-

dyₐ
```

To differentiate this with respect to

```
dyₐ
```

everything inside the equation must ultimately be expressed in terms of

```
dyₐ
```

Substitution accomplishes exactly that.

---

# What Happens Next?

Right now,

our equation contains a **fraction inside another fraction**.

The next step is **pure algebra**.

We'll simplify the nested fraction,

producing one clean expression for

```
dy_b(dyₐ)
```

Once that is complete,

we'll finally be ready to differentiate the complete arbitrage function.

---

# ✅ Step 6 Complete

At this point we understand:

- How to derive the output of the second swap using the Constant Product Formula.
- Why the pool reserves become
  - `x_b + dxₐ`
  - `y_b - dy_b`
- How to derive
  ```
              y_bdxₐ
  dy_b = ----------------
           x_b + dxₐ
  ```
- Why `dy_b` still depends on the intermediate variable `dxₐ`.
- Why substitution is necessary.
- How substituting the expression for `dxₐ` eliminates the intermediate variable and leaves the entire arbitrage dependent on only **one input variable: `dyₐ`**.

The next step is purely **algebraic simplification**, turning the nested fraction into a clean function that can finally be differentiated to compute the **Optimal Amount In**.
