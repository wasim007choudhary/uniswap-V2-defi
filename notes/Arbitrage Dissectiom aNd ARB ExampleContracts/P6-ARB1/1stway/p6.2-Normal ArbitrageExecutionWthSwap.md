# Part 3 вЂ” `swap()` (Normal Arbitrage)

Before diving into Flash Swaps, we first completely understood the **normal arbitrage** flow implemented by `swap()`.

Unlike the Flash Swap version, this approach requires the user to already own the capital that will be used for arbitrage.

---

# Function

```solidity
function swap(Arb1Params memory arbParam) external {
    // before here the approval must happen first
    IERC20(arbParam.tokenIn).transferFrom(
        msg.sender,
        address(this),
        arbParam.inputAmount
    );

    uint256 amountOut = _swap(arbParam);

    if (amountOut - arbParam.inputAmount < arbParam.minProfitFromArb) {
        revert ARB1__swap__RequiredMinimumProfitLimitNotReached();
    }

    IERC20(arbParam.tokenIn).transfer(msg.sender, amountOut);
}
```

---

# Purpose of this Function

This function executes a **normal arbitrage**.

Unlike Flash Swaps,

the contract does **not** borrow funds.

Instead,

the user supplies the capital.

The flow is simply:

```text
Bob

в†“

Give Arb Contract my DAI

в†“

Buy Cheap

в†“

Sell Expensive

в†“

Receive DAI + Profit
```

---

# Where does execution start?

The entire execution begins when Bob calls:

```solidity
arb.swap(arbParam);
```

Therefore,

inside this function:

```text
msg.sender

в†“

Bob
```

This is extremely important because later we call:

```solidity
transferFrom(msg.sender, ...)
```

which means:

```text
transferFrom(
    Bob,
    ArbContract,
    amount
)
```

---

# Before Calling `swap()`

One of the first questions we asked was:

> Can Bob simply call `swap()`?

The answer is:

**No.**

Before calling this function,

Bob must first approve the Arb Contract.

Example:

```solidity
DAI.approve(address(arbContract), 100e18);
```

This approval happens on the **ERC20 token contract**.

Not on the Arb Contract.

---

# Why is approval required?

Initially,

Bob owns:

```text
Bob

100 DAI
```

The Arb Contract owns:

```text
0 DAI
```

The contract cannot simply take Bob's tokens.

ERC20 tokens never allow that.

Instead,

Bob must explicitly give permission.

The ERC20 contract stores:

```text
Owner

в†“

Bob

Spender

в†“

Arb Contract

Allowance

в†“

100 DAI
```

Only after this approval can the Arb Contract pull Bob's tokens.

# First Line

```solidity

    msg.sender,
    address(this),
    arbParam.inputAmount
);
```

This is the first executable line.

Notice:

```text
msg.sender

в†“

```

Therefore this becomes:

```text

    Bob,
    ArbContract,
    100 DAI
)
```

---

# What does `transferFrom()` actually do?

This line **does not perform the arbitrage.**

Its only purpose is to move the starting capital into the contract.

Before:

```text
Bob

100 DAI

в†“

Arb Contract

0 DAI
```

After:

```text
Bob

0 DAI

в†“

Arb Contract

100 DAI
```

(assuming Bob only had 100 DAI for simplicity)

Now the Arb Contract owns the funds needed for arbitrage.

---

# Huge Realization

We asked:

> Why doesn't the Router simply take the DAI from Bob?

Because Bob never calls the Router.

Bob calls:

```text
swap()
```

Later,

the **Arb Contract** calls the Router.

Therefore the Router expects the **Arb Contract** to own the tokens.

The Arb Contract must first become the owner.

---

# Why `transferFrom()` instead of `transfer()`?

This was another important realization.

Initially we wondered:

Couldn't we simply write:

```solidity
transfer(...)
```

The answer is:

**No.**

---

## `transfer()`

Means:

```text
Move MY tokens.
```

The caller moves its own balance.

---

## `transferFrom()`

Means:

```text
Move SOMEONE ELSE'S tokens.

(assuming they approved me)
```

Since Bob owns the DAI,

the Arb Contract must use:

```solidity
transferFrom(
    Bob,
    ArbContract,
    amount
)
```

It cannot use:

```solidity
transfer(...)
```

because the Arb Contract doesn't own Bob's DAI yet.

---

# Important `msg.sender` Realization

One subtle concept we discussed is that **`msg.sender` changes across contract calls.**

Inside:

```solidity
swap(...)
```

```text
msg.sender

в†“

Bob
```

because Bob called the Arb Contract.

But once the Arb Contract executes:

```solidity
IERC20(...).transferFrom(...)
```

the caller of the ERC20 contract becomes:

```text
msg.sender

в†“

Arb Contract
```

The ERC20 contract therefore asks:

> Has Bob approved the Arb Contract?

If yes,

the transfer succeeds.

Otherwise,

it reverts.

---

# Next Line

```solidity
uint256 amountOut = _swap(arbParam);
```

At this point,

the preparation phase is complete.

The Arb Contract now owns the capital.

Everything before this line was simply collecting funds.

Now the actual arbitrage begins.

Inside `_swap()`:

```text
DAI

в†“

Buy Cheap

в†“

WETH

в†“

Sell Expensive

в†“

DAI
```

The function finally returns:

```text
amountOut
```

which is the final amount of `tokenIn` after completing both swaps.

---

# Profit Check

After `_swap()` returns:

```solidity
if (amountOut - arbParam.inputAmount < arbParam.minProfitFromArb) {
    revert ...
}
```

One question we asked was:

Why was:

```solidity
amountOutMin = 0
```

used during the Router swaps?

Wouldn't that ignore slippage?

The answer was:

This contract doesn't validate **each individual swap**.

Instead,

it validates the **entire arbitrage**.

Suppose:

Start:

```text
100 DAI
```

End:

```text
108 DAI
```

Profit:

```text
8 DAI
```

If Bob required:

```text
5 DAI
```

then:

```text
8 >= 5

в†“

Success
```

If instead the arbitrage ends with:

```text
103 DAI
```

Profit:

```text
3 DAI
```

then:

```text
3 < 5

в†“

Revert
```

The contract therefore protects the **overall arbitrage profitability**, not the output of each individual swap.

---

# Returning Funds

Finally,

if the arbitrage was profitable enough,

the contract executes:

```solidity
IERC20(arbParam.tokenIn).transfer(
    msg.sender,
    amountOut
);
```

Notice that this uses:

```solidity
transfer()
```

not

```solidity
transferFrom()
```

Why?

Because now the Arb Contract owns the DAI.

The owner is transferring **its own** tokens back to Bob.

No approval is required.

# Key Takeaways

- This is a **normal arbitrage**, not a Flash Swap.
- Bob supplies the starting capital.
- Bob must approve the Arb Contract **before** calling `swap()`.
- `transferFrom()` is used because the Arb Contract is moving **Bob's** tokens.
- `transfer()` is used at the end because the Arb Contract is now moving **its own** tokens.
- `_swap()` performs the actual arbitrage.
- The contract validates **final profit**, not individual swap outputs.
- The function can be divided into three phases:

```text
1. Collect Funds

в†“

2. Execute Arbitrage

в†“

3. Verify Profit & Return Funds
```

---

# Final Understanding

This function acts as an **orchestrator**.

It does not contain the arbitrage algorithm itself.

Its responsibilities are:

1. Pull the user's funds.
2. Execute the arbitrage through `_swap()`.
3. Verify that the arbitrage achieved the required minimum profit.
4. Return the original asset plus profit back to the user.

The actual **buy low в†’ sell high** logic lives entirely inside `_swap()`.
