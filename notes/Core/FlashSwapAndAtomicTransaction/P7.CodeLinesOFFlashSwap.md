# Flash Swap Related Lines Inside `swap()`

The majority of `swap()` has already been dissected in the **Swap** function dissection. Visit **[notes/Core/UV2Pair--swap.md]** before going any down further. Won't repeat shit again and again.Cheers!

Here, we'll only revisit the lines that are directly responsible for enabling and securing **flash swaps**.

---

# 1. Triggering The Callback

```solidity
if (data.length > 0) {
    IUV2Callee(to).uniswapV2Call(
        msg.sender,
        amount0Out,
        amount1Out,
        data
    );
}
```

### Purpose

This is the heart of Uniswap V2's flash swap implementation.

If:

```text
data.length == 0
```

the Pair performs a normal swap.

If:

```text
data.length > 0
```

the Pair treats it as a flash swap and calls the borrower's callback.

The Pair **does not read** `data`.

It simply forwards the exact same bytes unchanged to the callback.

---

## EVM Execution Flow

When the Pair executes:

```solidity
IUV2Callee(to).uniswapV2Call(...);
```

the EVM temporarily pauses execution of `Pair.swap()`.

```text
Your Contract

flashSwap()

        │
        ▼
pair.swap()

        │
        ▼
Pair Sends Tokens

        │
        ▼
Pair Executes

uniswapV2Call(...)

        │
        ▼
⏸️ EVM pauses Pair.swap()

        │
        ▼
EVM starts executing

YourContract.uniswapV2Call()

        │
        ▼
Your Custom Logic

Arbitrage

Repayment

Anything You Want

        │
        ▼
uniswapV2Call() finishes

        │
        ▼
▶️ EVM resumes Pair.swap()

Exactly where it paused
```

The callback is **not** a new transaction.

It is simply another function call inside the same transaction.

# 2. Reading Final Balances

```solidity
balance0 = IERC20(_token0).balanceOf(address(this));
balance1 = IERC20(_token1).balanceOf(address(this));
```

### Purpose

This is **not exclusive to flash swaps**.

These balance checks are performed for **both normal swaps and flash swaps**.

After all token transfers (and after the callback, if one occurred), the Pair reads its final balances to determine how many tokens it actually received.

* During a **normal swap**, it verifies the user's payment.
* During a **flash swap**, it verifies that the borrowed tokens (plus the LP fee) have been repaid.

The Pair never trusts the borrower.

It verifies by measuring its own balances.

---

# 3. Calculating Returned Amount

```solidity
uint256 amount0In =
    balance0 > reserve0 - amount0Out
        ? balance0 - (reserve0 - amount0Out)
        : 0;

uint256 amount1In =
    balance1 > reserve1 - amount1Out
        ? balance1 - (reserve1 - amount1Out)
        : 0;
```

### Purpose

This logic is shared by **both normal swaps and flash swaps**.

It calculates how many tokens actually entered the Pair.

* During a **normal swap**, this is the user's swap input.
* During a **flash swap**, this is the repayment (borrowed amount + LP fee).

---

# 4. Fee & Invariant Verification

```solidity
uint256 balance0Adjusted = (balance0 * 1000) - (amount0In * 3);
uint256 balance1Adjusted = (balance1 * 1000) - (amount1In * 3);

if (
    balance0Adjusted * balance1Adjusted <
    uint256(reserve0) * uint256(reserve1) * 1000**2
) {
    revert;
}
```

### Purpose

This check is also shared by **both normal swaps and flash swaps**.

It applies the standard **0.3% LP fee** and verifies that the Pair has received enough tokens.

If the invariant is broken, the Pair immediately executes:

```solidity
revert;
```

The EVM then rolls back the **entire transaction**, including:

* Borrowing the tokens.
* The callback.
* Arbitrage.
* Token transfers.
* Any other state changes.

It becomes as if the flash swap never happened.

---

# Final Note

Only this block is exclusive to flash swaps:

```solidity
if (data.length > 0) {
    IUV2Callee(to).uniswapV2Call(...);
}
```

Everything that follows—reading balances, calculating input amounts, applying the LP fee, and verifying the invariant—is **shared by both normal swaps and flash swaps**.

For flash swaps, these existing checks naturally become the security mechanism that ensures the borrowed liquidity has been fully repaid before the transaction is allowed to complete.
