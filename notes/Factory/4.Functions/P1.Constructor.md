# 3.1 Constructor

```solidity
constructor(address _feeToSetter) public {
    feeToSetter = _feeToSetter;
}
```

## Overview

The Factory constructor has a single responsibility:

Initialize the protocol administrator (`feeToSetter`) during deployment.

```text
Deploy Factory

↓

Pass Administrator Address

↓

Store It As feeToSetter
```

Unlike `feeTo`, which starts as `address(0)` (protocol fees disabled), the administrator must exist immediately after deployment so the protocol can later configure `feeTo`.

---

## Why Is `feeTo` Not Initialized?

Notice the constructor only sets:

```solidity
feeToSetter = _feeToSetter;
```

It does **not** set:

```solidity
feeTo
```

This is intentional.

Since storage variables are automatically initialized to their default values, `feeTo` starts as:

```text
address(0)
```

which means protocol fees are **disabled** by default.

Later, governance can enable protocol fees by assigning a non-zero address through:

```solidity
setFeeTo(...)
```

---


## Constructor Visibility In Modern Solidity

You will notice the constructor is written as:

```solidity
constructor(address _feeToSetter) public
```

This is correct for Solidity **0.5.16**.

In modern Solidity versions (0.7.0+), constructors we no longer need to say **public** as it is autmatically that by default now!

Instead, they are simply written as:

```solidity
constructor(address _feeToSetter) {
    feeToSetter = _feeToSetter;
}
```

Constructors are only executed once during deployment, so specifying `public` is unnecessary. The language now treats constructors appropriately without requiring a visibility modifier.

If you still see:

```solidity
constructor(...) public
```

in code today, it usually means one of the following:

* The project targets an older Solidity version.
* The code was copied from an older codebase or tutorial.
* The developer learned Solidity during earlier compiler versions and hasn't updated the syntax yet. **He is an OG** , respect him bruh! 

So whenever reading older protocols like **Uniswap V2**, don't be surprised to see constructor visibility specified—it was simply the correct syntax at the time.\

**Note** - using `public` keyword won't cause error tho, its just unneccesary!

---

## Why Pass The Address Through The Constructor?

Instead of hardcoding an administrator,

```solidity
feeToSetter = 0xABC...;
```

or simply assigning:

```solidity
feeToSetter = msg.sender;
```

the Factory accepts the administrator as a constructor parameter.

This makes the contract reusable across different deployments.

For example:

* Ethereum Mainnet
* Sepolia
* Local Anvil
* Private Chains
* Test Environments

Each deployment can choose a different administrator without modifying the source code.

---

## Why Isn't `feeToSetter` Immutable?

In modern Solidity, a natural question is:

> Why isn't `feeToSetter` declared as `immutable`?

There are two reasons.

### 1. Solidity Version

Uniswap V2 was written in Solidity **0.5.16**.

The `immutable` keyword did not exist yet. It was introduced later in Solidity 0.6.5.

### 2. The Address Can Change

More importantly, even if `immutable` had existed, it would **not** be appropriate here.

Later in the Factory we will see:

```solidity
setFeeToSetter(address _feeToSetter)
```

This function allows governance to transfer administrative control to a new address.

An `immutable` variable can only be assigned during deployment and can never change afterwards.

```text
immutable

↓

Assigned During Deployment

↓

Read Only Forever
```

That behavior is the opposite of what the Factory requires.

The protocol intentionally allows governance to replace the administrator over time.

---

## Historical Note

The constructor is declared as:

```solidity
constructor(...) public
```

because Uniswap V2 targets Solidity **0.5.16**.

In modern Solidity versions, constructors no longer use visibility specifiers, so the equivalent syntax is simply:

```solidity
constructor(address _feeToSetter) {
    ...
}
```

---

## Final Realization

The constructor performs only one task:

Initialize the protocol administrator.

The protocol fee recipient (`feeTo`) is intentionally left unset so protocol fees begin **disabled**, while the administrator (`feeToSetter`) is immediately available to configure or transfer governance in the future.
